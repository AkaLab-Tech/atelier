### M7.1 — Dogfood on a real (non-toy) project

Run a full task cycle on an actual project. Capture friction.

**Target:** `~/Work/deminut` — multi-repo project (Phase 8 workspace candidate). Active repos: `deminut-api`, `deminut-printer-bridge`, `deminut-spa`.

**Progress:**
- 2026-06-11 — Cleaned up the project dir before setup: moved 4 inactive repos (`deminut-landing-new`, `deminut-landing-web`, `printer-client`, `printer-updater`), non-git assets (`branding`, `demos`, `fuentes`), and 2 stale `.code-workspace` files to `~/Work/deminut-archive/` (moved, not deleted). Active dir now holds only the 3 workspace repos + their worktrees.

**Friction captured:**
- ✅ **SSH remotes vs atelier HTTPS-only constraint (resolved 2026-06-11).** All 3 active repos used `git@github.com:Deminut-com/*.git` remotes; atelier's hard constraint (PLAN.md §2 step 5) is HTTPS-only. Re-pointed each `origin` to `https://github.com/Deminut-com/*.git` and verified HTTPS access works for the `Deminut-com` org. **Open question for atelier:** `/setup-workspace` / `/setup-project` should detect SSH remotes and offer to convert (or reject), rather than failing later in the flow.
- 📝 Repo name mismatch: `deminut-printer-bridge` local dir maps to remote `deminut-print-bridge` (print, not printer). Harmless but worth knowing if atelier ever infers remote from dir name.
- 🔧 **F63 — `atelier-setup-project` aborts headlessly + no headless settings override (fixed, PR #169).** Two issues surfaced running setup on the existing repos: (1) every interactive prompt uses `read`, so a non-TTY / piped invocation hits EOF and dies with exit 1 under `set -e` instead of taking the safe default — this aborted setup on all 3 repos; (2) a pre-atelier `.claude/settings.json` (deminut-api had a large hand-grown allowlist) could only be overwritten interactively — `--yes` preserves it (F26-safe) and `--reconfigure` only targets already-registered projects, so there was no headless overwrite path for a new project. Fix: non-TTY stdin now implies non-interactive (mirrors git-wt), and a new `--override` flag overwrites a non-atelier-managed settings.json with a timestamped backup, headless, including first-time setup. `setup-workspace.md` documents forwarding `--override` (operator-confirmed) when cascading to a member with legacy settings.
- 🐛 **F64 — `is_atelier_managed_settings` stale marker (fixed, PR #169).** It keyed off `.permissions.defaultMode == "acceptEdits"`, but the per-project template intentionally omits `defaultMode` (auto-mode is a session-level setting written by `install.sh` as `defaultMode="auto"` into `$ATELIER_CONFIG_DIR/settings.json`, verified by `atelier-doctor`). So the check always failed: every atelier-managed project was misclassified as legacy, F38 drift-resync never fired, and the F26 warning fired on every project. Now detects on the template's distinctive `deny` guardrails. **Behavior change:** re-running `setup-project` on a drifted atelier project now auto-resyncs settings to the current template (with backup) as F38 always intended.
- 🔧 **F65 — reviewer access can't be granted headlessly (fixed, PR #170).** The independent `AtelierReviewer` identity lacks access to the private `Deminut-com` repos. `step_reviewer_access` only grants when run in an interactive TTY (admin + `y` at the prompt); headless it just warns — the same wall as F63. Both identities are configured (`author`=`Miguelslo27`, admin on all 3 repos; `reviewer`=`AtelierReviewer`), so the grant should happen through the atelier flow, not by hand. Fix: a new `--grant-reviewer` flag lets the reviewer-access step invite (via author) and accept (via reviewer) headlessly. The invite+accept body is unchanged F56 code; F65 only adds the headless gate. To be exercised on the real `Deminut-com` repos during the deminut setup run with `--grant-reviewer`.

No worktree/commits created. Surfaced the cluster of findings F66–F72 below — the real takeaway of M7.1 so far: **atelier's task cycle assumes a pristine checkout on a base branch + an atelier-format ROADMAP with planned `[ready]` tasks + an empty `IN_PROGRESS.md`, none of which an actively-developed multi-repo project satisfies.**

**Cycle findings (to implement one by one, discussing each solution first):**
- ✅ **F66 — `/next-task` step-1 guard is over-strict given worktree isolation (implemented, PR #172).** Step 1 refused when the working tree was dirty or the current branch was not a base branch, even though every task runs in an isolated worktree. **Resolution (went further than the original note):** step 1 resolves the base + fetches `origin/<base>` and never touches the operator's checkout; the backlog source (`origin/<base>:ROADMAP.md`) and claim registry (open `task/*` PRs) are reframed as a pluggable **task provider** (Linear-ready); concurrency is `.atelier.json → taskConcurrency.max`; and the `ROADMAP→IN_PROGRESS` move happens inside the worktree, after it is created.
- ✅ **F67 — `/next-task` step-2 refuses on legacy (non-atelier) `IN_PROGRESS.md` (resolved by F66 + F68).** F66 removed the step-2 guard entirely: the concurrency signal is now the open `task/*` PRs, not the base `IN_PROGRESS.md`, so legacy content there no longer blocks `/next-task`. The remaining concern — the step-6 move pasting into a still-legacy `IN_PROGRESS.md` — is handled by adopting the project first: `/adopt-roadmap` resets `IN_PROGRESS.md` to an empty slot (F68). `setup-project` already detects this via `preserved-nonempty` and offers the adoption.
- ✅ **F68 — real ROADMAP format is not atelier-shaped; `/adopt-roadmap` had no §5 output (addressed — claude-roadmap-tools PR #15 + this PR).** Real ROADMAPs use `TASK-NN` / "Prioridad Alta" / nested checkboxes; `/adopt-roadmap` normalized them only into the tracking-flow layout (`High`/`Med`/`Low`), not PLAN.md §5 — and `plan-task` is not a format bridge (it needs an `#id`-bearing §5 task). **Solution chosen (option A):** `/adopt-roadmap` gains `--format atelier` (crt #15) that emits the §5 layout (`P0`/`P1`/`P2` + type tags + `#id` + `~estimate`, `TODO` placeholders, no `[ready]`); `setup-project` Phase 3 now passes `--format atelier` when delegating. **Onboarding pipeline (documented):** `/adopt-roadmap --format atelier` → fill `TODO` placeholders → `/atelier:plan-task <id>` per task → `/atelier:next-task`. **Deferred (option B):** whether atelier's `task-discovery` should accept the `High`/`Med`/`Low` layout natively for operator projects — see ROADMAP Low Priority / Ideas.
- 🟢 **F69 — headless runs default to interactive without `ATELIER_AUTO`.** The cycle was launched with `claude -p` but no `ATELIER_AUTO` / `--yes`, so `/next-task` ran interactive, hit an `AskUserQuestion`, and halted (exit 0, nothing done). **Classification: minor / operator-error.** **Proposed solution:** document that headless/autonomous invocations must set `ATELIER_AUTO=1`; consider auto-detecting a non-TTY stdin as non-interactive in the command prose (mirrors the F63 helper change).
- 🟢 **F70 — language leak into the atelier session.** Running under `CLAUDE_CONFIG_DIR=$ATELIER_CONFIG_DIR`, `/next-task` replied in Spanish. Source unconfirmed — likely the checked-in global `~/.claude/CLAUDE.md` (Spanish) loading into the atelier session. **Classification: minor, F53-class (personal directives leaking into the autonomous flow).** **Proposed solution:** confirm the source and ensure atelier sessions don't inherit the operator's personal language/confirmation directives.
- 🟢 **F71 — `atelier-setup-workspace` records `setupVersion: "unknown"`.** `workspaces.json` for `deminut` got `"setupVersion": "unknown"` instead of the plugin version. **Classification: minor.** **Proposed solution:** have the workspace helper resolve and record the plugin version like `atelier-setup-project` does.
- 🟢 **F72 — `atelier-workspace-status` misaligns with long in-progress titles.** Output columns wrap/shift when a member's in-progress title is long. **Classification: cosmetic.** **Proposed solution:** truncate or column-pad the in-progress title field.
