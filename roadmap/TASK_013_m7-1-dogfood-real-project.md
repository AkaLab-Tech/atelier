# TASK_013 — M7.1 — Dogfood on a real (non-toy) project

Run a full task cycle on an actual project. Capture friction.

**Target:** `~/Work/deminut` — multi-repo project (Phase 8 workspace candidate). Active repos: `deminut-api`, `deminut-printer-bridge`, `deminut-spa`.

**Progress:**
- 2026-06-11 — Cleaned up the project dir before setup: moved 4 inactive repos (`deminut-landing-new`, `deminut-landing-web`, `printer-client`, `printer-updater`), non-git assets (`branding`, `demos`, `fuentes`), and 2 stale `.code-workspace` files to `~/Work/deminut-archive/` (moved, not deleted). Active dir now holds only the 3 workspace repos + their worktrees.

**Friction captured:**
- ✅ **SSH remotes vs atelier HTTPS-only constraint (resolved 2026-06-11).** All 3 active repos used `git@github.com:Deminut-com/*.git` remotes; atelier's hard constraint (PLAN.md §2 step 5) is HTTPS-only. Re-pointed each `origin` to `https://github.com/Deminut-com/*.git` and verified HTTPS access works for the `Deminut-com` org. **Open question for atelier:** `/setup-workspace` / `/setup-project` should detect SSH remotes and offer to convert (or reject), rather than failing later in the flow.
- 📝 Repo name mismatch: `deminut-printer-bridge` local dir maps to remote `deminut-print-bridge` (print, not printer). Harmless but worth knowing if atelier ever infers remote from dir name.
- 🔧 **F63 — `atelier-setup-project` aborts headlessly + no headless settings override (fixed, PR #169).** Two issues surfaced running setup on the existing repos: (1) every interactive prompt uses `read`, so a non-TTY / piped invocation hits EOF and dies with exit 1 under `set -e` instead of taking the safe default — this aborted setup on all 3 repos; (2) a pre-atelier `.claude/settings.json` (deminut-api had a large hand-grown allowlist) could only be overwritten interactively — `--yes` preserves it (F26-safe) and `--reconfigure` only targets already-registered projects, so there was no headless overwrite path for a new project. Fix: non-TTY stdin now implies non-interactive (mirrors git-wt), and a new `--override` flag overwrites a non-atelier-managed settings.json with a timestamped backup, headless, including first-time setup. `setup-workspace.md` documents forwarding `--override` (operator-confirmed) when cascading to a member with legacy settings.
- 🐛 **F64 — `is_atelier_managed_settings` stale marker (fixed, PR #169).** It keyed off `.permissions.defaultMode == "acceptEdits"`, but the per-project template intentionally omits `defaultMode` (auto-mode is a session-level setting written by `install.sh` as `defaultMode="auto"` into `$ATELIER_CONFIG_DIR/settings.json`, verified by `atelier-doctor`). So the check always failed: every atelier-managed project was misclassified as legacy, F38 drift-resync never fired, and the F26 warning fired on every project. Now detects on the template's distinctive `deny` guardrails. **Behavior change:** re-running `setup-project` on a drifted atelier project now auto-resyncs settings to the current template (with backup) as F38 always intended.
- 🔧 **F65 — reviewer access can't be granted headlessly (fixed, PR #170).** The independent `AtelierReviewer` identity lacks access to the private `Deminut-com` repos. `step_reviewer_access` only grants when run in an interactive TTY (admin + `y` at the prompt); headless it just warns — the same wall as F63. Both identities are configured (`author`=`Miguelslo27`, admin on all 3 repos; `reviewer`=`AtelierReviewer`), so the grant should happen through the atelier flow, not by hand. Fix: a new `--grant-reviewer` flag lets the reviewer-access step invite (via author) and accept (via reviewer) headlessly. The invite+accept body is unchanged F56 code; F65 only adds the headless gate. To be exercised on the real `Deminut-com` repos during the deminut setup run with `--grant-reviewer`.

**Progress (cont.):**
- 2026-06-11 — Setup completed via the atelier flow: all 3 repos onboarded with `atelier-setup-project` (deminut-api with `--override`, legacy settings backed up), reviewer access granted to `AtelierReviewer` (READ) on all 3 via `--grant-reviewer`, workspace `deminut` created (3 members + tokens), `atelier-workspace-status` aggregates correctly. Atelier config committed to each repo via worktree + PR: deminut-api #176 (also carries the operator's pre-existing ROADMAP backlog as a separate commit), deminut-print-bridge #99, deminut-spa #232.
- 2026-06-11 — First **full-cycle** attempt: ran `/atelier:next-task` headless on `deminut-spa`. It stopped at step 1 (working tree dirty + non-base branch) without claiming a task. No worktree/commits created. Surfaced the cluster of findings F66–F72 below — the real takeaway of M7.1 so far: **atelier's task cycle assumes a pristine checkout on a base branch + an atelier-format ROADMAP with planned `[ready]` tasks + an empty `IN_PROGRESS.md`, none of which an actively-developed multi-repo project satisfies.**

**Cycle findings (to implement one by one, discussing each solution first):**
- ✅ **F66 — `/next-task` step-1 guard is over-strict given worktree isolation (implemented, PR #172).** Step 1 refused when the working tree was dirty or the current branch was not a base branch, even though every task runs in an isolated worktree. **Resolution (went further than the original note):** step 1 resolves the base + fetches `origin/<base>` and never touches the operator's checkout; the backlog source (`origin/<base>:ROADMAP.md`) and claim registry (open `task/*` PRs) are reframed as a pluggable **task provider** (Linear-ready); concurrency is `.atelier.json → taskConcurrency.max`; and the `ROADMAP→IN_PROGRESS` move happens inside the worktree, after it is created.
- ✅ **F67 — `/next-task` step-2 refuses on legacy (non-atelier) `IN_PROGRESS.md` (resolved by F66 + F68).** F66 removed the step-2 guard entirely: the concurrency signal is now the open `task/*` PRs, not the base `IN_PROGRESS.md`, so legacy content there no longer blocks `/next-task`. The remaining concern — the step-6 move pasting into a still-legacy `IN_PROGRESS.md` — is handled by adopting the project first: `/adopt-roadmap` resets `IN_PROGRESS.md` to an empty slot (F68). `setup-project` already detects this via `preserved-nonempty` and offers the adoption.
- ✅ **F68 — real ROADMAP format is not atelier-shaped; `/adopt-roadmap` had no §5 output (addressed — claude-roadmap-tools PR #15 + PR #173).** Real ROADMAPs use `TASK-NN` / "Prioridad Alta" / nested checkboxes; `/adopt-roadmap` normalized them only into the tracking-flow layout (`High`/`Med`/`Low`), not PLAN.md §5 — and `plan-task` is not a format bridge (it needs an `#id`-bearing §5 task). **Solution chosen (option A):** `/adopt-roadmap` gains `--format atelier` (crt #15) that emits the §5 layout (`P0`/`P1`/`P2` + type tags + `#id` + `~estimate`, `TODO` placeholders, no `[ready]`); `setup-project` Phase 3 now passes `--format atelier` when delegating. **Onboarding pipeline (documented):** `/adopt-roadmap --format atelier` → fill `TODO` placeholders → `/atelier:plan-task <id>` per task → `/atelier:next-task`. **Deferred (option B):** whether atelier's `task-discovery` should accept the `High`/`Med`/`Low` layout natively for operator projects — see ROADMAP Low Priority / Ideas.
- ✅ **F69 — headless runs default to interactive without `ATELIER_AUTO` (documented, PR #174).** `claude -p "/atelier:next-task"` with no `ATELIER_AUTO` / `--yes` ran interactive, hit an `AskUserQuestion`, and halted. Unlike a shell helper, a slash command can't reliably probe `[ -t 0 ]` (it is model prose, not a process), so non-TTY auto-detection isn't viable at the command layer. `next-task.md`'s interaction-mode section now states explicitly that headless / autonomous (`claude -p`) invocations **must** set `ATELIER_AUTO=1` (or pass `--yes`), or the command will stop at the first decision point.
- ✅ **F70 — language "leak" into the atelier session (investigated — not a leak; spun off F73).** `/next-task` replied in Spanish. **Root cause: not a personal-CLAUDE.md leak.** Verified empirically — `CLAUDE_CONFIG_DIR=$ATELIER_CONFIG_DIR claude -p` in a neutral dir reports *no* language directive (`none`), so the F53 isolation holds and `~/.claude/CLAUDE.md` does not load into atelier sessions. The Spanish was the model **mirroring the project's own content** (deminut-spa's `ROADMAP.md` is written in Spanish). Harmless for chat, but it surfaced a real gap → **F73**.
- ✅ **F73 — atelier agents do not pin a deliverable language (implemented, PR #175).** No agent constrained the language of commits / PR descriptions / code comments, so on a non-English project the autonomous flow could mirror the project language. **Decision (option B): configurable, English by default.** New `.atelier.json → deliverableLanguage` (default `"English"`, absent ⇒ English) and a `operator-rules.md` § Deliverable language rule — loaded into every session by the `SessionStart` hook, so all agents inherit it without per-prompt edits. Governs commits, PR titles/bodies, code comments, and atelier-generated docs (`.plan/<id>.md`, drafted `CLAUDE.md`, `ROADMAP`/`IN_PROGRESS`/`HISTORY` entries); does **not** govern operator-facing chat or the project's pre-existing content.
- ✅ **F71 — `atelier-setup-workspace` records `setupVersion: "unknown"` (fixed, PR #174).** The helper derived the plugin root from `BASH_SOURCE` without following the `~/.local/bin/*` → `<checkout>/scripts/*` symlink, so the parent dir was `~/.local` (no `plugin.json`). Now resolves the symlink chain (like `atelier-setup-project`) before locating `.claude-plugin/plugin.json`.
- ✅ **F74 — onboarding never flags a non-§5 `ROADMAP.md` when `IN_PROGRESS.md` is empty (implemented).** The legacy-tracking adoption offer (`setup-project` Phase 3) triggered only on `atelier-tracking-layout=preserved-nonempty`, which `detect_tracking_layout` derives **from `IN_PROGRESS.md` alone** (checkbox or `##` heading present). deminut-spa's legacy `IN_PROGRESS.md` says "(Sin tareas en progreso)" — no checkboxes, no `##` — so it classified `preserved-empty`, Phase 3 never ran, and setup completed without anyone noticing the `ROADMAP.md` is non-§5 (Spanish headings, `TASK-NN` ids, no type/estimate): **zero tasks plannable or claimable, and nothing surfaced it**. **Resolution:** `atelier-setup-project` now emits a second signal `atelier-roadmap-format=conforming|non-conforming|absent` from a new `detect_roadmap_format()` (a ROADMAP is `conforming` if freshly created from the §5 template, or if it has a `P0`/`P1`/`P2` priority heading; `non-conforming` otherwise). `setup-project.md` Phase 3 was split into **3a** (legacy `IN_PROGRESS.md`, unchanged) and **3b** (non-§5 `ROADMAP.md`), which fires **independently of `IN_PROGRESS.md` state** and offers `/adopt-roadmap --format atelier`. Hermetic regression test `hooks/tests/setup-project-roadmap-format-f74.test.sh` (wired into `structural.yml`). **Deliberately NOT added to `atelier-doctor`:** a per-project doctor check would emit false `✗` for every atelier-internal repo and any project intentionally using the `High`/`Med`/`Low` layout — that needs an "operator project vs internal/High-Med-Low" distinction first (relates to this ROADMAP's "first-class High/Med/Low layout" idea and M7.4).
- 📝 **F75 — operator docs drifted from the implemented flow (fixed, PR #177).** Reviewing the deminut onboarding surfaced that `docs/operator-guide.md` predates several shipped behaviors: (1) **no mention of the planning gate** (M4.30) — Steps 5–6 said "write a task, run `task`", but the autonomous flow only claims `[ready]` tasks planned via `/atelier:plan-task`, and the guide's example item lines lacked the backtick type tag and `#id`, so they would never be claimable; (2) **no onboarding path for a project with a pre-existing roadmap** (`/adopt-roadmap --format atelier` pipeline, F68); (3) **nothing about the onboarding PR** — setup's versioned artifacts land via a PR the operator must merge themselves (no auto-merge path), and `next-task` reads the roadmap from `origin/<base>` (F66), so nothing is claimable until that PR lands; (4) **no headless documentation** (`ATELIER_AUTO=1`, F69) and no note that `plan-task` approval is interactive-only and its commit is never pushed by atelier; (5) `docs/troubleshooting.md` still shipped the **stale** "`/next-task` stops at step 2 — a task is already in progress" entry describing the pre-F66/F67 guard that no longer exists. All five addressed in PR #177.
- ✅ **F72 — `atelier-workspace-status` garbled the status row (fixed, PR #174).** Root cause was `open_count`'s `grep -cE … || printf '0'`: `grep -c` prints `0` **and** exits 1 on a zero count, so the fallback printed a second `0` → `"0\n0"`, and the embedded newline wrapped the row (which is why only zero-open members broke). Fixed to `|| true` (matching the x-repo rows). Also reflowed to two lines per member — fixed-width name/status/counts on the first line, the full in-progress title (untruncated, UTF-8-safe) on an indented second line — so long or accented titles never shift the columns.
- ✅ **F76, F77 — closed directly in `HISTORY.md` (PRs #178, #179).** Two self-contained dogfood fixes that shipped after the cluster above; rather than threading them inline here they were logged as standalone `HISTORY.md` entries: **F76** — `/import-conversations` no-TTY path now uses a conversational picker instead of bouncing the operator to the terminal (v0.26.1); **F77** — `atelier-update` no longer crashes (`TEMPLATES[@]: unbound variable`) on releases that touch no templates (v0.26.2). Noted here for cluster traceability.
