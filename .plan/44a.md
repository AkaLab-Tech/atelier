# Plan — #44a Generalize task-orchestrator (non-task PR mode), reframe pr-opener, correct the two-party invariant

- **Status:** ready (approved — product lead, batch approval via AskUserQuestion, 2026-07-07)
- **Type / priority / estimate:** bug · P1 · ~TODO (suggest ~2h — one new orchestrator section + one reframed agent section + two doc rewrites + test-assertion updates)
- **blocked_by:** none

## Approach

Establish the corrected mechanism and its documented invariant, without yet rewiring any command (so no cross-file structural test breaks). Four edits, all prose:

1. **`agents/task-orchestrator.md` — add a strictly-additive, guarded "non-task PR coordination mode".** The current prompt is task-centric end to end (picks a `[ready]` task, cuts a `task/<id>-<slug>` worktree in step 2, moves `ROADMAP → IN_PROGRESS` in step 3, loads `.plan/<id>.md` in step 4, then runs the specialist chain). Add a new top-level section — e.g. "## Non-task PR coordination mode (additive)" placed before "## Core responsibilities" — that:
   - **Is entered only on an explicit guarded signal in the briefing:** a field such as `mode: non-task-pr` carrying `{ repo (owner/name), worktree (absolute, already prepared on `head`), base, head (a non-protected `chore/*`/`docs/*`/`fix/*`/plan-tracking branch, never `task/<id>-<slug>`), title, body }` and **no** `task_id` / roadmap item. State the guard explicitly: **when `mode: non-task-pr` is absent the orchestrator behaves exactly as today** — this is the regression firewall for `/next-task`.
   - **Skips the task-only bookkeeping:** no `git-wt` worktree creation (the caller prepared the worktree), no `task-discovery` / `IN_PROGRESS` scan (step 1), no `ROADMAP → IN_PROGRESS` move (step 3), no `.plan/<id>.md` load (step 4), no `IN_PROGRESS → HISTORY` move, no size-gate ownership. Enumerate these skips so the boundary is unambiguous.
   - **Runs the coordination the same way `/next-task` does, one level down:** dispatch `pr-opener` (via `Task`) as the authoring primitive for the non-task branch (NOT `pr-author` — that stays `task/<id>` only), then the existing `reviewer` dispatch → pre-merge CI wait → `auto-merge` skill segment **unchanged**. Explicitly reuse the review→merge machinery already documented in step 6/8 rather than re-describing it — a cross-reference keeps the diff small and the behaviour identical.
   - **Terminal states:** map `pr-opener`'s `held`/refused returns and `auto-merge`'s `held`/`merged` to the same operator-facing report shape; note that the review-fix loop and the `retry-with-logs` budget apply as in the task path (the authoring specialist differs, the coordination contract does not).
   - Add matching "## Decision rules" bullets: the non-task mode **never** creates a worktree, **never** edits `ROADMAP.md`/`IN_PROGRESS.md`/`HISTORY.md`, **never** dispatches `pr-author`, and **never** runs when `task_id` is present (the two modes are mutually exclusive on the guard).

2. **`agents/pr-opener.md` — reframe the "Why a separate agent" section (do NOT change its mechanics).** The commit/push/`gh pr create`/identity steps stay exactly as written. What changes is the *rationale*: today it claims "the session that calls `Task(pr-opener)` stays a clean, non-authoring actor and can safely dispatch `reviewer` / `auto-merge` afterward" — that is the disproven authoring-only theory. Rewrite that paragraph (and the `<example>` commentary that echoes it) to: `pr-opener` is the **authoring primitive the (generalized) `task-orchestrator` dispatches for non-task branches**; it exists so the *orchestrator* — one level below the driving session — owns author→review→merge for a non-task PR, exactly as `pr-author` serves the task path. Explicitly delete the claim that a driving session may dispatch `reviewer`/`auto-merge` after calling `pr-opener` from its own turn.

3. **`operator-rules.md` — correct the § "PR authoring is always sub-agent work" invariant.** Keep the two-axis (git-identity vs actor/session) explanation — it is correct and load-bearing. Replace the *rule* (lines ~178–191): the current "never hand-author a PR you will then dispatch reviewer against; delegate authoring to pr-author/pr-opener" understates it. New rule: **a session that is driving the conversation never coordinates both authoring and review of the same PR — it delegates the entire author→review→merge coordination one level down to `task-orchestrator`, which dispatches the author (`pr-author` for `task/<id>`, `pr-opener` for non-task branches) and `reviewer` as *its* sub-agents.** State plainly: **delegating only the authoring is NOT sufficient** — the classifier attributes the sub-agent's push back to the orchestrating session (empirically confirmed by PR #293). Rework the "one benign exception" so it still holds: opening a PR and NOT self-reviewing it in the same session (e.g. `/finish-task` handing review to a later session/operator) remains fine; the violation is a driving session coordinating both halves. Consider renaming the header to "PR review is never coordinated by the session that drives authoring" (optional — the product lead may prefer to keep the stable anchor string that other files cross-reference; see Risks).

4. **`PLAN.md §6/§7` — update the invariant note to match.** In §7's "**Invariant — PR authoring is always sub-agent work**" note, replace the authoring-only wording with the corrected coordination-delegation invariant and name the orchestrator's non-task mode as the mechanism. Add the `task-orchestrator` non-task-mode capability to §6's push/PR/merge rules (a sentence: standalone PR flows delegate the whole coordination to the orchestrator; no driving session runs both `gh pr create` and the `reviewer`/`auto-merge` dispatch for the same PR). Keep the `pr-opener` row in the §7 agents table but correct its one-line purpose from "so the dispatching session stays a clean, non-authoring reviewer" to "authoring primitive the orchestrator dispatches for non-task PRs".

5. **`hooks/tests/pr-opener-delegation.test.sh` — update the doc-assertion groups to the corrected wording (Groups 3 & 4 here; Group 2 is #44b's).** The current Group 3 asserts the *old* rule strings (`'never hand-author a PR you will'`, `'clean, non-authoring actor'`) and Group 1 asserts pr-opener's old rationale (`'clean, non-authoring actor'`). Retarget these to the corrected invariant strings this sub-task introduces (e.g. "delegates the entire author→review→merge coordination", "delegating only the authoring is NOT sufficient", pr-opener "authoring primitive the orchestrator dispatches"). Keep the assertions that remain true (file exists, tool list, task/<id> contrast, no-tracking-move, no-playwright). This keeps the suite green under the new prose and locks the correction in.

The change is additive at the orchestrator level (guarded new mode), a rationale rewrite at the pr-opener level, and a wording correction at the doc level — no command behaviour changes in this sub-task, so `/next-task`, `/align`, `/release` all run exactly as before until #44b/#44c rewire them.

## Risks / open questions

- **Cross-reference stability of the operator-rules header.** `PLAN.md §7` and `hooks/tests/pr-opener-delegation.test.sh` grep the literal header/anchor string `PR authoring is always sub-agent work`. Renaming the header (Approach step 3, optional) would require updating both cross-references in the same commit. Safe default: keep the header string, correct only the body — flagged for the product lead's preference.
- **Guard-signal naming.** `mode: non-task-pr` is a proposal; the implementer should align it with however `task-orchestrator` already distinguishes `resume_mode` cases so the two guard mechanisms read consistently. Not behaviour-affecting, but worth a consistent vocabulary.
- **Keeping the diff under budget.** Four prose files plus a test; the orchestrator section must reuse the existing review→merge segment by reference rather than restating it, or the line count creeps toward the 200 limit. The implementer should resist re-documenting the CI-wait/auto-merge steps.
