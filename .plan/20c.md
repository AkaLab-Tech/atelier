# Plan — #20c plan-task planning gate: Project Ready field

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** feat · P1 · ~TODO (epic estimate is `~TODO`; product lead should set a real number — suggest ~3h: `plan-task` is fully files-only today, so this is the deepest of the three slices)
- **blocked_by:** #18 (M9.1 keystone), #20a (backend selection must exist so a repo can be `github-project`-backed)

## Approach

`commands/plan-task.md` is entirely files-only today. Its Phase 1 reads `ROADMAP.md` from cwd, Phase 4 step 1 flips `[ready]` by **editing a `ROADMAP.md` line** (inserting the literal `[ready]` token after the checkbox), and Phase 4 step 3 commits `ROADMAP.md` + `.plan` together. For a `github-project`-backed repo there is no `ROADMAP.md` line to edit — the `[ready]` marker becomes the Project's dedicated **`Ready`** field (PLAN.md §16.5). This slice makes `plan-task` backend-aware so approval flips that field via crt's `RoadmapBackend` instead of editing a line, **while `.plan/<id>.md` stays a committed tracked repo file in both backends**.

The split §16.5 mandates is the spine of this slice: `.plan/<id>.md` is repo state (spec + evidence the implementer needs in the worktree, committed to git regardless of backend); only the `Ready` flag moves to the Project. So even for github-project, this command still commits `.plan/<id>.md` to the repo — it just does **not** edit/commit `ROADMAP.md`.

Concrete changes to `commands/plan-task.md`:

1. **Phase 1 — resolve the backend.** After confirming a project root, run `atelier-task-backend <project-root>` (the M9.1 resolver, same as next-task step 2). Branch the rest of the command on `files` vs non-`files`.
   - **Phase 1 step 3 (clean-paths check).** For `files`, keep `git status --porcelain ROADMAP.md .plan` as-is. For non-`files`, only `.plan` must be clean (there is no `ROADMAP.md` to guard — and editing it is not part of the flow). Adjust the check so a github-project repo without a §5 `ROADMAP.md` is not refused by the existing Phase 1 step 1 "must be run from a project root with ROADMAP.md present" guard — for a non-`files` backend, presence of `.roadmap.json` (backend resolved) substitutes for `ROADMAP.md` presence. The implementer must confirm a github-project repo can still satisfy Phase 1.

2. **Phase 2 — planner dispatch (unchanged).** The planner already writes `.plan/<id>.md` regardless of backend (the planner is a separate agent; this slice does not touch it). For a non-`files` backend the planner's ROADMAP-block reading would need the backend — but that is the planner's concern, out of scope for this slice unless the product lead wants it folded in (flagged below).

3. **Phase 4 — flip readiness, backend-aware.**
   - **`files` (unchanged):** edit the `ROADMAP.md` line to insert `[ready]`, flip the `.plan` status, `git add ROADMAP.md .plan`, commit.
   - **`github-project` (new):** **do not** edit `ROADMAP.md`. Instead, for each id in `ready_to_mark`, set the Project's `Ready` field via the backend (the crt `RoadmapBackend` operation that writes a custom field — confirm the exact method name against `docs/RoadmapBackend.md`; likely a `moveTask`/`updateTask`-class call or a dedicated field setter the `GitHubProjectBackend` exposes for `Ready`). Then flip the `.plan/<id>.md` draft status and `git add .plan` + commit (plan only — `ROADMAP.md` is not part of this backend's repo state). The commit message stays `chore(plan): mark <task_id> ready with approved plan`.

4. **Phase 5 — discard, backend-aware.** For `files`, keep `git checkout -- ROADMAP.md` + `git clean -f .plan`. For github-project, the discard reverts only `.plan` (no `ROADMAP.md` to revert) and — if the `Ready` field was already flipped before a later failure — un-sets it via the backend. Because approval is the gate and the field flip happens **after** approval in Phase 4, a rejection never reaches the flip, so the common discard path is `.plan`-only; the field-unset path covers a mid-Phase-4 failure.

5. **Interaction mode (unchanged, reinforced).** Approval stays interactive-only for **both** backends — in headless/`--yes`/`ATELIER_AUTO`, run the planner, write the draft, and stop without flipping `Ready`. OAuth for the Project write cannot be auto-resolved, so this is doubly mandatory for github-project (§16.5: "approval remains interactive-only, never headless").

6. **Hard refusals + `allowed-tools`.** Add the `Bash(atelier-task-backend:*)` grant to the frontmatter (it is not there today). Add whatever grant the backend `Ready`-write call needs (per the project's crt-invocation convention — likely the same skill/command path next-task uses for `moveTask`). Keep all existing refusals: never flip `Ready` without explicit approval; never start the implement chain; never run from a task worktree; never push. Add a refusal: "never edit `ROADMAP.md` for a non-`files` backend — there is none; the `Ready` flip goes through the backend."

## Affected areas

- `commands/plan-task.md` — backend resolution in Phase 1; backend-aware clean-paths + ROADMAP-presence guard; Phase 4 split (`files` edits the line; `github-project` sets the Project `Ready` field via the backend, commits `.plan` only); Phase 5 backend-aware discard; frontmatter `allowed-tools` (`atelier-task-backend` + the `Ready`-write grant); a new hard-refusal bullet. The `.plan/<id>.md`-stays-committed invariant is preserved across both backends.

## Acceptance criteria

- [ ] For a `github-project` repo, approving a plan sets the Project's `Ready` field via the `RoadmapBackend` (a backend call) instead of editing a `ROADMAP.md` line.
- [ ] `.plan/<id>.md` is still committed as a tracked repo file for the `github-project` backend (only the `Ready` flag lives in the Project, not the plan artifact).
- [ ] The planning gate is satisfied only when both the Project `Ready` field is set **and** `.plan/<id>.md` is committed (mirrors the §5 `ready-without-plan` inconsistency surfaced by next-task/#20b).
- [ ] Approval stays interactive-only: in headless / `--yes` / `ATELIER_AUTO`, the command writes the draft and stops without flipping `Ready` (no auto-approval, no OAuth-requiring write).
- [ ] The `files` backend path is unchanged — `[ready]` is still inserted into the `ROADMAP.md` line and committed with `.plan`.
- [ ] On rejection/discard for a `github-project` repo, no `Ready` field is left set and the draft `.plan` files are removed; the repo returns to its pre-command state.

## Risks / open questions

- **Exact backend method to set `Ready`.** `docs/RoadmapBackend.md`'s documented verbs are `listTasks`/`getTask`/`addTask`/`moveTask`/`appendHistoryEntry`/`isAvailable` — none is obviously a "set a custom boolean field" operation. The `GitHubProjectBackend` (M9.2, #19) must expose a way to write the `Ready` custom field; the implementer must confirm the exact call (a dedicated setter, an `updateTask`, or `moveTask` semantics) against the merged crt v0.5.0 contract. **If the contract has no `Ready`-write operation, this slice is blocked on a crt change** — surface to the product lead before implementing. This is the single biggest open question of the epic.
- **Does the planner need backend awareness?** This slice scopes only the `/plan-task` *command* (the gate flip). The `planner` agent still reads the task block; for a github-project repo that block comes from the backend, not `ROADMAP.md`. The plan assumes the planner already handles this (or that #20b's task-discovery routing covers the read path) — if not, planner backend-awareness is a gap the product lead may want folded into this slice or split into a follow-up. Flagging.
- **Phase 1 ROADMAP-presence guard.** Today Phase 1 step 1 refuses if cwd has no `ROADMAP.md`. A pure github-project repo may legitimately have no §5 `ROADMAP.md`. The plan substitutes `.roadmap.json` presence as the github-project equivalent; the implementer must make sure this does not loosen the guard for a genuinely-misconfigured files repo.
- **Coupling to #20b.** The `Ready`-field *read* (selection gate) is #20b; the *write* (this slice) is #20c. Keep the field name and semantics identical across both committed plans (both cite §16.5). The second of the two to be implemented should re-read the sibling's committed plan.
