# Plan — #36 Fix flaky SIGPIPE/pipefail race in `operator-language.test.sh`

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** bug · P2 · ~30m
- **blocked_by:** none

## Approach

The flake is a classic `printf … | grep -q` race under `set -o pipefail`. In
`hooks/tests/operator-language.test.sh` the hook output `$H_SET` is ~26KB. The
pipelines at lines 45–47 do `printf '%s' "$H_SET" | grep -q PATTERN`: `grep -q`
exits (closing its end of the pipe) on the first match, and if `printf` has not
yet flushed all 26KB it takes SIGPIPE → "write error: Broken pipe". With
`pipefail` active the pipeline's exit status becomes non-zero, so the
`&& pass || fail` flips to a spurious `fail`. Whether printf finishes before
grep closes is a pipe-buffer-flush race, so it fires intermittently and reds-out
the `structural` CI job on unrelated PRs (observed on docs-only PRs #265/#266,
both green on untouched re-run). Lines 43–44 already document this exact trap for
the `$H_SET` capture, then reintroduce it immediately below.

Fix: remove the writer process. Replace each `printf '%s' "$H_SET" | grep -q PATTERN`
with a here-string `grep -q PATTERN <<<"$H_SET"`. The shell feeds the string to
grep's stdin directly (no `printf` subprocess to receive SIGPIPE), so grep
closing early is harmless. The match semantics are byte-identical — same
pattern, same `-q`, same `&& pass || fail` — so no assertion changes meaning.

Steps, in order:
1. Convert lines 45, 46, 47 (`$H_SET` / ~26KB — the proven-flaky ones) to
   `grep -q PATTERN <<<"$H_SET"`. This is the fix that closes the bug.
2. For consistency and to retire the antipattern class within this file, also
   convert the two small-payload uses at lines 28 and 33
   (`printf '%s' "$(set_lang --show)" | grep -q …` → `grep -q … <<<"$(set_lang --show)"`).
   These are not observed-flaky (the `--show` output is a few bytes and flushes
   in one write before grep closes), but the conversion is trivial and removes
   the footgun uniformly.
3. Leave the two reads that consume their entire input as-is — they cannot
   SIGPIPE:
   - line 54 `printf '%s' "$H_SET" | grep -abo … | head -1` — `grep -abo` emits
     the match line before `head -1` closes, the result is captured in `$(…)`
     (assignment, whose status is unchecked and not under `set -e`), and `_off`
     still receives the offset. Comparatively safe; verified, no change.
   - line 65 `printf '%s' "$H_UNSET" | grep -c …` — `grep -c` reads the whole
     stream to count, so it never closes early. Safe; no change.
4. Run the test locally to confirm it still passes:
   `hooks/tests/operator-language.test.sh` (requires `jq`; exits 0 on all-pass).

## Risks / open questions

- **Sibling tests share the pattern but over small payloads.** An audit of
  `hooks/tests/*.test.sh` shows the same `printf '%s' "$VAR" | grep -q` idiom
  under `pipefail` in `atelier-align-survey`, `list-projects-render`,
  `workspace-status-render`, `workspace-status-mixed-backend`,
  `orient-decision-tree`, and `migrate-task-ids`. In every case the piped
  variable is small (a few lines / hundreds of bytes), so `printf` completes its
  single write before `grep -q` closes — the SIGPIPE race does not realistically
  fire, and none have been observed flaky. **Open question for the product lead:**
  scope this fix to the proven-flaky `operator-language.test.sh` only (recommended
  — minimal diff, closes the observed bug), or opportunistically convert the
  siblings to here-strings too to retire the antipattern workspace-wide? The
  latter is low-risk but widens the diff to ~7 files; it could also land as a
  separate follow-up chore. Test files are exempt from the PR size gate either
  way, so size is not the constraint — reviewability/scope is.
- **`<<<` here-strings are Bash, not POSIX sh.** The file already uses
  `#!/usr/bin/env bash` and Bash-only constructs (`BASH_SOURCE`, `[[`-style
  idioms elsewhere in the suite), so `<<<` is safe here. No portability
  regression.
- **No behavior change to ship code.** This touches only a test harness; the
  hook (`hooks/load-operator-rules.sh`) and setter are untouched, so there is no
  risk to the operator-language feature itself.
