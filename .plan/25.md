# Plan — #25 `/atelier:babysit-prs` — watch open `task/*` PRs and drive them to merge

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** feat · P2 · ~4h (a new command file + light PLAN.md/skill/agent touches; no source code in this repo — atelier ships as markdown prompts)
- **blocked_by:** none (composes with #23 CI-wait and #24 review-fix loop, but carries **no hard dependency** on either — degrades gracefully when they are not yet merged; see Risks)

## Approach

Today nothing re-evaluates an open `task/*` PR once the chain that produced it
ends or holds. `/atelier:next-task` opens the PR and runs `reviewer → auto-merge`
**once** (its job ends at hand-off, `commands/next-task.md:178`); if CI was still
running, a review hadn't landed, or a guardrail held, the operator must
re-invoke manually. There is no unattended watcher — exactly what the
`auto-merge` skill already forward-references at `skills/auto-merge/SKILL.md:72`
and `:209` ("…or let #25 babysit-prs handle the unattended case").

**Add `commands/babysit-prs.md`: an out-of-chain, convergent watcher** that
enumerates every open `task/*` PR, triages each (read-only first), drives the
*eligible* ones to merge by reusing the machinery that already exists, and
re-evaluates until the set reaches a fixed point — reporting the PRs that are
legitimately held for the operator. The command does **not** reinvent review or
merge logic; it is the **fan-out + convergence loop** around per-PR drive
machinery atelier already has.

The key reuse insight: the per-PR drive is **exactly** `resume-task`'s
`pr-open-resume` path (`commands/resume-task.md:125-161`), which re-enters
`task-orchestrator` at the `reviewer → auto-merge` segment — and that segment
already carries #23's pre-merge CI wait and #24's review-fix loop when they have
landed. babysit-prs is that single-PR drive, fanned out across **all** open
`task/*` PRs and looped to convergence.

Concretely, `commands/babysit-prs.md`:

1. **Interaction mode** (mirror `next-task` / `resume-task`): non-interactive when
   `$ARGUMENTS` carries `--yes`/`-y` or `ATELIER_AUTO` is set; otherwise
   interactive. Optional `--workspace` flag to fan out across workspace members
   (default: current project only — see Risks for the scope decision).

2. **Enumerate the claim registry — read-only first.** Reuse the exact filter
   `status.md §3` and `next-task` step 2 use:
   ```bash
   gh pr list --state open \
     --json number,title,headRefName,isDraft,reviewDecision,statusCheckRollup,mergeable,url \
     --limit 50
   ```
   Keep only `headRefName` starting with `task/` (the claim registry, §16.4).
   Draft and non-`task/*` (out-of-band) PRs are reported and skipped, never driven.

3. **Triage each PR into a disposition** using the `auto-merge` six guardrails as
   the lens (never duplicating or weakening them):
   - **mergeable-now** (`APPROVED` + CI green + no guardrail trip) → drive to merge.
   - **unreviewed** (`reviewDecision` null/`REVIEW_REQUIRED`, no `atelier:reviewer`
     review yet) → dispatch the drive so `reviewer` runs, then `auto-merge`.
   - **CI-pending** (`statusCheckRollup` has `IN_PROGRESS`/`QUEUED`) → this is the
     one legitimately-transient state; #23's bounded CI wait handles it inside the
     drive, else mark `waiting` and re-poll on the next pass.
   - **request-changes** → #24's review-fix loop drives it inside the orchestrator;
     if `reviewFix.enabled:false` or the finding is structural, surface `needs fix`.
   - **guardrail-held** (forbidden files / oversize / pending human comments /
     unprotected-repo `reviewDecision`-empty) → surface the reason; **never**
     auto-merge; leave for the operator (terminal, not a re-poll).

4. **Drive eligible PRs by dispatching `task-orchestrator` in `pr-open` resume
   mode** (the `Task` tool), one per actionable PR — the same dispatch
   `resume-task` step 5 makes (`resume_mode: pr-open`, `pr_number`, `pr_url`,
   `branch`). This reuses #23 (CI wait), #24 (review-fix loop) and the
   `auto-merge` skill's six guardrails wholesale, so babysit-prs never
   re-implements merge logic the skill's hard refusals forbid duplicating
   (`SKILL.md:207-214`). The read-only triage front-end (step 3) ensures the
   orchestrator is only dispatched for PRs that are actually actionable; the rest
   are surfaced directly.

5. **One idempotent pass per invocation — cross-pass looping is delegated to the
   built-in `/loop`.** babysit-prs runs a **single** enumerate→triage→drive pass
   and returns; it does **not** implement its own sleep/poll loop. Unattended
   operation is the operator wrapping it in Claude Code's built-in `/loop`
   command — `/loop 10m /atelier:babysit-prs` (fixed cadence; stop with `Esc`;
   7-day cap) or `/loop /atelier:babysit-prs` (self-paced: the model picks a
   1 min–1 h delay and ends the loop by not scheduling another once every PR is
   terminal). Because every pass is keyed on the open `task/*` PR set and
   re-derives each disposition from scratch, re-running is safe and convergent:
   merged PRs drop out of the next enumeration and the loop terminates naturally
   when no `task/*` PR remains drivable. The command must therefore be **fully
   idempotent / side-effect-safe to re-run**, and end each pass by reporting
   whether another pass would help (PRs still `waiting (CI)`) or the set is
   terminal (so a self-paced `/loop` knows to stop). It must **not** poll
   internally for harness-tracked work — a dispatched `task-orchestrator` Task
   auto-reinvokes on completion (use that, or Channels for push-based CI events,
   rather than busy-waiting).

6. **Per-PR status report** (the acceptance requires it) — mirror `status.md`'s
   dashboard shape, one line per PR: `merged` / `waiting (CI)` / `needs fix
   (request-changes)` / `held — <guardrail reason>` / `skipped (draft|out-of-band)`.

7. **Strict §6 fidelity.** The only writes are `reviewer`'s review posts and
   guardrail-passing merges. Never touch non-`task/*` PRs, never weaken/skip a
   guardrail, never merge the never-auto-merge set, and honour the per-project
   `decisionPolicy` and the `/atelier:abort-auto` panic switch.

## Acceptance criteria

- [ ] `/atelier:babysit-prs` enumerates **only** open `task/*` PRs (the §16.4
      claim registry; same `gh pr list` filter as `status.md §3` / `next-task`
      step 2), read-only first; draft and non-`task/*` PRs are reported and
      skipped, never driven.
- [ ] Every **eligible** open `task/*` PR is driven to **merged without a manual
      re-invocation**: unreviewed PRs get a fresh-context `reviewer` review;
      approved + CI-green PRs proceed through the `auto-merge` six guardrails and
      squash-merge; the command loops/re-polls until the set reaches a fixed point.
- [ ] The drive **reuses existing machinery** — it dispatches `task-orchestrator`
      in `pr-open` resume mode (re-entering at `reviewer → auto-merge`), composing
      with #23's CI wait and #24's review-fix loop when present — and **does not
      re-implement** review or merge logic, nor weaken/skip any of the six
      guardrails (`auto-merge` stays evaluate-once and authoritative).
- [ ] PRs legitimately held are **clearly reported with the reason**, one status
      line each: `merged` / `waiting (CI)` / `needs fix (request-changes)` /
      `held — <forbidden|oversize|pending-comment|review-gate>` / `skipped`.
      An unprotected-repo `reviewDecision`-empty hold is reported as a terminal
      review-gate hold, **not** an endless re-poll.
- [ ] Each invocation is **one bounded, idempotent pass** (no internal sleep/poll
      loop); the recurring cadence is delegated to the built-in `/loop`
      (`/loop <interval> /atelier:babysit-prs`). The pass ends by reporting whether
      any PR is still `waiting (CI)` (another pass would help) or the set is
      terminal (a self-paced `/loop` can stop). It is **safe to re-run
      idempotently** and concurrently with `next-task`/`resume-task` (all keyed on
      open `task/*` PRs) without double-merging.
- [ ] The only state-changing writes are `reviewer` review posts and
      guardrail-passing merges; the command honours the per-project
      `decisionPolicy` and the `/atelier:abort-auto` panic switch, and **never**
      merges the never-auto-merge set (forbidden files, oversize, pending human
      comments, non-`APPROVED`).
- [ ] babysit-prs works **whether or not #23/#24 have merged** — with neither, a
      pending-CI PR is re-polled across passes and a `request-changes` PR is
      surfaced as `needs fix`; with them, CI is waited on and `request-changes`
      drives the auto-fix loop. No hard dependency on #23/#24's not-yet-merged code.
- [ ] `PLAN.md` §7, `skills/auto-merge/SKILL.md` (lines 72/209), and the new
      command are mutually consistent — the "#25 babysit-prs" forward references
      now point at a real command, with no contradiction with the skill's
      evaluate-once / never-loop-internally invariant (the loop lives in
      babysit-prs/the orchestrator, never inside the skill).

## Risks / open questions

- **`/loop` is a Claude Code built-in (resolved — was a draft error).** An earlier
  draft assumed no `/loop` exists; the repo-wide grep missed it because `/loop` is
  a **harness/built-in command, not an atelier repo skill**. It wraps any slash
  command and re-runs it each iteration (`/loop 10m /atelier:babysit-prs`), in
  either fixed-interval mode (cron-like; `Esc` stops; 7-day cap) or self-paced mode
  (the model picks a 1 min–1 h delay and ends by not scheduling). **Design
  consequence:** babysit-prs owns **one convergent pass**, not an internal loop —
  the recurring cadence is `/loop`'s job. No new `/loop` skill is built and no
  decomposition is triggered; babysit-prs only needs to (1) stay idempotent across
  re-invocations and (2) report whether the PR set is terminal so a self-paced
  `/loop` knows when to stop. (The roadmap sketch's "build on the `/loop` skill"
  is satisfied by *being loopable under* the built-in, not by creating one.)
- **Drive via orchestrator `pr-open` dispatch vs. inline reviewer+auto-merge (key
  design call).** Recommended: dispatch `task-orchestrator` in `pr-open` mode per
  PR — it already composes #23 CI-wait + #24 review-fix + the `auto-merge`
  guardrails, and avoids duplicating merge logic the skill explicitly forbids
  duplicating (`SKILL.md:207-214`). Alternative: babysit calls `reviewer` then the
  `auto-merge` skill directly — thinner per PR, but it re-implements the
  CI-wait/review-fix composition and risks drifting from the orchestrator's path.
  Confirm the dispatch approach.
- **Composition ordering — #23 and #24 are not yet merged.** Both are P2 with
  plans (`.plan/23.md`, `.plan/24.md`; #24 marked ready) but unmerged. babysit-prs
  must, like #24, **carry no hard dependency** and degrade gracefully: without #23
  a pending-CI PR is simply re-polled next pass (the `auto-merge` skill holds on
  pending CI as today); without #24 a `request-changes` PR is surfaced as
  `needs fix`. Confirm babysit ships standalone and merely *composes* with #23/#24
  when they land — the sketch's "compose … once those land" supports this.
- **Termination / fixed-point definition.** With cross-pass looping delegated to
  `/loop`, a pass returns `waiting` for CI-pending PRs and lets `/loop` re-invoke;
  the loop terminates when the open `task/*` set has no drivable PR left. Within a
  single pass, #23's bounded CI wait still applies to a PR being driven through the
  orchestrator. No internal cross-pass poll loop or `maxPasses` constant is needed.
- **Idempotency / concurrency with `next-task`/`resume-task`.** All three use open
  `task/*` PRs as the registry. Two babysit runs, or babysit racing an orchestrator
  that is finishing its own chain, must not double-merge. Rely on `auto-merge`'s
  evaluate-once guardrails + GitHub merge idempotency (a merged PR is closed and
  drops out of the next enumeration); recommend **sequential** per-PR drive within
  a pass (or a small bounded parallelism) to avoid racing shared bookkeeping.
- **Unprotected-repo `reviewDecision`-empty quirk.** On atelier-dev's own
  unprotected repo (and member repos), GitHub leaves `reviewDecision` empty even
  after a real `APPROVED` review, so `auto-merge` guardrail #2 holds. babysit must
  report this as a **terminal** review-gate hold (operator must apply branch
  protection), **not** an endless re-poll — same trap the #23 plan calls out.
- **Reviewer identity / repo access.** The drive reuses the `reviewer` agent,
  which needs `GH_CONFIG_DIR=$ATELIER_CONFIG_DIR/gh/reviewer` and reviewer read
  access. On a private repo where the reviewer isn't a collaborator, guardrail #2
  holds **terminally** (`SKILL.md:58`); babysit must surface that as a terminal
  held state, not a re-poll.
- **Workspace scope.** "across the current project (or workspace)" — default to
  the current project; `--workspace` fans out across members (each member's repo
  enumerated independently via the existing workspace registry). PLAN.md §11 lists
  multi-repo *tasks* as v1-out-of-scope, but this is per-repo PR *watching*, not a
  cross-repo task, so it is arguably in-bounds. v1 could ship project-only and
  defer the fan-out. Flag the scope decision for the product lead.
