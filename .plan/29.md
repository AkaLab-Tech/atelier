# Plan — #29 `atelier-update` hangs at the plugin-cache refresh (`claude plugin update` missing `</dev/null`)

- **Status:** ready (approved — product lead; full scope incl. sibling-helper hardening)
- **Type / priority / estimate:** bug · P2 · ~1.5h
- **blocked_by:** none

## Root cause (verified in-repo)

`refresh_plugin_cache()` in `scripts/atelier-update` (L58–72) runs, for each plugin:

```sh
62    if CLAUDE_CONFIG_DIR="$cfg" claude plugin update atelier@akalab-tech 2>/dev/null; then
...
67    if CLAUDE_CONFIG_DIR="$cfg" claude plugin update claude-roadmap-tools@akalab-tech 2>/dev/null; then
```

`claude plugin update` emits an **interactive confirmation prompt**. Both calls redirect **stderr** to `/dev/null` (`2>/dev/null`) — which *hides* the prompt — but leave **stdin attached to the terminal**. So the command blocks forever waiting on input the operator can't see they owe it. The script runs under `set -euo pipefail` (L25), but the hang is a blocking read, not an error, so neither `set -e` nor the `if`/`else` guard helps. Reproduced live 2026-06-24: a real `atelier-update` run reached `==> refreshing Claude Code plugin cache` → `Checking for updates for plugin "atelier@akalab-tech" at user scope…` and hung with a blinking cursor; the identical command with `</dev/null` appended completed in seconds (`Plugin "atelier" updated from 0.34.0 to 0.35.0`).

`refresh_plugin_cache()` is invoked on **both** of the script's apply paths — the resync path (L383, `$cache_drift && refresh_plugin_cache`) and the pulled-commits path (L541, unconditional). So every `atelier-update` run that needs a cache refresh hangs at its **final** step on any host where `claude plugin update` prompts. By then the `git pull` and template/symlink resync have already succeeded (the on-disk clone is correct), but the Claude Code plugin cache never lands on the new version, so open and future sessions keep loading the **old** plugin until the operator Ctrl-Cs and re-runs by hand — silently defeating the point of the update.

### In-repo precedent for the idiom

`scripts/atelier-uninstall` `step_plugin_uninstall()` (L204–215) already guards its mutation exactly this way:

```sh
209       claude plugin uninstall "$plugin" </dev/null >/dev/null 2>&1
```

with the comment (L205–207) explaining the subshell/capture rationale. This is the established in-repo idiom; `refresh_plugin_cache()` simply omitted the `</dev/null` half.

### Audit of every `claude plugin` call under `scripts/` (`grep -rn 'claude plugin' scripts/`)

Mutating verbs (`update`/`install`/`uninstall`) are the ones that prompt; read-only `list` and `marketplace add` do not block on a confirmation. Classifying each call site:

- **`atelier-update` L62, L67** — `claude plugin update …` — **THE BUG**: no `</dev/null`. Fix required.
- **`atelier-uninstall` L209** — `claude plugin uninstall … </dev/null` — already guarded → leave as-is (the precedent).
- **`atelier-setup-coolify` L65, `atelier-setup-neon` L64, `atelier-setup-vercel` L64** — `claude plugin install "$PLUGIN_ID"` — **no `</dev/null`**: same latent hazard (an install prompt would hang a non-interactive setup run). Harden for consistency (scope item 2).
- **`atelier-setup-{coolify,neon,vercel}` L64/L63/L63** — `claude plugin marketplace add … >/dev/null 2>&1 || true` — `marketplace add` is non-mutating-of-plugins and `|| true`-guarded; no confirmation prompt observed. Harden anyway for belt-and-braces consistency (cheap, same line), but it is **not** load-bearing.
- **`atelier-setup-{coolify,neon,vercel}` L58/L57/L57, `atelier-doctor` L190/L361/L392/L425** — `claude plugin list --json` — read-only query, no prompt → out of scope.
- **`atelier-doctor` L195, L211, L212** — `push_fix_auto "… claude plugin install/update …"` — these are **string literals** printed as suggested fix commands, **not** invocations the doctor runs → out of scope (and the regression-guard matcher must not false-positive on them — see Risks).

## Approach

1. **Fix `scripts/atelier-update` `refresh_plugin_cache()` (required).** Append `</dev/null` to both `claude plugin update` calls (L62, L67), placed before the existing `2>/dev/null`, matching the `atelier-uninstall` L209 ordering (`… </dev/null >/dev/null 2>&1` → here `… </dev/null 2>/dev/null`):
   - L62: `if CLAUDE_CONFIG_DIR="$cfg" claude plugin update atelier@akalab-tech </dev/null 2>/dev/null; then`
   - L67: `if CLAUDE_CONFIG_DIR="$cfg" claude plugin update claude-roadmap-tools@akalab-tech </dev/null 2>/dev/null; then`
   - Add a one-line comment above the calls noting *why* (`# </dev/null: claude plugin update prompts interactively; without it the cache refresh hangs — mirrors atelier-uninstall L209`), so a future cleanup doesn't strip it back. This preserves every existing behaviour (the `2>/dev/null` suppression, the `if/else` OK-vs-warn reporting); it only detaches stdin so the prompt auto-declines/non-interactively proceeds instead of blocking.

2. **Harden the sibling setup helpers (consistency — recommended, low-risk).** In `atelier-setup-coolify`, `atelier-setup-neon`, `atelier-setup-vercel`, append `</dev/null` to the `claude plugin install "$PLUGIN_ID"` call (each ~L64–65) so an install confirmation can't hang a non-interactive `/setup-*` run. **Decision & justification:** include them. They are the only other *mutating* `claude plugin` calls under `scripts/`, they share the identical failure mode (the install prompt is the same class as the update prompt), and the change is one token per file with no behavioural cost on the happy path. The `marketplace add` calls are already `>/dev/null 2>&1 || true` and non-prompting; appending `</dev/null` there too is optional belt-and-braces — **recommended to include** only so the regression guard (item 3) can assert a uniform rule across *all* `claude plugin` mutations without per-call exceptions. (If the product lead prefers the strictly-minimal diff, items 2's `marketplace add` hardening and even the setup-helper `install` hardening can be dropped — see open question.)

3. **Add a structural regression guard** `hooks/tests/atelier-update-plugin-stdin.test.sh` (new), following the established `pass`/`fail` + `REPO_ROOT` convention (mirrors `safe-commit-worktree.test.sh` L17–27) and auto-discovered by `.github/workflows/structural.yml`'s "hooks/tests regression suite (auto-discovered)" step (L41–54) — **no workflow edit** (which also keeps the change off the `.github/workflows/**` auto-merge guardrail; the #28 plan used the same auto-discovery trick).
   - **Static, not hermetic-runtime.** Unlike #28's `gh`-stub test, there is no way to hermetically *exercise* `claude plugin update` (no real `claude` CLI / no network in CI, and the bug is a blocking read that a stub can't faithfully reproduce). So the guard is a **source-level assertion**: scan every `scripts/atelier-*` helper and assert that every *invocation* of `claude plugin (update|install|uninstall)` has stdin redirected from `/dev/null` on the same logical command.
   - **Matcher design (the crux — must avoid false positives):** match lines that *invoke* a mutating verb, i.e. lines containing `claude plugin (update|install|uninstall)` that are **not** (a) inside a heredoc/usage block, (b) a comment, or (c) a quoted suggestion string such as `atelier-doctor`'s `push_fix_auto "… claude plugin install …"`. Concretely: grep `scripts/atelier-*` for `claude plugin \(update\|install\|uninstall\)`, then for each hit **exclude** lines where the match is preceded on the line by `push_fix_auto`, by an odd number of quotes, or by a `#` comment leader, and exclude lines inside `<<EOF … EOF` usage heredocs. For each surviving (real-invocation) line, assert it (or its line-continuation `\`-joined logical command) contains `</dev/null`. Fail listing any offending `file:line`.
   - **Assertions:**
     - (a) `atelier-update` L62 and L67 each carry `</dev/null` (red against the unfixed script — the core regression).
     - (b) the three `atelier-setup-*` `claude plugin install` calls each carry `</dev/null` (guards item 2).
     - (c) the already-correct `atelier-uninstall` `uninstall` call still passes (proves the matcher recognises the precedent, not just the fix).
     - (d) **no false positive**: the `atelier-doctor` `push_fix_auto "… claude plugin install/update …"` literals and the `claude plugin list` read-only calls are **not** flagged (proves the matcher excludes suggestion strings and non-mutating verbs).
   - `set -uo pipefail`; `REPO_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/../.." && pwd)"`; pure `grep`/`bash` — no external deps (so no `jq`/`gh` skip-guard needed). Name ends in `.test.sh` for auto-discovery; `bash -n` on it is also exercised by the syntax-check step (L29). `chmod +x` to match convention (runner uses `bash "$t"`, so not strictly required).

## Affected areas

- `scripts/atelier-update` — **L62 & L67** (required fix: append `</dev/null` to both `claude plugin update` calls in `refresh_plugin_cache()`); one explanatory comment above them. No other behavioural change — the OK/warn reporting and `2>/dev/null` suppression are untouched.
- `scripts/atelier-setup-coolify`, `scripts/atelier-setup-neon`, `scripts/atelier-setup-vercel` — append `</dev/null` to each `claude plugin install "$PLUGIN_ID"` call (~L64–65 each); optionally to the `marketplace add` calls for a uniform rule. Behaviour-neutral on the non-prompting happy path.
- `hooks/tests/atelier-update-plugin-stdin.test.sh` — **new** structural regression test asserting every mutating `claude plugin` invocation under `scripts/` redirects stdin from `/dev/null`; auto-discovered by the structural suite; red against the unfixed `atelier-update`.
- No other files change. `.github/workflows/structural.yml` is **not** edited (auto-discovery). ROADMAP/IN_PROGRESS/HISTORY bookkeeping is handled by the chain/command, not this code change.

## Acceptance criteria

- [ ] `atelier-update` (and the `/atelier:update` wrapper) completes the plugin-cache refresh **non-interactively** — no hang at `==> refreshing Claude Code plugin cache` — on a host where `claude plugin update` would otherwise prompt, landing the Claude Code plugin cache on the clone's version. Both `refresh_plugin_cache()` invocation paths (resync path L383, pulled-commits path L541) are covered by the single fix in the shared helper.
- [ ] Both `claude plugin update` calls in `refresh_plugin_cache()` redirect stdin from `/dev/null`; the existing OK-vs-warn reporting and `2>/dev/null` stderr suppression are unchanged.
- [ ] The sibling `claude plugin install` mutations in `atelier-setup-coolify` / `-neon` / `-vercel` redirect stdin from `/dev/null` (so an install confirmation can't hang a non-interactive `/setup-*` run) — or the product lead has explicitly opted for the minimal diff (see open question), in which case the regression guard's scope is narrowed to match.
- [ ] A new `hooks/tests/atelier-update-plugin-stdin.test.sh` asserts that **every** `claude plugin (update|install|uninstall)` *invocation* under `scripts/` redirects stdin from `/dev/null`; it excludes read-only `list`/`marketplace add` calls and quoted suggestion strings (`atelier-doctor` `push_fix_auto`); it is auto-discovered and green in `.github/workflows/structural.yml`, and fails (red) against the unfixed `atelier-update`.

## Risks / open questions

- **Regression-guard matcher fidelity (primary open question).** A naive `grep 'claude plugin'` over `scripts/` would false-positive on (i) `atelier-doctor`'s `push_fix_auto "… claude plugin install/update …"` suggestion **strings** (printed, never executed) and `atelier-update`'s own usage heredoc (L204/L219/L234), and false-negative-protect against (ii) the read-only `claude plugin list`/`marketplace add` calls that don't prompt. The matcher must restrict to the **mutating verbs** (`update|install|uninstall`) **as invocations** (not inside comments, heredocs, or quoted literals). **Question for the product lead:** is a `grep`-based static guard with these exclusions acceptable as the regression contract (recommended — it is hermetic, version-independent, and directly encodes "every mutation has `</dev/null`"), or do you want a runtime guard instead? A faithful runtime reproduction isn't feasible (no `claude`/network in CI, and reproducing a *blocking read* in a stub is fragile), so the static guard is the recommended path; the implementer should confirm the test is **red** against the unfixed `atelier-update` before claiming green.
- **Scope of the sibling hardening (decision to confirm).** Item 2 widens the diff to three setup helpers for *consistency*, though the live-reproduced hang is only in `atelier-update`. Recommended to include (one token per file; the install prompt is the same hazard class) and it lets the regression guard assert one uniform rule. **If you prefer the strictly-minimal bug-only diff**, drop the setup-helper edits and narrow the test to `atelier-update` (and `atelier-uninstall` as the passing precedent); the setup helpers then keep the latent hazard until a future consistency pass. Either way the test scope must match the chosen surface.
- **`</dev/null` semantics on the update prompt.** Detaching stdin makes `claude plugin update` read EOF instead of a TTY. The expectation (from the live repro) is that it then proceeds/auto-confirms non-interactively and completes — exactly as `atelier-uninstall` relies on for `uninstall`. The implementer should confirm on a real host that `</dev/null` yields a *successful* update (not a silent decline), since the whole point is landing the new version; if a given `claude` build instead *aborts* on EOF, the OK/warn `if`/`else` already degrades gracefully (warn + continue) rather than hanging — so the hang is fixed regardless, but the cache-lands-on-new-version acceptance depends on the auto-proceed behaviour observed in the repro.
