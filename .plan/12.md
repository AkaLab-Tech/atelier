## Approach

Rather than write one implementation plan for 6 unrelated ideas (which `task-decomposer` would reject as not-actually-one-task, and which the size budget wouldn't tolerate as a single PR regardless), split visibility per idea without fabricating detail that doesn't exist yet:

1. v2.1 — Skill auto-injector hook (`UserPromptSubmit`) to load skills by context signals.
2. v2.2 — Router skill with subcommands (`/atelier setup|doctor|update|reconfigure`).
3. v2.3 — `PermissionRequest` Bash hook for dynamic permissions, replacing static `settings.template.json`.
4. v2.4 — Project-memory hooks (`SessionStart` + `PostToolUse`) to auto-persist project learnings.
5. v2.5 — `/learner` + `/skillify` to extract reusable patterns from successful tasks.
6. v2.6 — Node.js hook dispatcher (`scripts/run.cjs`) for portable, fail-open hook execution.

None of these get a `.plan/12<letter>.md` file or a `Ready` flip in this pass — a one-line idea has no approach/acceptance to approve, so "planning" one would mean the planner *inventing* scope the product lead hasn't actually specified. That is a different failure mode than the deferred-but-fully-specified items (`#14`, `#15`): those have complete approach/acceptance already; these do not.
