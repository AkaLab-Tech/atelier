## Approach

This is the heart of M9.4: build the **single** `backend → files` reverse mechanism once, parameterized by source backend, so `github-project → files` (#21b) and `linear → files` (#21c) are literally the same code path. There is strong prior art to lift from, not invent:

- The forward remote steps `5b` (`files → linear`) and `5c` (`files → github-project`) in `commands/migrate-roadmap.md` (lines 65–144) — the **safety template** (bucket-ordered processing, partial-failure stop-immediately, atomic `.roadmap.json` checkpoint, orphan-state precondition at step 1, line 29).
- The **Mirror auto-refresh on activation** procedure in `skills/roadmap-tracking-flow/SKILL.md` (§533–620) — this is *already* a remote→local-files reconstruction driven by `listTasks` across the three buckets, with `backendId` coherence (step 4, line 561), per-bucket safe-failure (step 5, line 566), and a history window. The reverse migration generalizes this from "refresh a read-only mirror" into "full reconstruct + flip authority to local files."

Work, in order:

1. **Add a new step `### 5d. <remote backend> → files (indexed)`** to `commands/migrate-roadmap.md`, sibling to 5b/5c. It is invoked when the source backend (resolved at step 1 from `.roadmap.json`) is a remote backend and target is `files`. Written **backend-agnostically** — every backend-specific operation is "call the active backend's `listTasks` / `getTask`", never a Linear- or GitHub-specific call. The sub-steps:
   1. **Precondition + availability gate.** Require `.roadmap.json` present with a remote `backend`. Call the active backend's `isAvailable()`; if `false`, refuse (cannot reconstruct from an unreachable source) — do not half-write. Require a clean working tree for reviewability (same prompt as step 1, line 30).
   2. **Pull all three buckets** via `listTasks("history")`, `listTasks("in_progress")`, `listTasks("roadmap")`. For each task, also resolve the fields the forward path wrote: `type`, `estimate`, `#id`/`TASK_NNN` handle, `Ready`, `blocked_by`, and `backendId`. (`getTask(id)` enriches body/metadata where `listTasks` is shallow.)
   3. **Show the full reconstruction plan** before writing anything (mirrors 5b.3 / 5c.3): every task grouped by destination bucket, the `roadmap/TASK_NNN_*.md` files that will be (re)created, and the explicit note that `.roadmap.json` will be **removed** at the end (the authority flip). Confirm or abort with zero side effects.
   4. **Reconstruct the indexed `files` layout into a staging set**, then commit it to the tree atomically (write all files, then the index files, as one change set — no half-written tree):
      - Per task → `roadmap/TASK_NNN_<slug>.md` with the body and YAML frontmatter. **Preserve identity**: keep the `TASK_NNN` human handle (from the `#id` field the forward path stored) and carry `backendId` only if a round-trip handle is wanted — but since the destination is `files` (the local layout becomes authoritative), strip `backend`/`backendId` frontmatter so the resulting tree is a clean `files` project (document this as the deliberate authority-flip, parallel to how the forward path *writes* `backendId`).

   5. **Remove `.roadmap.json`** as the **inverse atomic checkpoint** — its *absence* after a clean reverse migration means the local `files` layout is now authoritative (exactly inverting "presence = remote is authoritative" from 5b.6 / 5c.6, lines 111 / 140).
   6. **Partial-failure guard.** If any `listTasks`/`getTask` fails mid-pull, or any file write fails: **stop, write nothing destructive, and never touch the remote source.** Leave `.roadmap.json` in place (so the project is still validly remote-backed) and surface which buckets succeeded vs failed (mirrors SKILL.md §566 per-bucket safe-failure + 5b.5/5c.5 stop-immediately). The reverse path is read-only against the remote — there is nothing to clean up remotely, which is *safer* than the forward path; state that explicitly.

## Risks / open questions

- **Frontmatter on the reconstructed files — keep or strip `backendId`?** Plan recommends **strip** (clean `files` project after authority flip), which differs from the offline-mirror case where `backendId` is retained. The product lead should confirm: stripping means a later re-migration to the *same* remote would create duplicate items (no `backendId` to reconcile against) — but that is the correct semantics for "this project is now files-native." If round-trip back to the original remote must be deduplicating, we'd keep `backendId`. Recommend strip + document.
- **`HISTORY.md` reconstruction from a bounded history window.** The remote `history` bucket may be large; the Mirror-refresh path bounds it with `historyWindow`. For a *migration* (one-shot, authoritative), the plan assumes **`historyWindow: "all"` semantics** (pull everything — a migration must be lossless, not windowed). Confirm: a windowed reverse migration would silently drop old history, violating the lossless acceptance.
- **Clean-tree atomicity in pure markdown.** "Atomic write to a clean tree" is enforced by convention (write-all-then-index as one change set, reviewable as a single diff), not a transaction. This matches how `FilesBackend` already defines atomicity (RoadmapBackend.md line 119). Acceptable, but worth stating so the implementer doesn't over-engineer.
- **`#21b`/`#21c` are blocked on this** — keep the 5d step's interface (how a direction routes into it) explicit so the two consumers only add a matrix row + a routing line, staying under budget.
