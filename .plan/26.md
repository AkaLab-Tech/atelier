# Plan — #26 Auto post-merge cleanup completion (base pull + orphan sweep)

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** feat · P2 · ~2–3h
- **blocked_by:** none (#23/#24/#25 are merged/Done)

## Approach

Extend the `auto-merge` skill's existing **§ Post-merge cleanup** (`skills/auto-merge/SKILL.md`, lines 150–168) with two new steps that run **only after a successful squash-merge**, reusing existing surfaces rather than hand-rolling git:

1. **Fast-forward the local base branch** (new post-merge step, after worktree removal). The squash commit now lives on `origin/<base>`; the operator's local base ref is left stale until the next `/next-task` fetch. Close that gap, guarded so a dirty or diverged base is **never** force-updated:
   - Resolve `<base>` with the same policy `/next-task` step 1 and `git-wt` use: `dev` if `git ls-remote --heads origin dev` is non-empty, else `main` (atelier-dev resolves to `main`).
   - `git -C <project_root> fetch origin <base>` to refresh `origin/<base>` (this updates the remote-tracking ref unconditionally and safely).
   - Then bring the **local** `<base>` ref forward, branching on the main worktree's checkout state:
     - If `<base>` is the branch checked out in the main worktree **and** the worktree is clean (`git -C <project_root> status --porcelain` empty): `git -C <project_root> merge --ff-only origin/<base>`.
     - If `<base>` is not checked out in any worktree: `git -C <project_root> fetch origin <base>:<base>` (refspec form is fast-forward-only by construction).
     - If the worktree is dirty, on a different branch that can't be ff'd, or `merge --ff-only` exits non-zero (diverged): **skip and surface** a note (`base <base> not fast-forwarded: <reason>`). Never `--force`, never `reset --hard`, never stash the operator's checkout. Idempotent: an already-current base is a no-op success.

2. **Orphan `task/*` sweep** (new post-merge step) — delegate to the existing `atelier-housekeeping` binary (`scripts/atelier-housekeeping`) scoped to this one project. **Do not duplicate** the enumeration/safety logic — the binary already enforces every required rail (protected branches, open PRs, dirty worktrees, active/`[BLOCKED]`/`[OVERSIZE]` tasks via `IN_PROGRESS.md`, unmerged-work → "needs review" skip, no `--force`/non-ff push):
   - Invoke `atelier-housekeeping --project <project_root> --yes --no-stamp` (no `--include-unmerged`, so unmerged orphans are only reported, never deleted). `--no-stamp` keeps the per-task sweep from suppressing the operator's daily `/atelier:housekeeping` nudge.
   - The merge just deleted *this* task's remote branch via `gh pr merge --delete-branch`; the sweep mops up **prior cycles'** local + remote `task/*` debris. It is naturally idempotent — re-running enumerates fresh and exits `OK: nothing to clean up`.
   - Capture the binary's stdout (or run it once with `--json` first to count, then `--yes`) and fold the swept counts into the auto-merge structured report.

3. **Gate both behaviours behind a `.atelier.json` config block** so the behaviour is opt-out-able and the default is explicit:
   - Add a `postMergeCleanup` block to the root `.atelier.json` with `fastForwardBase` and `sweepOrphans` booleans (proposed defaults below — see Open questions). Read it in the skill the same way `auto-merge` already reads `prSize`/`reviewFix.enabled` and the orchestrator reads `ciWait`/`reviewFix` (absent block → documented default).
   - Document the block (with its `_comment`) alongside the existing `taskDecomposer`/`prSize`/`ciWait` blocks.

4. **Surface the outcomes** in the orchestrator's close-out report (`agents/task-orchestrator.md` step 8, the `auto-merge reports merged` bullet at line 224) and in the skill's structured-output template (lines 170–202) and Hard refusals (lines 204–214).

The sweep stays **enumerate-then-act** because the binary already works that way; the skill never calls `git worktree remove` / `git branch -d` / `git push --delete` itself — it shells out to the binary, preserving the single auditable deletion path.

## Affected areas

- `skills/auto-merge/SKILL.md` — add two steps to **§ Post-merge cleanup** (base ff + sweep delegation), read `postMergeCleanup` config, extend the **§ Structured output** template with `Base:` and `Swept:` lines, and add Hard refusals (never force-update a dirty/diverged base; never hand-roll the sweep — always delegate to `atelier-housekeeping`).
- `.atelier.json` (project root) — new `postMergeCleanup: { fastForwardBase, sweepOrphans }` block with a `_comment`. (Operator-owned config; editable by the implementer — it is not on the forbidden-edit list with `package.json`/workflows.)
- `agents/task-orchestrator.md` — step 8 (`auto-merge` `merged` close-out, line 224): include base-ff status and swept counts in the report to the operator.
- `scripts/atelier-housekeeping` — **likely no change.** It already supports `--project`, `--yes`, `--no-stamp`, `--json`, and reads `ATELIER_AUTO` for non-interactive mode. Touch only if a terser `--quiet` post-merge output proves necessary; prefer reusing the existing flags.
- Docs (if a doc enumerates the post-merge cleanup steps, e.g. under `docs/`) — keep the description in sync. Verify during implementation; do not add a new doc speculatively.

## Acceptance criteria

- [ ] After a successful auto-merge, the local base branch (`dev` if `origin/dev` exists, else `main`) is fast-forwarded to include the just-merged squash commit **without a manual step**.
- [ ] The base fast-forward is **guarded**: a dirty main worktree, a non-base checkout that can't be ff'd, or a diverged base causes the step to **skip with a surfaced note** — never a `--force`, `reset --hard`, stash, or any mutation of the operator's working tree.
- [ ] After a successful auto-merge, orphan `task/*` branches (local **and** remote) left by prior cycles are swept **without a separate manual `/atelier:housekeeping` run**, by delegating to `atelier-housekeeping --project <root> --yes --no-stamp`.
- [ ] The automatic sweep honours every existing safety rail: never touches active/`[BLOCKED]`/`[OVERSIZE]` tasks, open PRs, dirty worktrees, or protected branches; never deletes unmerged work (no `--include-unmerged`); never a `--force`/non-ff push.
- [ ] Both steps are **idempotent**: re-running after everything is already clean/current is a no-op success (base already up to date; sweep prints `nothing to clean up`).
- [ ] Both behaviours are gated by a `.atelier.json` `postMergeCleanup` block (documented with a `_comment`); an absent block falls back to the documented default, and setting a flag `false` disables that step.
- [ ] The auto-merge structured report and the orchestrator's `merged` close-out surface what was fast-forwarded and what was swept (counts), so the operator sees the cleanup that happened.
- [ ] Neither step runs when `auto-merge` reports `held` — cleanup completion only fires on a successful merge.

## Risks / open questions

- **Default for `sweepOrphans` (product-lead call).** An automatic sweep after *every* merge performs **remote `task/*` deletions** more aggressively than the operator-invoked daily `/atelier:housekeeping`. The operator's stated preference is full autonomy (memory: "operator wants autonomous atelier", "merge autonomy full"), which argues for `true` by default. The conservative alternative is `sweepOrphans: false` by default (operator opts in), with `fastForwardBase: true` (purely local, zero-risk). **Recommendation:** `fastForwardBase: true`, `sweepOrphans: true` by default given the autonomy preference — but confirm, since this changes remote state automatically.
- **`--yes` bypasses housekeeping's "always ask first" contract.** `commands/housekeeping.md` is interactive by design. Invoking the binary with `--yes` from the merge chain is justified by the auto-merge "the gate is the consent" authorization model, but it means the per-task sweep is *less* interactive than the daily command. Worth an explicit note in the skill so the asymmetry is intentional and documented, not accidental.
- **`--no-stamp` vs. the daily nudge.** Using `--no-stamp` keeps the post-merge sweep from silencing the daily `/atelier:housekeeping` cadence stamp (`housekeeping-last-check`). Confirm that's the desired interaction (the alternative — stamping on every merge — would suppress the daily nudge whenever any task merged that day, which is probably *also* fine, but changes operator-visible behaviour).
- **Base resolution is duplicated in three places** (`/next-task`, `git-wt`, and the housekeeping binary's hardcoded `main`). The new ff step adds a fourth consumer of the `dev`-else-`main` rule. Not in scope to refactor into a shared helper, but flag it: if `origin/dev` is the base, the binary's `branch --merged main` / `PROTECTED_BRANCHES` would need to agree — for atelier-dev (`main`) this is moot, but a `dev`-based project would expose the inconsistency. Consider scoping #26 to `main`-base correctness and noting the `dev` gap as a follow-up.
- **Where the ff runs.** The base ff must target the **main worktree** (`project_root`), not the task worktree (which auto-merge has just `git wt rm`'d). The skill must use `git -C <project_root>` explicitly and detect the main worktree's checked-out branch before choosing `merge --ff-only` vs. the `<base>:<base>` refspec.
- **Sweep latency.** `atelier-housekeeping` calls `gh pr list` per branch to classify PR state; on a project with many stale `task/*` refs this adds noticeable time to every merge. Acceptable for a P2 convenience, but if it becomes a drag the sweep could be made `--report`-then-act or rate-limited. Note, don't pre-optimise.
