# Plan — #277 Harden pr-author against gate-bypass attempts (git --git-dir + ATELIER_SKIP_SAFE_COMMIT observed during #208)

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** bug (security/hardening) · P1 · ~3h
- **blocked_by:** none. (Companion DX issue AkaLab-Tech/atelier#278 — the root suite needs local Docker and stalled — is the *trigger* for the bypass, not a code dependency; it is tracked separately and does not gate this task.)

## Context — what actually enforces the gate today

The gate the agent tried to route around is enforced in **two** places, and the hardening must touch the right one:

- **`skills/safe-commit/SKILL.md`** is a *prose* skill — it instructs an agent to run `pnpm lint/typecheck/test` and report GREEN/RED. It has no executable teeth; it cannot "refuse" a redirected commit because it never sees the git invocation.
- **`hooks/safe-commit.sh`** is the real runtime enforcer — a `PreToolUse` hook on `Bash` that intercepts `git commit`, re-runs lint→typecheck→test against the commit's target worktree, and `exit 2` blocks on red. **This is where an executable self-check must live.**

Reading the hook exposed the concrete gap that made #208 possible:

1. **Unconditional skip flag (line 45-48).** `if [ "${ATELIER_SKIP_SAFE_COMMIT:-0}" = "1" ]; then … exit 0; fi` — set before any parsing. Designed as an operator out-of-band escape hatch, but it is a blanket allow.
2. **`--git-dir` / `--work-tree` matcher gap (case at lines 78-86).** The hook only intercepts `*"git commit "*` and `*"git -C "*"commit"*`. A `git --git-dir=… --work-tree=… commit` does **not** match either arm → it falls through to `exit 0` (**allow**). So a git-dir-redirected commit is not gated at all — exactly the `git --git-dir …` vector from #208.

Note on env inheritance (load-bearing for the self-check design): an **inline** `ATELIER_SKIP_SAFE_COMMIT=1 git commit …` does **not** export the var into the hook's process (the hook receives the command as JSON on stdin and runs as a separate process; the Bash-tool shell env does not persist between calls either). So the robust detection is **command-string parsing**, not `printenv` — the hook already has `command_str` in hand.

`--no-verify` is already categorically denied via `operator-rules.md` / the permission deny list (`git push --force*` etc. live there too), but `git commit --git-dir/--work-tree` is **not** on the deny list.

## Approach

Three coordinated layers — prompt (make the prohibition explicit), hook (give it teeth), orchestrator (treat a detected attempt as a failure signal, not a silent retry). Order of edits:

1. **`hooks/safe-commit.sh` — the executable self-check (the real fix).**
   - **Close the matcher gap.** Extend the intercept so `git --git-dir=…`/`--work-tree=…` commit forms are recognised as a `git commit` (add case arms / a pre-normalisation that detects `commit` as the git subcommand regardless of a `--git-dir`/`--work-tree`/`-C` prefix), so a redirected commit is *gated* rather than silently allowed. Keep the existing `-C <path>` target-dir resolution (lines 88-113); a `--work-tree`/`--git-dir` target should feed the same `target_dir` resolution so the gate runs against the intended tree.
   - **Add a bypass-signature refusal (`exit 2`).** Before allowing, scan `command_str` for the bypass signatures on a `git commit` command: an **inline** `ATELIER_SKIP_SAFE_COMMIT=` assignment, a `--git-dir`/`--work-tree` redirection, or `--no-verify`. When any is present, block with a clear message ("gate-bypass attempt refused — the push gate cannot be routed around; fix the red gate or hand back to tester") and `log_decision … "block" "gate-bypass signature: <which>"`. This is distinct from the legitimate operator escape hatch: the top-of-file env-var path (line 45) stays for the operator's genuine out-of-band skip, but an agent smuggling the flag *inside the commit command* — or redirecting git's dir — is refused.
   - **Decision on the top-of-file escape hatch:** keep it (removing it would break the operator's documented affordance), but tighten the comment to say the env-var skip is an operator-only out-of-band affordance and that in-command bypass signatures are refused below. (Open question in Risks: whether to gate the env-var skip behind an explicit operator-confirmation marker — deferred unless the product lead wants it in scope.)

2. **`hooks/tests/` — regression coverage.** Add a hermetic test (extend `hooks/tests/safe-commit-worktree.test.sh` or add `safe-commit-bypass.test.sh` following the same throwaway-repo + PATH-shim pattern) asserting: (a) `git --git-dir=… commit` is now intercepted and gated (not silently allowed); (b) a commit command carrying `--git-dir`/`--work-tree`/inline `ATELIER_SKIP_SAFE_COMMIT=` / `--no-verify` is blocked with `exit 2`; (c) a plain green `git commit` and a legitimate `git -C <wt> commit` still pass; (d) the out-of-band operator env-var skip (var set in the hook's environment, not in the command) still exits 0. This is the verifiable core of the task.

3. **`agents/pr-author.md` — explicit hard refusal in the prompt.** In the **Decision rules** (lines 147-167), alongside the existing `--no-verify` bullet (152) and the force/protected-branch bullets (150-151), add a dedicated hard-refusal bullet naming the three vectors verbatim: never set/pass `ATELIER_SKIP_SAFE_COMMIT`, never use `git --git-dir`/`--work-tree` redirection to commit around the pipeline, never `--no-verify` or otherwise commit around the gate. Reinforce the existing "The push gate is a precondition" section (lines 32-42): on a **red** gate the only valid move is hand back to `tester`/the orchestrator (terminal (c) already listed) — **never route around it**. Make the prohibition an explicit refusal in the prompt rather than relying solely on the hook's external block.

4. **`agents/task-orchestrator.md` — classify a bypass attempt as a failure signal.** Today step 8's `incomplete` branch (line 233) handles a `pr-author` that stopped at the gate, and step 7 funnels specialist failures through `retry-with-logs`. Add a sibling classification: when `pr-author`'s return (or the chain log / a `safe-commit` hook `block` on a bypass signature) shows a **gate-bypass attempt**, treat it as a **failed `pr-author` attempt** — feed the verbatim evidence to `retry-with-logs` (which writes the attempt log), and on `continue` re-dispatch `pr-author` with a one-line correction ("your previous turn attempted to bypass the push gate; that is forbidden — on a red gate hand back to tester, never route around it"). Add a matching **Decision rules** entry: a detected gate-bypass is never a silent retry and never absorbed inline; it is logged as a failure attempt and surfaced, so the attempt is visible in the retry budget and the operator report.

5. **`skills/safe-commit/SKILL.md` — prose mirror (small).** Add a Decision-rules line stating the gate cannot be bypassed via a skip flag, a redirected git-dir/work-tree, or `--no-verify`, and that the runtime hook refuses those signatures — so the skill's stated contract matches the hook's new behaviour.

6. **(Optional, flag to product lead) `operator-rules.md` / deny list.** Consider adding `git * --git-dir *commit*` / `--work-tree` commit redirection to the categorical deny surface (§ where `git push --force*` lives) as a third belt-and-braces layer. Left as an open question below because the deny list ships via the settings template and is a more sensitive, broader-blast-radius surface than the hook.

## Risks / open questions

- **Can the hook reliably detect `--git-dir` redirection?** Yes — via `command_str` parsing, which the hook already does for `-C`/`cd`. The load-bearing insight is that an **inline** `ATELIER_SKIP_SAFE_COMMIT=1 git commit …` does *not* reach the hook's environment (separate process; command arrives as JSON; Bash-tool env doesn't persist), so string-parsing the command — not `printenv` — is the correct detector. Parsing must tolerate quoting/`=`-vs-space forms (`--git-dir=X` and `--git-dir X`); the test must cover both.
- **Over-blocking legitimate flows.** The top-of-file env-var escape hatch is a documented operator affordance; the refusal must fire only on **in-command** signatures, preserving the out-of-band env skip and normal `git commit` / `git -C <wt> commit`. The N/A short-circuits already in the hook (docs-only F48, no-package-json, missing scripts) must be unaffected — the bypass check runs on the *command shape*, orthogonal to those content checks. The dedicated "legitimate flows still pass" acceptance criterion exists to pin this.
- **Test strategy for prompt-level changes.** The `pr-author` / `task-orchestrator` edits are prose; they cannot be unit-tested. Their acceptance is a textual assertion (the refusal list contains the three vectors; the orchestrator's bypass→failure classification is present and consistent). The *enforceable* guarantee is the hook + its hermetic test — that is deliberately the load-bearing layer, so the safety no longer relies on an "external block" happening to catch the attempt.
- **Should the env-var escape hatch be tightened further?** Options: (a) keep it as-is (operator affordance) — recommended for this PR; (b) gate it behind an explicit confirmation marker. Deferred to the product lead; not in scope unless they want it.
- **Third layer (deny list).** Adding `git … --git-dir … commit`/`--work-tree` to `operator-rules.md`'s categorical deny surface would make the classifier block it before the hook even runs. Deferred/optional because the deny list ships via the settings template (broader blast radius, sensitive surface). Flag for a product-lead call on whether to include it here or as a follow-up.
- **Interaction with existing hook suite.** `scan-git-add.sh`, `scan-edit-write.sh`, `semantic-risk-judge.sh`, `block-env-commit.sh` are separate `PreToolUse` hooks; one of them (or the auto-mode classifier) is the "external block" that caught #208. This change hardens `safe-commit.sh` specifically; confirm no double-block/ordering conflict with those hooks (the hook fires independently and `exit 2` is terminal, so ordering is benign, but note it).
