# Plan — #35 Permission matrix: colon-refspec `git push origin task/x:main` bypasses the protected-branch deny

- **Status:** ready (approved — product lead, batch approval via AskUserQuestion, 2026-07-07)
- **Type / priority / estimate:** bug · P2 (latent — requires an agent to construct a colon-refspec, which the normal flow never does) · ~1.5h
- **blocked_by:** none

## Approach

The static deny matrix requires the protected-branch name as a space-separated arg (`Bash(git push * main|master|develop|staging)`), so a colon-refspec destination (`git push origin task/x:main`, or the `+`-prefixed force form `git push origin +HEAD:main`) slips past the space-requiring deny while still landing on a protected branch — and the `task/*` allow-list entry matches the *source* side of the refspec, widening the gap rather than closing it.

1. **Harden the deny patterns** to catch refspec-shaped destinations: add `Bash(git push * *:main)`, `*:master`, `*:develop`, `*:staging` (and their `+`-prefixed force variants) to `permissions.deny` in `templates/settings.template.json`.
2. **Tighten the `task/*` allow** so its match requires the *entire* argument to be a bare `task/<slug>` ref (no embedded `:`), so a colon-refspec can never satisfy the allow rule even incidentally.
3. **Origin-only colon-delete gap** (same review, minor): the existing `Bash(git push origin :*)` deny is origin-only; note it's acceptable since atelier worktrees only ever carry `origin` as a remote (document this assumption explicitly rather than adding a remote-agnostic rule that could over-match).
4. **Regression tests**: extend the permission-matrix hermetic suite with cases for colon-refspec-to-protected and `+`-refspec force — both must be denied by the static matrix (not fall through to the classifier), while the legitimate `git push origin task/*` and `--force-with-lease origin task/*` flows still pass.

## Affected areas

- `templates/settings.template.json` — `permissions.deny` / `permissions.allow` patterns.
- `hooks/tests/` — new or extended permission-matrix regression test.
- Any already-instantiated per-project `.claude/settings.json` templates are **not** touched by this PR (per-task settings regenerate from the template each task — see `CLAUDE.md`); this is a template-only fix that takes effect on the next `/setup-project` / `atelier-setup-project --per-task-settings` run.

## Acceptance criteria

- [ ] `git push origin task/x:main` and `git push origin +HEAD:main` are denied by the static matrix (verified via the permission test harness, not by observing the classifier).
- [ ] `git push origin task/<slug>` and `git push --force-with-lease origin task/<slug>` still pass.
- [ ] New regression test cases land in the existing hermetic suite and are covered by CI's structural workflow (auto-discovered, no workflow edit).

## Risks / open questions

- **Latent, not exploited.** No known agent behavior constructs a colon-refspec today; this is a defense-in-depth hardening, not a fix for an observed incident. Priority reflects that (P2, not P0/P1).
- **Pattern-matching false positives.** Claude Code's `Bash(...)` allow/deny pattern syntax is limited (glob-like, not full regex) — the implementer should verify the proposed `*:main` pattern actually matches the colon-refspec shape in the harness's matcher before relying on it; if the syntax can't express this precisely, the fix may need to move to a `PreToolUse` hook check instead (mirroring how `semantic-risk-judge` or `block-env-commit` do content-based checks beyond what the static `Bash()` patterns can express).
