## Acceptance criteria

- [ ] For a `github-project` repo, `/atelier:next-task` obtains the backlog via `listTasks("roadmap")` (not by reading local `ROADMAP.md`) and passes the list to `task-discovery`.
- [ ] Selection applies the same P0→P1→P2 / `blocked_by` / planning-gate / `[OVERSIZE]`/`[BLOCKED]` rules to the Project-sourced list, with priority derived from the Project's declared priority-field convention.
- [ ] The planning gate for a non-`files` backend is satisfied iff the backend's `Ready` field is set **and** a committed `.plan/<id>.md` exists; a `Ready`-set item with no committed plan surfaces the `ready-without-plan` inconsistency.
- [ ] The chosen task is enriched via `getTask(id)` and the claim is recorded via `moveTask(id, "roadmap", "in_progress")` — **never** editing local `ROADMAP.md` / `IN_PROGRESS.md` for this backend.
- [ ] The claim registry stays the set of open `task/*` PRs regardless of backend; the Project's "In Progress" column is not the concurrency signal.
- [ ] The `files` backend path is unchanged.

## Risks / open questions

- **Where the `Ready`-field read is actually performed.** The planning-gate filter (task-discovery step 4) currently checks a `[ready]` text token + a committed `.plan` file. For github-project the token becomes the backend's `Ready` field — but reading that field requires a backend call. Open question: does `listTasks`/`getTask` already surface the `Ready` field in its returned record (so the skill reads it from the list, no extra call), or must the skill make a dedicated read? The implementer should confirm against crt's `docs/RoadmapBackend.md` + `GitHubProjectBackend` shape (M9.2) and document the source. This is the seam most likely to need a small contract clarification with crt.
- **Coupling to #20c.** The `Ready`-field semantics this slice references in selection are *set* by #20c (plan-task). The prose here must describe reading the field; #20c describes writing it. Keep the two consistent — both cite §16.5. Since #20b and #20c are independent (both only depend on #20a), the implementer of whichever lands second should re-read the sibling's committed plan to keep the `Ready`-field wording aligned.
- **No live e2e here.** A true end-to-end claim against a real GitHub Project is M9.5 (#22) territory; this slice is prose correctness against the M9.2 contract. The acceptance criteria are verifiable by review against `docs/RoadmapBackend.md` + the already-merged routing, not by a live OAuth run.
