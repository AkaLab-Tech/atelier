# Plan — #31 feat: /setup-project detects missing branch protection + offers an autonomous fix

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** feat · P1 · ~3h
- **blocked_by:** none

## Approach

Add a branch-protection probe-and-offer step that mirrors the existing
`step_reviewer_access()` in `scripts/atelier-setup-project` (lines 1146–1234) — same
detect / admin-check / headless-flag / interactive-prompt / re-verify shape, same
advisory-never-fails posture. Factor the read-only **classification** into a small
shared helper so `atelier-doctor` can reuse it without copying the apply path.

The classification reads the default branch's protection and returns one of four
states (matching AC #1):
- `protected-sufficient` — `required_pull_request_reviews.required_approving_review_count >= 1`.
- `protected-insufficient` — protection exists but no required approving reviews
  (the empty-`reviewDecision` trap that holds auto-merge guardrail #2 forever).
- `unprotected` — `gh api .../protection` returns 404 (Branch not protected).
- `no-admin` — the API returns 403 (token lacks repo-admin; cannot read or apply).

Implementation order:
1. **Shared classifier helper** in `scripts/atelier-setup-project` (used by the
   setup step) and an equivalent in `scripts/atelier-doctor`. Resolve `owner/repo`
   and the default branch via the author identity exactly as `step_reviewer_access`
   does (`env GH_CONFIG_DIR="$author_dir" gh repo view --json nameWithOwner,defaultBranchRef`).
   Probe with `gh api repos/{owner}/{repo}/branches/{branch}/protection`, distinguishing
   404 (unprotected) from 403 (no-admin) from 200 (parse the JSON).
2. **`step_branch_protection()`** in `atelier-setup-project`: classify, then on
   `protected-insufficient` / `unprotected` explain the guardrail-#2 link and offer
   the apply. Apply path (AC #3) is an idempotent
   `gh api -X PUT repos/{owner}/{repo}/branches/{branch}/protection` (author identity,
   admin) with a **merged** payload that sets
   `required_pull_request_reviews.required_approving_review_count=1`,
   `required_status_checks=null` (never invent checks that don't exist),
   `enforce_admins=false` (so the `AtelierAuthor` bot can still squash-merge its own
   approved PRs), `restrictions=null`. On `no-admin` (AC #5) do **not** fail — print
   the exact manual `gh api -X PUT ...` command and the admin-token note.
3. **Policy / flag gating** (AC #4): mirror `--grant-reviewer` with a new
   `--apply-branch-protection` headless opt-in flag, plus the `$NONINTERACTIVE` /
   TTY guards used by `step_reviewer_access` and `step_decision_policy`. Read
   `.atelier.json` `decisionPolicy` for the `branch-protection` category (see open
   question) to decide auto-apply vs present-and-wait. Add `BRANCH_PROTECTION_STATUS`
   to the summary block (around line 1375) and call `step_branch_protection` in the
   main flow (after line 1349). Register the new flag in the arg parser (near line
   203) and `--help` text (near line 80).
4. **Doctor check** (`check_branch_protection`) in `scripts/atelier-doctor`: run the
   classifier, `push_host` a `✓ / ✗ / ↷` row, and on a gap register the apply
   command via the doctor's fix-block mechanism (`FIX_BLOCKS` / `FIX_AUTO_COMMANDS`,
   lines 95–126) — runnable under `--fix` only when admin is present, manual-only
   otherwise. Wire it into the host-checks run alongside the other `check_*` calls.
5. **Docs**: update `commands/setup-project.md` (Phase-1 step list + a new
   `--apply-branch-protection` entry in the argument-hint/contract) and add a row to
   `commands/doctor.md`'s reference output so the operator sees the new check. No
   logic moves into the slash commands — both binaries own the work, per their
   established "all logic lives in the bash binary" contract.

## Acceptance criteria

- [ ] `/setup-project` and `/doctor` probe the default branch via
      `gh api repos/{owner}/{repo}/branches/{branch}/protection` and classify it as
      one of: `protected-sufficient` / `protected-insufficient` (no required approving
      reviews) / `unprotected` / `no-admin` (403 without repo-admin).
- [ ] On `protected-insufficient` or `unprotected`, the output explains the gap and
      ties it explicitly to the auto-merge guardrail-#2 hold (empty `reviewDecision`),
      and offers to apply the minimal rule.
- [ ] The offered fix is executable by atelier:
      `gh api -X PUT repos/{owner}/{repo}/branches/{branch}/protection` with
      `required_pull_request_reviews.required_approving_review_count=1`, no required
      status checks that do not exist, and configured so the dual-identity flow still
      works — the `AtelierReviewer` APPROVED review yields `reviewDecision: APPROVED`
      and the `AtelierAuthor` push/merge path is not locked out (`enforce_admins=false`).
      The apply is idempotent.
- [ ] Decision policy is honored: under `auto`, apply autonomously; under `ask`
      (or no TTY / non-interactive without the opt-in flag), present the change and
      wait for the operator.
- [ ] `no-admin` case: setup does **not** fail — it explains the gap and prints the
      exact manual `gh` command / steps for the operator to run with an admin-scoped
      token.
- [ ] Idempotent + safe: never weakens an already-stronger existing rule; never
      enables protection that would block the bot author from merging its own
      approved PRs (`enforce_admins` stays off; `restrictions` stays null).

## Risks / open questions

- **How "decision-broker policy" maps into a bash binary.** The decision-broker
  skill/catalog (`agents/decision-broker/catalog.json`) is consumed by *agents*, not
  by the setup/doctor binaries, and it has **no** `branch-protection` category today.
  Two viable routes: (a) mirror `--grant-reviewer` exactly — a `--apply-branch-protection`
  flag + interactive prompt + non-interactive refusal, reading `.atelier.json`
  `decisionPolicy.byCategory["branch-protection"]` directly via `jq` for the auto/ask
  decision; or (b) add a `branch-protection` category to the catalog so agent-driven
  paths can route it too. Recommend (a) for this task (keeps it self-contained and
  consistent with the existing reviewer-access precedent) and noting (b) as a possible
  follow-up. **Product lead should confirm.**
- **Org-owned member repos:** the `AtelierAuthor` bot identity often lacks repo-admin
  even when the operator's own account has it (see memory note
  `auto-merge-needs-branch-protection`). The classifier must check `viewerPermission`
  under the author identity (as `step_reviewer_access` does) and route to `no-admin`
  rather than erroring when admin is absent.
- **GitHub PUT payload completeness:** the protection PUT requires several keys
  present (`required_status_checks`, `enforce_admins`, `required_pull_request_reviews`,
  `restrictions`) even when null. The implementer must send a complete object, and
  when an existing rule is present, **merge** rather than overwrite so stronger
  settings (status checks, code-owner reviews) are never weakened.
- **Free private repos:** branch protection on the default branch may be unavailable
  on some plans, returning a different error than 403/404. Treat unexpected errors as
  an advisory `↷` skip with the raw message, never a hard failure.
- **Testing:** these are bash binaries with no unit harness in-repo; verification is
  manual against a live repo. The tester step should at minimum cover the
  classifier's 200/403/404 branching with mocked `gh` output if feasible.
