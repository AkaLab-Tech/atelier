# Plan — #22b Mixed-backend workspace status aggregation in `atelier-workspace-status`

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** feat · P1 · ~2h
- **blocked_by:** #22a (the backend-aware `atelier-resolve-dep` verdict this consumes) — merged in PR #233

## Approach

`scripts/atelier-workspace-status` renders one row per workspace member plus a "cross-repo blocked" section. Today every per-member computation assumes the `files` layout and degrades **silently** for a non-`files` member:

- `open_count()` (lines 114–122) greps the member's local `ROADMAP.md`; a `github-project`/`linear` member has no conforming local `ROADMAP.md`, so it reports `0` open tasks — wrong, and indistinguishable from a genuinely empty backlog.
- `roadmap_format()` (lines 128–135) returns `absent` (no local `ROADMAP.md`) — misleading for a backend-tracked member.
- `member_xrepo_blocked()` (lines 195–217) parses unchecked `blocked_by:` lines straight out of the member's local `ROADMAP.md` and calls `atelier-resolve-dep` per blocker (line 212).

The goal is *honest degradation + correct cross-repo resolution*, not a full backend re-render (reading a remote backlog for every member would force OAuth into a read-only dashboard — out of scope here; the runbook in #22c covers the live-backend exercise).

1. **Detect each member's backend once.** In the gather loop (lines 220–232), call `atelier-task-backend "$path"` per member and carry a `BACKEND` value alongside `M_TOKENS`/`M_PATHS` (new parallel array `M_BACKENDS`).
2. **Degrade the file-based columns sanely for non-`files` members.** When a member's backend is not `files`:
   - `open_count` → render `backend:<name>` (or `—`) instead of a misleading `0`, signalling "backlog lives in `<backend>`, not counted offline."
   - `roadmap_format` → render the backend name (e.g. `github-project`) rather than `absent`, so the operator sees *why* there's no local ROADMAP, not a false drift signal.
   - `in_progress_task` → keep reading `IN_PROGRESS.md` if present (the claim registry stays git-anchored per §16.4, and some backends keep an offline mirror), but fall back to `—` cleanly when absent.
   - Surface this in both the JSON shape (add a `backend` field per member at lines 242–245) and the human dashboard row.
3. **Make cross-repo blocked resolution backend-aware via #22a.** `member_xrepo_blocked` already calls `atelier-resolve-dep --from … --token … --id …` per blocker. After #22a, that helper can return `backend-deferred` (exit 6). Teach this function to interpret exit `6`: resolve the blocking sibling task through its backend's `getTask`/`listTasks` (the script can't drive MCP, so it emits the blocker row with verdict `backend-deferred` and the dashboard's command wrapper — `commands/workspace-status.md` — notes that such rows are resolved by the AI layer). Treat `backend-deferred` as "not yet known to be satisfied" for the blocked-count footer (conservative: a deferred blocker is shown, never silently dropped). All other exit codes keep their current meaning (0 satisfied → skip; 3/4/5 → show with verdict).
4. **Document the mixed-backend behaviour.** Add a short comment block near the per-member computations explaining the degraded columns, and update `commands/workspace-status.md` (and/or a `docs/` note) so the operator knows a `backend:<name>` cell means "tracked remotely, not counted offline" and a `backend-deferred` blocker row is resolved through that member's backend.

## Affected areas

- `scripts/atelier-workspace-status` — add per-member backend detection (`M_BACKENDS`); degrade `open_count` / `roadmap_format` / `in_progress_task` rendering for non-`files` members; add a `backend` field to the JSON member objects; teach `member_xrepo_blocked` to handle the `backend-deferred` (exit 6) verdict from #22a; add an explanatory comment block.
- `commands/workspace-status.md` — note that `backend:<name>` cells and `backend-deferred` blocker rows reflect remotely-tracked members (verbatim-passthrough command, so a short clarifying line only).

## Acceptance criteria

- [ ] A workspace whose members **mix backends** (e.g. one `files`, one `github-project`) renders one row per member with no silent `0`/`absent` lies: a non-`files` member's open-count and roadmap-format cells show its backend, not a misleading empty/absent value.
- [ ] The cross-repo blocked section resolves each `blocked_by:<token>#id` through the target member's backend: when `atelier-resolve-dep` returns `backend-deferred` (exit 6, from #22a) the blocker is shown with that verdict and counted as unsatisfied (never silently dropped); `satisfied` (exit 0) blockers are omitted exactly as today.
- [ ] `--json` output includes a `backend` field per member and preserves the existing `crossRepoBlocked` array shape (member/task/title/blocker/verdict), with `backend-deferred` a possible verdict value.
- [ ] The `files`-only path (all members `files`) renders byte-identically to today — no regression to the existing single-backend dashboard.
- [ ] The mixed-backend rendering rules are documented (comment block in the script + a clarifying line in `commands/workspace-status.md`).

## Risks / open questions

- **Read-only scope.** This dashboard stays offline: it does **not** call `listTasks` to fetch a remote member's live backlog count (that needs OAuth and belongs to #22c's live exercise). The plan deliberately shows `backend:<name>` instead of a real remote count. Confirm that honest "tracked remotely, not counted here" degradation is acceptable for the status command, vs. an opt-in `--live` flag (which would be scope creep — recommend deferring).
- **Footer accumulators.** The human dashboard has footer counters (`S_NONSEC5`, `S_DRIFT`, etc., line 275) keyed off the file-based columns. Non-`files` members must not trip the "non-conforming ROADMAP drift" footer (which would be a false alarm) — the implementer must exclude backend-tracked members from that specific drift signal.
- **Depends on #22a's exact verdict word.** This plan assumes `backend-deferred` / exit `6`; #22a shipped exactly that (PR #233), so align to it.
