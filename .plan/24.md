# Plan — #24 auto review-fix loop: re-dispatch implementer/tester on `request-changes`, bounded, then escalate

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** feat · P2 · ~4h (prompt/doc edits across 4–6 files; no source code in this repo)
- **blocked_by:** none (composes with #23 but does **not** depend on unmerged #23 work — see Risks)

## Approach

Today the chain stops the moment `reviewer` returns `request-changes`.
`agents/task-orchestrator.md` step 8 (line 175) reads: *"do **not** invoke
`auto-merge`. Surface the findings, leave the PR open for the implementer to
address in a follow-up."* That "follow-up" is manual — there is no automatic
fix→re-review iteration. This task replaces that terminal branch with a
**bounded, automated review-fix loop** that re-dispatches `implementer` (and
`tester` when coverage is implicated) to address the reviewer's findings on the
**same `task/*` worktree/branch**, re-pushes, and re-runs `reviewer` — until the
PR converges to `approve` (then proceeds to the normal approve path → CI wait
(if #23 has landed) → `auto-merge`) or the loop's bound is exhausted (then it
escalates to the operator with the accumulated findings + attempt logs and
leaves the PR open).

The loop lives in the **orchestrator** — same locus #23 uses for the pre-merge
CI wait — because the orchestrator already owns the chain, the retry budget, and
interruptibility. The `reviewer` stays a pure fresh-context evaluator and
`pr-author` stays the only agent that commits/pushes.

Concretely:

1. **`agents/task-orchestrator.md` — rewrite step 8's `request-changes` branch
   (line 175) into a bounded loop, and add a dedicated "Review-fix loop"
   sub-section** (mirroring the existing "Inner loop — implementer ↔ `/validate`"
   section at lines 140–155). On `reviewer` → `request-changes`:

   a. **Triage the findings first — not every `request-changes` is code-fixable.**
      Parse the reviewer's structured report (`agents/reviewer.md` output, lines
      193–224: severity `critical`/`important`, the per-finding `[severity] file:line`
      blocks, plus the **Auto-merge** blocker line). Partition findings into:
      - **Code-addressable** — `correctness`, `test coverage`, `code quality`,
        `security` findings. These drive the fix loop (implementer ± tester).
      - **Structural / non-code** — `scope alignment` (ships unrelated changes),
        the `size` finding (oversize → already the `[OVERSIZE]` path), missing
        dependency-install justification (PLAN.md §4), or pending human comments.
        These are **not** fixed by re-running `implementer`; route them to the
        existing handlers (oversize → step 8 oversized branch; scope → the
        `scope-creep-detected` decision-broker category) or **escalate directly**
        to the operator without spending a fix cycle. (Open question — see Risks.)

   b. **If there are code-addressable findings and the loop bound is not yet
      exhausted**, re-dispatch `implementer` on the existing worktree with a
      briefing that includes (per the step-6 briefing contract, lines 127): the
      `<worktree-path>`, the approved `.plan/<id>.md`, **the reviewer's verbatim
      findings**, and **the current PR diff** so the implementer fixes against the
      reviewer's concrete concerns — not a re-implementation. Then run the existing
      **inner loop** (`/validate` fast layer, lines 140–155) and, when the findings
      implicate coverage, `tester` — exactly as the first pass does. Each fix attempt
      that fails `/validate` flows through `retry-with-logs` as it does today (no
      change to that mechanism).

   c. **Re-push the fix to the open PR — via `pr-author`, not inline.** This needs
      a new `pr-author` capability (see file #2): the orchestrator must **never**
      commit/push feature code itself (delegation boundary, lines 41–52, 218). On a
      fix cycle the PR already exists, so `pr-author` must push follow-up commit(s)
      to the existing `task/<id>-<slug>` branch and **skip** both the
      `IN_PROGRESS → HISTORY` tracking move (already done on the first pass) and
      `gh pr create` (the PR exists).

   d. **Re-dispatch `reviewer` (fresh context) against the same PR.** `reviewer`
      already supports re-review and re-evaluates from scratch
      (`agents/reviewer.md` lines 24–31, 41–48, 242). **Do not feed it the prior
      findings as anchoring context** — that would violate its fresh-context
      invariant (see Risks; this is a deliberate departure from the ROADMAP
      sketch's "feed it the prior findings"). It naturally sees the fix commits in
      the updated diff.

   e. **Decision on the new verdict:**
      - `approve` → fall through to the **same approve path the orchestrator
        already has** (step 8 merged/held logic, lines 172–174); when #23 has
        landed this is its CI-wait → `auto-merge`, otherwise it is today's direct
        `auto-merge`. #24 must not assume #23's sub-step exists.
      - `request-changes` again → increment the review-fix cycle count and loop
        back to (a), **iff** the bound is not exhausted.

   f. **Bound + exhaustion.** Bound the number of review-fix *cycles* (config:
      `reviewFix.maxCycles`, default 2 — see file #4). The bound is **tied into the
      §8 budget** in that every fix attempt is still logged through
      `retry-with-logs` (so the total per-task work budget of 6 attempts remains a
      hard ceiling that can independently end the loop — whichever cap is hit first
      stops it). On exhaustion: surface the **accumulated review findings across all
      cycles + the `.task-log/*.md` paths**, leave the PR open, and yield to the
      operator — the same terminal shape as today's manual `request-changes`, but
      after N automated attempts. (Whether exhaustion should instead open a
      `blocked` issue via `unblocker` is an open question — see Risks.)

   g. **Decision-rules entries** (lines 209–221): the review-fix loop is bounded
      and budget-tied; it never weakens a guardrail; it never feeds prior findings
      to the `reviewer`; structural findings (scope/size/deps/human-comments) are
      not auto-fixed; the orchestrator still never commits/pushes inline (fix
      re-push goes through `pr-author`).

2. **`agents/pr-author.md` — add a "re-push to an already-open PR" mode.** Today
   `pr-author` assumes it opens a *new* PR: step 3 does the `IN_PROGRESS → HISTORY`
   move, step 5 is the size gate, step 6 is `gh pr create` (lines 57, 74–89, 89–107).
   Add an entry condition: when an open PR already exists for the branch
   (detect via `gh pr view --json number,state`), run the push gate (`safe-commit`)
   + commit + push the follow-up fix commit(s) to the existing branch, **skip** the
   tracking move (already committed on the first pass — re-doing it would be a
   malformed double move) and **skip** `gh pr create`, and return the existing PR
   URL + new commit SHA. The size gate should still run on the cumulative branch
   diff (a fix could push the PR oversize). Keep the change minimal and behind an
   explicit "follow-up" signal in the dispatch briefing so the normal first-pass
   flow is untouched.

3. **`agents/reviewer.md` — light touch.** The re-review path already exists
   (lines 24–31, 242). Add one line under "Fresh context" (lines 41–48) or the
   re-review example noting it is the agent the orchestrator's review-fix loop
   re-invokes, and re-affirming that **the orchestrator must not pass prior
   findings** — fresh context is the property that makes the second-human safety
   hold. No behavioral change.

4. **`templates/atelier.template.json` + this repo's `.atelier.json` — add a
   `reviewFix` config block** (sibling to `prSize` / `taskConcurrency`, and to the
   `ciWait` block #23 proposes), with `_comment` in the same house style:
   - `enabled` (default `true`) — opt-out of the automated loop project-wide;
     when `false`, `request-changes` reverts to today's surface-and-stop terminal.
   - `maxCycles` (default `2`) — number of automated fix→re-review cycles before
     escalation. Absent block → built-in defaults. Keeps the bound out of the
     prompt as a magic number and makes it per-project / version-controlled.

5. **`PLAN.md` §6 and §8 — record the loop in the design source of truth**
   (`CLAUDE.md`: the plan must not drift). §6 (Merge / auto-merge gate) — note that
   a `request-changes` verdict now triggers a bounded automated fix→re-review loop
   before falling back to human, and that convergence to `approve` re-enters the
   normal gate. §8 (Failure recovery, lines 429–443) — note that review-fix attempts
   are logged through the same `.task-log/` budget so the loop cannot spin forever.

## Acceptance criteria

- [ ] When `reviewer` returns `request-changes` with **code-addressable** findings
      (correctness / test coverage / code quality / security) and the loop is
      enabled, the orchestrator automatically re-dispatches `implementer` (and
      `tester` when coverage is implicated) on the **same worktree/branch** with the
      reviewer's verbatim findings + the current diff, re-pushes the fix, and
      re-runs `reviewer` — **without** a manual re-invoke.
- [ ] The fix is committed and pushed by `pr-author` (its new re-push-to-open-PR
      mode), **never** by the orchestrator inline; the re-push skips the
      `IN_PROGRESS → HISTORY` move and `gh pr create` (the PR already exists) and
      returns the existing PR URL + the new commit SHA.
- [ ] Each re-review is a **fresh-context** `reviewer` invocation evaluating the
      updated diff; the orchestrator does **not** pass prior findings to the
      reviewer (its fresh-context invariant is preserved).
- [ ] The loop is **bounded**: it runs at most `reviewFix.maxCycles` (default 2)
      fix→re-review cycles, and every fix attempt is logged through
      `retry-with-logs` so the §8 6-attempt budget remains a hard ceiling —
      whichever cap is reached first ends the loop. It cannot spin forever.
- [ ] On convergence to `approve`, the PR proceeds down the orchestrator's normal
      approve path (CI wait → `auto-merge` when #23 has landed; direct `auto-merge`
      otherwise). #24 works whether or not #23 is merged and adds no dependency on
      #23's CI-wait sub-step.
- [ ] On exhaustion, the orchestrator surfaces the **accumulated findings across
      all cycles + every `.task-log/*.md` path**, leaves the PR open, and yields to
      the operator (terminal — no auto-merge, no silent budget extension).
- [ ] **Structural / non-code findings** (scope-alignment, the `size`/oversize
      finding, missing dependency-install justification, pending human comments) do
      **not** trigger an implementer re-dispatch — they route to their existing
      handlers (oversize path, `scope-creep-detected` broker) or escalate directly
      to the operator without consuming a fix cycle.
- [ ] `reviewFix.enabled: false` (or an absent block defaulting otherwise) reverts
      `request-changes` to today's surface-and-stop terminal behavior; defaults are
      documented in `templates/atelier.template.json` with an `_comment` matching
      the `prSize`/`taskConcurrency` style.
- [ ] `PLAN.md` §6 + §8, `agents/task-orchestrator.md`, `agents/reviewer.md`, and
      `agents/pr-author.md` are mutually consistent — no contradiction between
      "reviewer never anchors on the prior cycle" and the new loop, and no
      contradiction with the orchestrator's "never commit/push inline" boundary.

## Risks / open questions

- **Budget shape: shared §8 counter vs. a separate review-cycle cap (key design
  call).** The ROADMAP says "bound the loop … tied into the PLAN.md §8 retry
  budget." `retry-with-logs` (`skills/retry-with-logs/SKILL.md`) explicitly warns
  against two counters that can drift, yet a review-fix cycle is a *different axis*
  from an implementer-fails-`/validate` attempt (the implementer/tester succeeded;
  the reviewer disagrees). Recommended: keep `retry-with-logs` as the single
  per-attempt logger/ceiling (every fix attempt is logged and the 6-attempt cap
  still bites) **and** add a small dedicated `reviewFix.maxCycles` that bounds the
  number of *review cycles* — whichever is hit first stops the loop. The product
  lead should confirm this dual-bound rather than overloading the 6-attempt counter
  to also mean "review cycles" (which would leave a struggled-to-implement task with
  almost no review budget).
- **Feeding prior findings to the reviewer — contradicts its fresh-context
  invariant.** The ROADMAP sketch says "feed [the reviewer] the prior findings +
  the fixing diff." But `agents/reviewer.md` (lines 41–48, 242: *"Never approve
  based on the previous reviewer cycle"*) makes fresh, un-anchored context the
  whole point of the second-human safety property. Recommended (and assumed in this
  plan): feed prior findings to the **implementer** (which needs them to fix), and
  give the **reviewer** only the updated PR — it sees the fix in the diff on its
  own. Product lead should confirm this departure from the sketch.
- **`pr-author` has no re-push path today.** It is built to open a *new* PR
  (`gh pr create` + tracking move). The fix loop requires pushing follow-up commits
  to an open PR without re-doing either. This plan adds that mode to `pr-author`
  (option chosen so commit/push stays out of the orchestrator). Alternative: a small
  dedicated push skill — heavier, and splits the push responsibility. Recommend
  extending `pr-author`.
- **Which findings are "code-addressable" — the triage boundary.** Re-dispatching
  `implementer` cannot fix an oversize PR, a scope-alignment failure, missing
  dependency justification, or pending human comments. The plan scopes the auto-fix
  loop to correctness/coverage/quality/security and routes the rest to existing
  handlers / direct escalation. The exact partition (especially: does a
  `request-changes` driven *only* by a size finding skip the loop entirely?) needs
  the product lead's sign-off so the loop doesn't burn cycles on un-fixable verdicts.
- **Exhaustion: surface-and-leave-open vs. `blocked` issue via `unblocker`.** The
  PR is the artifact and is already reviewable, so this plan escalates by surfacing
  accumulated findings + logs and leaving the PR open (today's `request-changes`
  shape). `unblocker`/`blocked` issues are designed for hard-stops that produced no
  PR. Open: does the product lead want a `blocked` issue opened on exhaustion for
  queue visibility, or is leaving the open PR + a clear terminal report enough?
- **Composition with #23 (concurrent).** #24 is the `request-changes` branch of
  step 8; #23 is the `approve` branch (pre-merge CI wait). They are disjoint and
  compose cleanly. #24 deliberately routes a converged `approve` back into the
  orchestrator's *existing* approve path rather than referencing #23's not-yet-merged
  sub-step, so the two PRs do not collide and #24 carries no dependency on #23. Both
  touch `agents/task-orchestrator.md` step 8 and `.atelier.json`/template config —
  expect a small merge reconciliation in step 8 and the config blocks whichever lands
  second; flag for the merge gate.
- **Infinite-fix / oscillation guard.** A reviewer that flips approve↔request-changes
  on near-identical diffs (or an implementer that "fixes" by reintroducing a prior
  issue) is contained by the bound, but worth surfacing in the exhaustion report so
  the operator can see the loop oscillated rather than converged.
