# Plan — #23 orchestrator waits for the CI cycle before the auto-merge gate

- **Status:** approved
- **Type / priority / estimate:** feat · P2 · ~3h (doc/prompt edits across 3–5 files; no source code in this repo)
- **blocked_by:** none

## Approach

Today the chain stops short of merge whenever the specialist work finishes before
CI goes green. `skills/auto-merge/SKILL.md` guardrail #3 (lines 60–70) evaluates
`statusCheckRollup` **once** and treats any `IN_PROGRESS`/`QUEUED` check as a
`held` state, and the skill's hard refusal (`SKILL.md:207`,
*"Never retry on a transient guardrail failure (CI still running)"*) forbids the
skill from waiting on its own. So a still-running CI yields `held: CI not green`
and the operator must re-invoke manually.

**Add a bounded, fail-fast CI wait in the orchestrator, immediately before it
invokes the `auto-merge` skill** — keep the skill single-shot (it still evaluates
once and never loops), and locate the *waiting* upstream where the orchestrator
already owns the chain, the budget, and interruptibility. Concretely:

1. **`agents/task-orchestrator.md`** — insert a new "pre-merge CI wait" sub-step
   between `reviewer` returning `approve` and the `auto-merge` skill invocation
   (the chain list at lines 136–137, and the close-the-loop logic at step 8,
   lines 172–177). The sub-step:
   - Queries the PR's check state once. If `statusCheckRollup` is empty after a
     short settle window → no CI; proceed straight to `auto-merge` (the skill
     already treats empty as pass).
   - If any check is `IN_PROGRESS`/`QUEUED` → **wait** using a single blocking
     `gh pr checks <NN> --watch --interval <s> --fail-fast` call wrapped in an
     overall timeout (one blocking call, not a poll loop — respects the
     orchestrator's "never re-invoke an identical Bash command" rule and avoids a
     spinner that breaks `claude -p`). On completion, re-query
     `statusCheckRollup` and reuse the skill's exact pass/fail criteria.
   - If any check resolves to `FAILURE`/`CANCELLED`/`TIMED_OUT`/`STARTUP_FAILURE`
     → **stop immediately** (fail-fast, never wait further), do **not** invoke
     `auto-merge`, surface the red CI as a terminal state, and yield. Do **not**
     auto-redispatch `implementer`/`tester` — that re-dispatch loop is task #24's
     scope.
   - On all checks `SUCCESS` (or skipped/`NEUTRAL`) within budget → invoke
     `auto-merge` as today; the six guardrails run unchanged.
   - On timeout → surface `held: CI still running after <budget>` and yield. The
     wait does **not** consume the 6-attempt `retry-with-logs` budget (it is not a
     specialist attempt) and is interruptible.
   - Add a matching note to **Decision rules** (lines 209–221): the CI wait is
     bounded, fail-fast, budget-exempt, and never bypasses or weakens a guardrail.

2. **`skills/auto-merge/SKILL.md`** — reconcile wording so the skill and the new
   orchestrator behavior do not contradict each other. The skill stays
   evaluate-once. Add a one-line pointer under guardrail #3 (after line 70) noting
   that the bounded *wait* for still-running CI lives upstream in
   `task-orchestrator`, and refine the line-207 refusal to: the skill itself never
   loops/retries on transient CI — it reports the held state and returns; the
   orchestrator is responsible for waiting then re-invoking. Direct operator
   invocation ("merge the PR") therefore still holds-and-returns on pending CI
   (re-invoke or #25 babysit-prs covers that), which is intentional.

3. **`PLAN.md` §6** (Merge subsection, lines 356–362) — design source of truth;
   per `CLAUDE.md` the plan must not drift. Record that the auto-merge cycle
   **waits (bounded) for CI to complete** before evaluating the gate, and that a
   CI failure is terminal (human/▶#24), not a wait condition.

4. **`templates/atelier.template.json`** (+ this repo's own `.atelier.json`) —
   add an optional `ciWait` config block (`maxWaitSeconds`, `pollIntervalSeconds`)
   with sensible defaults and an `_comment` in the `prSize`/`taskConcurrency`
   style, so the wait budget is per-project, version-controlled, and discoverable.
   Absent block → built-in default. This keeps the budget out of the prompt as a
   magic number.

5. **`commands/resume-task.md`** (lines 132–138, optional/light touch) — its
   `pr-open` resume already says *"Pending checks are fine; the auto-merge skill
   re-evaluates them."* Align the wording so resume routes through the same
   bounded wait rather than implying an immediate single evaluation. Keep this
   minimal to avoid bleeding into #25's scope.

## Acceptance criteria

- [ ] When the chain finishes and `reviewer` approves but CI is still
      `IN_PROGRESS`/`QUEUED`, the orchestrator **waits** (bounded) for the checks
      to complete, then evaluates the auto-merge gate — without a manual re-invoke.
- [ ] Pending (`IN_PROGRESS`/`QUEUED`) is distinguished from failed: any
      `FAILURE`/`CANCELLED`/`TIMED_OUT`/`STARTUP_FAILURE` is **never waited on** —
      the orchestrator stops immediately, does not merge, surfaces the red CI as a
      terminal state, and yields (no auto-redispatch of implementer/tester — that
      is #24).
- [ ] When all checks are green within budget, the orchestrator proceeds to the
      `auto-merge` skill, which evaluates the six PLAN.md §6 guardrails unchanged
      and merges iff all pass.
- [ ] Repos with no CI (empty `statusCheckRollup`) are not waited on — the cycle
      proceeds immediately after a short settle window (to avoid concluding "no CI"
      while a workflow is still registering its checks), and the report notes
      "no CI configured."
- [ ] The wait is budget-bounded and interruptible; on timeout the orchestrator
      surfaces `held: CI still running after <budget>` and yields **without**
      looping and **without** consuming the 6-attempt `retry-with-logs` budget.
- [ ] The wait never bypasses or weakens any guardrail. After CI resolves, the
      auto-merge skill still evaluates all six; on unprotected repos the
      guardrail-#2 (`reviewDecision`) hold may still fire even with green CI, and
      the report makes that distinction clear (CI resolved ≠ gate passed).
- [ ] `PLAN.md` §6, `skills/auto-merge/SKILL.md` (guardrail #3 + line-207
      refusal), and the orchestrator prompt are mutually consistent — no remaining
      contradiction between "skill never retries on transient CI" and the new
      bounded wait.
- [ ] (If the config route is approved) the wait budget defaults are documented in
      `templates/atelier.template.json` with an `_comment` matching the
      `prSize`/`taskConcurrency` style; an absent `ciWait` block falls back to the
      built-in default.

## Risks / open questions

- **Where the wait lives — orchestrator vs skill (key design call).** Recommended:
  orchestrator (owns chain/budget/interruptibility; keeps the skill a pure
  single-shot evaluator). Consequence the product lead should accept: a *direct*
  `auto-merge` skill invocation ("merge the PR") still holds-and-returns on
  pending CI rather than waiting; that path is covered by re-invoke or by #25
  babysit-prs. If the lead instead wants the wait to apply everywhere the skill
  runs, it must move into the skill and the line-207 refusal is rewritten more
  substantially.
- **Robust "CI complete" detection.** Recommend a single blocking
  `gh pr checks <NN> --watch --fail-fast` (it blocks until every check reaches a
  terminal state) wrapped in an overall timeout, then a final
  `gh pr view --json statusCheckRollup` re-query so the pass/fail decision reuses
  the skill's exact criteria. Open: confirm `--watch` behaves under `claude -p`
  (no TTY) without a spinner that corrupts piped output — fall back to a bounded
  `sleep`+re-query loop if it does. Either way, avoid emitting *identical*
  successive `gh pr view` calls (orchestrator "never retry on success" rule).
- **No-CI vs not-yet-registered race.** An empty `statusCheckRollup` can mean
  "this repo has no CI" **or** "the PR was just opened and the workflow hasn't
  attached its check runs yet." Concluding "no CI" too early would skip a wait
  that should have happened. Open: settle-window duration / number of confirming
  polls before declaring no-CI. Keep it short (CI usually registers in seconds).
- **Unprotected-repo `reviewDecision`-empty quirk.** On the workspace member repos
  (and atelier-dev's own unprotected repo), GitHub leaves `reviewDecision` empty
  even after a real APPROVED review, so guardrail #2 holds. The CI wait must not
  mask this: waiting for CI then re-invoking auto-merge can still legitimately
  yield `held` on guardrail #2. The report must clearly separate "CI resolved
  green" from "gate passed" so the operator isn't misled into thinking the wait
  failed when it's actually the review guardrail holding.
- **Budget home + default value.** `ciWait` block in `.atelier.json` (recommended,
  version-controlled) vs a constant baked into the orchestrator prompt. Open: the
  default `maxWaitSeconds` — long enough for a typical CI run, short enough not to
  stall the chain (suggest ~15–20 min default, ~15s poll interval).
- **Boundary with #24 and #25.** #23 is the *in-chain, synchronous* wait only. A
  CI **failure** here is surfaced as terminal — it does NOT trigger an
  implementer/tester re-dispatch (that is #24's review/CI-fix loop), and it does
  NOT build the out-of-chain open-PR watcher (#25 babysit-prs). Keep this PR
  scoped to the wait so the three tasks compose cleanly as the ROADMAP block
  intends.
