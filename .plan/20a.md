# Plan ‚Äî #20a setup-project backend selection (delegate to /create-roadmap --backend)

- **Status:** ready (approved ‚Äî product lead)
- **Type / priority / estimate:** feat ¬∑ P1 ¬∑ ~TODO (epic estimate is `~TODO`; product lead should set a real number ‚Äî suggest ~3h: bash helper prompt/flag/marker + command prose + delegation contract)
- **blocked_by:** #18 (M9.1 backend-aware provider keystone ‚Äî must be merged so the resolver + routing this slice writes into exist)

## Approach

Add a backend choice to `/atelier:setup-project` that delegates the actual `.roadmap.json` write and GitHub MCP registration to `claude-roadmap-tools`' `/create-roadmap --backend ‚Ä¶` ‚Äî atelier never re-implements either. Today the flow is entirely files-only: `scripts/atelier-setup-project` `step_roadmap_files()` (line 940) unconditionally writes the ¬ß5 `files` ROADMAP/IN_PROGRESS/HISTORY layout, and `commands/setup-project.md` Phase 1 just relays the helper's stdout. The change threads a backend selection through both, in this order:

1. **Backend resolution in the slash command (`commands/setup-project.md`).** Add a step (after Phase 1's helper run, or as a sub-phase of Phase 1) that determines the backend choice:
   - **Explicit flag** ‚Äî accept a `--backend <files|linear|github-project>` token in `$ARGUMENTS` (mirror the existing `--yes`/`-y`/`--mode=` parsing). When present, use it verbatim; validate against the three-value set and refuse an unknown value with usage text.
   - **Interactive (no flag)** ‚Äî use `AskUserQuestion` to offer the three options (`files` [default] / `linear` / `github-project`) with a one-line description each. `files` is the no-op safe default that preserves today's behaviour exactly.
   - **Headless safe default** ‚Äî when `--yes`/`-y`/`$ATELIER_AUTO` and no explicit `--backend`, **default to `files`** and do not prompt. A remote backend needs OAuth that cannot be auto-resolved (PLAN.md ¬ß16.4), so a headless run never silently selects one. If a headless run *does* pass `--backend github-project|linear`, print a clear note that it still requires interactive OAuth on first `listTasks`/`isAvailable` and proceed to delegate (crt owns the OAuth prompt).

2. **Delegate non-`files` to crt (`commands/setup-project.md`).** When the chosen backend is not `files`, after the bash helper has scaffolded the rest (settings, `.atelier.json`, `.npmrc`, `.gitignore`, root `CLAUDE.md` via Phase 2), invoke crt's `/create-roadmap --backend <choice>` so crt writes `.roadmap.json` and runs the MCP-registration one-liner. This mirrors the existing delegation pattern the command already uses for `/adopt-roadmap` (Phase 3). If `claude-roadmap-tools` is not installed (the command does not resolve), point the operator at `claude plugin install claude-roadmap-tools@akalab-tech` and stop ‚Äî never write `.roadmap.json` inline. For `files`, the helper's existing `step_roadmap_files()` output is the complete answer; no delegation, no `.roadmap.json` (absent ‚áí `files` per `atelier-task-backend`).

3. **`atelier-backend=` marker line (`scripts/atelier-setup-project`).** Add a `BACKEND_STATUS` variable and emit `printf 'atelier-backend=%s\n' "$BACKEND_STATUS"` alongside the existing marker block (lines 1352‚Äì1355). The helper itself stays the *detector/relay* ‚Äî it does not write `.roadmap.json` (that is crt's job via the slash command's delegation). The marker reports what the helper believes the backend to be: it reads an existing `<path>/.roadmap.json` (`jq -r '.backend // "files"'`, the same logic `atelier-task-backend` uses) so a re-run on an already-configured github-project repo reports `github-project`; on a fresh/files repo it reports `files`. The slash command surfaces this marker verbatim and uses `files` vs non-`files` to decide whether step 2's delegation is still needed (idempotent re-run: if `.roadmap.json` already names the chosen backend, skip the `/create-roadmap` delegation).

4. **`allowed-tools` grant (`commands/setup-project.md` frontmatter).** Add `AskUserQuestion` (already present) and a `SlashCommand`/`Bash` grant for invoking `/create-roadmap` per the project's command-invocation convention (match however Phase 3 already invokes `/adopt-roadmap`). No new `Bash` write grants for `.roadmap.json` ‚Äî that stays inside crt.

Keep every existing guard intact: idempotency (re-run preserves files), the `files` path is byte-for-byte unchanged, hard refusals (no inline `.roadmap.json` write, no `git init`, no `Edit`/`mkdir`/`sed`/`jq` direct from the slash command) all still hold.

## Affected areas

- `commands/setup-project.md` ‚Äî new backend-selection phase (flag parse + `AskUserQuestion` + headless default + delegation to `/create-roadmap --backend`); frontmatter `allowed-tools` + `argument-hint` update to advertise `--backend`; surface the `atelier-backend=` marker; a hard-refusal bullet ("never write `.roadmap.json` inline ‚Äî delegate to crt").
- `scripts/atelier-setup-project` ‚Äî `BACKEND_STATUS` resolution (read existing `.roadmap.json` backend, default `files`); new `atelier-backend=` marker emission in the marker block (~line 1355); `--help`/CLI-contract text mentioning the marker. No change to `step_roadmap_files()` ‚Äî the files layout it writes is still correct for the `files` backend.

## Acceptance criteria

- [ ] Running `/atelier:setup-project` interactively offers a backend choice of `files | linear | github-project` (with `files` as the default).
- [ ] Choosing a non-`files` backend delegates to `/create-roadmap --backend <choice>` ‚Äî with **no** inline `.roadmap.json` write and **no** inline MCP registration from atelier (crt owns both).
- [ ] A headless / non-interactive run (`--yes` / `-y` / `$ATELIER_AUTO`) defaults to `files` unless an explicit `--backend` flag is passed; it never silently selects a remote backend.
- [ ] If `claude-roadmap-tools` is not installed, a non-`files` selection points the operator at the install command and stops, rather than writing `.roadmap.json` directly.
- [ ] The bash helper emits an `atelier-backend=<files|linear|github-project>` marker line (read from an existing `.roadmap.json`, default `files`) and the slash command surfaces it.
- [ ] The `files` path is unchanged ‚Äî an existing files project re-running setup sees identical behaviour and no `.roadmap.json` is created.

## Risks / open questions

- **How the slash command invokes `/create-roadmap`.** atelier already delegates to crt's `/adopt-roadmap` (Phase 3) ‚Äî the implementer must reuse that exact invocation mechanism (`SlashCommand` tool vs prose instruction) and `allowed-tools` grant for consistency. Open question for the product lead: confirm `/create-roadmap` is the right crt entry point for a *fresh* setup (vs `/migrate-roadmap`, which #21/M9.4 owns for switching an existing project).
- **Ordering vs Phase 2/3.** The backend choice plausibly belongs *after* the bash helper scaffolds settings/`.atelier.json` but the delegation to `/create-roadmap` may want to run before or after the root-`CLAUDE.md` draft (Phase 2). The plan places delegation after Phase 2; the implementer should confirm `/create-roadmap` does not depend on `CLAUDE.md` and that a delegation failure leaves a recoverable half-setup (files scaffolded, `.roadmap.json` absent ‚áí resolves to `files`, which is a safe fallback).
- **Headless + explicit remote backend.** A `--yes --backend github-project` run will still hit interactive OAuth inside crt on first backend call. The plan surfaces a note and proceeds; the product lead may instead prefer a hard refusal in headless mode for any non-`files` backend. Flagging for decision.
- **`linear` parity.** The prompt offers `linear` too (M9.1 already supports it). The implementer should not GitHub-special-case the delegation ‚Äî `/create-roadmap --backend <choice>` is uniform across `linear` and `github-project`.
