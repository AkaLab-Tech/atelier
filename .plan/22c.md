# Plan — #22c End-to-end validation on a real `github-project`-backed member + mixed-backend workspace

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** feat · P1 · ~3h
- **blocked_by:** #22b (the mixed-backend dashboard #22c exercises end-to-end)

## Approach

This is the **validation/runbook deliverable that closes M9**. The two prior slices (#22a backend-aware resolver, #22b mixed-backend dashboard) made the seams correct *offline*; #22c proves the whole thing works against a **real GitHub Project (OAuth)** and a **real 2–3 repo mixed-backend workspace**. The deliverable is a `docs/` runbook plus a validation log, modelled on the existing `docs/manual-e2e-mature-multirepo.md` (same "what you do / what you say / what you should see / how you know it worked" shape).

Two acceptance threads, each a stage in the runbook:

1. **Full autonomous cycle on a `github-project`-backed member.** Take one member repo configured with `backend: github-project` (via `/setup-project` + `/create-roadmap --backend github-project`, M9.3, already shipped) and drive a complete cycle through atelier:
   - `task` / `/next-task` discovers the backlog via `listTasks("roadmap")` (not local `ROADMAP.md`), honours the planning gate via the Project's **Ready** field + committed `.plan/<id>.md`, and claims an unblocked task.
   - implementer → tester → pr-author run in the worktree; the PR opens on `task/<id>-<slug>`.
   - reviewer + auto-merge gate fire; on merge, `pr-author` calls `appendHistoryEntry(id, prMetadata)` so the Project item moves `roadmap → in_progress → history` **via the backend** (Status field transitions per `githubProject.stateMap`), not via local file edits.
   - The runbook records the expected Project board state at each step and how the operator verifies it (the GitHub Projects UI columns + the `Atelier ID` / `Ready` custom fields).
2. **Mixed-backend workspace aggregation + cross-repo `blocked_by`.** Stand up a 2–3 repo workspace (`/setup-workspace`) whose members mix backends — at least one `files`, at least one `github-project`:
   - `atelier-workspace-status <slug>` (and `--json`) renders one honest row per member: the `files` member's open-count/format from its `ROADMAP.md`, the `github-project` member shown as backend-tracked (#22b), with no false `0`/`absent`/drift signals.
   - A cross-repo `blocked_by:<token>#id` from a `files` member onto a task in the `github-project` member is resolved **through the github-project member's backend**: `atelier-resolve-dep` returns `backend-deferred` (#22a) and the AI layer confirms the blocking task's bucket via `getTask`; the dependent task stays blocked until that task lands in the Project's `history` bucket, then becomes eligible. Also exercise the reverse direction (`github-project` member depending on a `files` member) to prove both resolution routes.
   - The runbook records each `atelier-resolve-dep` verdict (`satisfied` / `open` / `unknown-token` / `unknown-id` / `backend-deferred`) and the dashboard's cross-repo-blocked section against the real workspace.

The deliverable is the runbook (`docs/manual-e2e-workspaces-backends.md` or similarly named) **plus a completed validation findings log** capturing the actual run: what was observed at each step, any divergence from expectation, and a final pass/fail per acceptance thread. The runbook also notes the one-backend-per-repo v1 constraint (§16.7) and that a single Project shared across a workspace is deferred.

Closing M9: the PR that lands #22c also performs the `ROADMAP → IN_PROGRESS → HISTORY` bookkeeping for the epic per `CLAUDE.md` (same-PR tracking move), recording M9 as complete with the cross-repo PR references.

## Acceptance criteria

- [ ] A runbook exists under `docs/` that drives a **full autonomous cycle** (claim → implement → PR → merge → state move) on a real `github-project`-backed member, with each step's expected Project-board state and operator-verification described.
- [ ] The runbook drives a **2–3 repo mixed-backend workspace** (≥1 `files`, ≥1 `github-project`) and shows `atelier-workspace-status` aggregating honest per-member status with no false `0`/`absent`/drift for the backend-tracked member.
- [ ] The runbook exercises **cross-repo `blocked_by` resolved via each member's backend** in both directions (`files → github-project` and `github-project → files`), recording the `atelier-resolve-dep` verdict (incl. `backend-deferred`) and that a dependent task only becomes eligible once the blocking task reaches its backend's `history` bucket.
- [ ] A **completed validation findings log** records the actual run results per stage (observed vs. expected, divergences, final pass/fail), constituting the evidence M9's acceptance is met.
- [ ] The deliverable documents the §16.7 v1 constraint (one backend per repo; shared-Project workspaces deferred) and the closing-of-M9 bookkeeping.

## Risks / open questions

- **Real OAuth + real repos required.** This stage cannot run headless: it needs a GitHub MCP OAuth session and at least two real repos (one with a real GitHub Project). The product lead should confirm the test fixtures — which repos, which GitHub account/Project — before the implementer starts, or accept that the runbook is authored against a named-but-to-be-provisioned fixture and the *findings log* is filled in a subsequent manual pass. Recommend: author the runbook + a clearly-marked "results: pending live run" log in the PR, and treat the live findings as the merge-closing evidence (mirrors how `docs/manual-e2e-mature-multirepo.md` is structured as an executable plan).
- **Size budget.** Pure docs + a Markdown findings log; `docs/**` is large but not code, and the change is exempt-heavy. Still keep the runbook focused on the two acceptance threads — do not re-document the whole §15/§16 design (link to PLAN.md instead).
- **Dependency on a working live backend.** #22c assumes M9.2/M9.3/M9.4 (github-project backend, setup/next-task/planning-gate, migration) are live — they are (HISTORY.md). If the live run surfaces a backend bug, that is a *new* task (the runbook records it as a finding), not a reason to expand #22c.
- **Closing-of-M9 bookkeeping is the orchestrator's at merge**, not the planner's or implementer's mid-work — flagged so the same PR carries the IN_PROGRESS → HISTORY move for the epic.
