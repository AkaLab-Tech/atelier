# Plan — #27 `/plan-task` decomposition lost when unpushed (orchestrator worktree branches from origin/main)

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** bug · P2 · ~3h
- **blocked_by:** none

## Problem restated (one line)

`/atelier:plan-task` commits the epic rewrite + `[ready]` flips + `.plan/<id>.md` to the **local** base branch and never pushes; `/atelier:next-task` cuts the task worktree from **`origin/<base>`** and reads `git show origin/<base>:ROADMAP.md` — so a plan/decomposition commit that never reached origin is invisible and is silently dropped (observed on #22 → PR #233/#234).

## Recommendation — contract (b), with a loud guard as the safety net

Adopt **contract (b): `/plan-task` lands its decomposition + `.plan/` on `origin/<base>` before the task is claimable**, and add a **fail-loud guard** in `/next-task` so a still-unmerged plan is *rejected with a clear message*, never silently dropped. Rationale:

- **(a) is structurally wrong for atelier.** `next-task` step 2 already pins the entire eligibility model to *merged* state: "The local working copy is **not** the source of truth — a task the operator edited locally but has not merged is **not** eligible until it lands on `origin/<base>`" (`commands/next-task.md` lines 51, 181). Basing the worktree on "local main when ahead of origin" would special-case the plan commit against that invariant, reintroduce the operator's local checkout as a source of truth (which step 1/step 8 deliberately keep untouched), and break under `taskConcurrency.max > 1` and any non-`files` backend where there is no local ROADMAP at all. It also can't survive a clean clone / second machine.
- **(b) keeps one source of truth (`origin/<base>`) for backlog, plan, and worktree base** — the plan is just another tracked repo file (`.plan/<id>.md`), exactly as §16.5 already frames it ("`.plan/<id>.md` is always a tracked repo file"). The decomposition rides to origin the same way every other backlog change does: through a small PR.
- The two halves compose: (b) defines *how the plan reaches origin*; the guard defines *what happens if someone claims before it does*. The guard is cheap, hermetic-testable, and is the actual acceptance-bearing change ("never silently dropped").

The product lead's call is whether the plan-landing PR is **manual** (operator pushes/opens it, matching today's "operator decides when to push" prose) or **assisted** (a `/plan-task` follow-up that opens a tracking PR). This plan implements the **manual + loud-guard** variant as the default and notes the assisted variant as an open question (see Risks) — the guard makes both safe; assisted is a convenience layer that can land later without changing the contract.

## Approach

Three coordinated edits, smallest-blast-radius first. No code/agent behavior is invented beyond prose contracts and one new hermetic test — consistent with how this repo is built (model-prose commands + agents, regression-locked by `hooks/tests/*.test.sh`).

1. **`commands/plan-task.md` — make the landing requirement explicit (contract (b)).**
   - Today line 155 says: *"Never push the commit. The plan commit lives on the current branch; the operator decides when to push."* Replace the bare "operator decides when to push" with an explicit **claimability contract**: the plan commit is **not claimable until it is on `origin/<base>`**, because `/next-task` cuts the worktree from `origin/<base>` and reads the merged ROADMAP. Add to the Output's `Next:` line a pre-condition: *"push/merge this commit to `origin/<base>` first (a decomposition only becomes claimable once it is on `origin/<base>`); then run `/atelier:next-task`."*
   - Keep "never push to a protected branch directly" intact — landing happens via a normal PR to `origin/<base>` (squash-merge), not a direct push. For the decomposed case specifically, call out that the **epic rewrite + every `.plan/<sub-id>.md` must land together** in that one commit/PR (they already do — they are one commit), so a partial landing cannot occur.
   - This is prose-only; it changes the documented contract, not the command's tool grants.

2. **`commands/next-task.md` — add the fail-loud "plan-on-base" guard (the acceptance-bearing change).**
   - In **step 3** (`files` backend, both auto-pick and named paths), after the existing planning-gate check (`[ready]` + committed `.plan/<id>.md`), add a guard that verifies the plan/decomposition is actually visible on the worktree base — i.e. read everything from `origin/<base>`, which the command already fetched in step 1:
     - The `[ready]` marker is read from `git show origin/<base>:ROADMAP.md` (already the rule per line 51), so a local-only `[ready]` flip is already invisible to auto-pick — good. The gap is the **named-task** path and the **`.plan/<id>.md` existence check**: make both read `origin/<base>`, not the local working tree. Concretely: validate the plan file exists **on the base** via `git cat-file -e origin/<base>:.plan/<id>.md` (or `git show origin/<base>:.plan/<id>.md >/dev/null`). If the ROADMAP line shows `[ready]` on origin but `.plan/<id>.md` is absent from `origin/<base>`, **stop and refuse** with a precise message:
       ```text
       ✗ /next-task: task #<id> is marked [ready] but its plan/decomposition is not on origin/<base>.
          The plan was committed locally by /atelier:plan-task but never landed.
          A worktree cut from origin/<base> would silently operate on stale ROADMAP state.
          resolve: push/merge the plan commit to origin/<base> (e.g. open a PR for it), then re-run.
       ```
     - For a **decomposed epic**: if `origin/<base>:ROADMAP.md` still shows the *undecomposed* parent (no sub-task lines / no `.plan/<sub-id>.md` on base) while a local-only decomposition exists, the same refusal fires — the orchestrator must never claim a sub-task whose epic rewrite isn't merged.
   - Strengthen the **Hard refusals** list (line 178 area) with one new bullet: *"**Never** claim a `[ready]` task whose `.plan/<id>.md` (or, for a decomposed epic, the sub-task rewrite) is not present on `origin/<base>` — refuse with a pointer to land the plan commit. A worktree cut from `origin/<base>` would otherwise operate on stale ROADMAP state and drop the decomposition."*
   - This reuses the data `/next-task` already reads (`git fetch origin <base>` in step 1; `git show origin/<base>:…` in step 2) — `git cat-file -e` / `git show` against `origin/<base>` are already within the spirit of the existing `Bash(git show:*)` grant; confirm the frontmatter `allowed-tools` covers it (add `Bash(git cat-file:*)` if `git show` is not used for the existence probe).

3. **`agents/task-orchestrator.md` — mirror the guard at step 2/4 (defense in depth).**
   - Step 2 says the worktree is "cut from updated `main`"; step 4 already stops when `[ready]` is set but `.plan/<id>.md` is missing. Tighten step 4's missing-plan check so it is explicit that the `.plan/<id>.md` it reads must be the one carried on the worktree base (the worktree was cut from `origin/<base>`, so the worktree's own `.plan/<id>.md` IS the base copy — reading it inside the worktree already gives the right answer). Add one sentence: if `.plan/<id>.md` is absent **in the worktree** (i.e. it was not on `origin/<base>` at cut time), stop with the same "plan not on base — land the plan commit, then re-run" message rather than proceeding on a marker-without-plan. This makes the orchestrator a second, independent backstop even if `/next-task`'s guard is bypassed (e.g. `/resume-task`, direct dispatch).

4. **`hooks/tests/plan-commit-on-base.test.sh` — lock the contract (hermetic).**
   - Follow the established prose-grep pattern (`chk_prose` / `chk_absent`, `REPO_ROOT` from `BASH_SOURCE`, no network, exit 0/1) used by `plan-task-github-project-ready.test.sh` and `next-task-github-project-claim.test.sh`. Auto-discovered by `.github/workflows/structural.yml` (no workflow edit needed — that's deliberately off the `.github/workflows/**` guardrail).
   - **Prose assertions:**
     - `commands/plan-task.md` states a decomposition is only claimable once on `origin/<base>` (assert the new contract sentence is present; assert the old bare "operator decides when to push" is no longer the *only* statement — assert the new claimability clause coexists).
     - `commands/next-task.md` contains the new refusal message tokens (`is not on origin/<base>` / `never landed` / `push/merge the plan commit`) and the new Hard-refusals bullet.
     - `agents/task-orchestrator.md` step 4 carries the "plan not on base" backstop sentence.
   - **Optional behavioral assertion (recommended, matches `task-backend.test.sh` style):** build a throwaway temp git repo (`git init`, a base branch, a `task/*` worktree cut from it), commit a `.plan/27.md` only to local base and NOT to the simulated `origin`, and assert that the documented probe (`git cat-file -e origin/<base>:.plan/27.md`) exits non-zero — proving the guard's mechanism detects the dropped-plan condition. Keep it hermetic: a local bare repo as the `origin` remote, no network.

## Risks / open questions

- **Manual vs assisted landing (product-lead decision).** This plan implements **manual landing + loud guard**: `/plan-task` still does not push (preserving "operator decides when to push"), and the guard catches an unlanded plan at claim time. The detail file's option (b) mentions *"e.g. via a small tracking PR"* — an **assisted** variant where `/plan-task` opens that PR itself. Assisted is more autonomous (matches the operator's "autonomous atelier" preference) but adds push/PR responsibility to a command that is currently planning-only and explicitly forbids pushing. Recommendation: ship the guard now (it makes manual safe and is the acceptance-bearing change); treat the assisted tracking-PR as a fast-follow once the contract is locked. **Confirm which variant to ship.**
- **`origin/<base>` freshness.** The guard relies on step 1's `git fetch origin <base>` having run; the probe must read `origin/<base>` (post-fetch), not a stale local remote-tracking ref. The plan reuses the existing fetch — no extra fetch — but the test should assert the probe targets `origin/<base>`, not local `<base>`.
- **Non-`files` backends.** For `github-project`/`linear` there is no `ROADMAP.md` on origin, but `.plan/<id>.md` is still a tracked repo file (§16.5). The plan focuses the guard on the `files` backend (where the bug was observed); the `.plan`-on-base existence probe is backend-agnostic and *should* extend to non-`files` too. Open question: include the non-`files` `.plan`-on-base check in this task, or scope strictly to `files` and file a follow-up. Recommendation: include the `.plan`-on-base probe for all backends (it's the same one-liner) but only add the ROADMAP-decomposition refusal text to the `files` path.
- **`git wt` base resolution.** `next-task` step 5 cuts from `origin/<base>`; `task-orchestrator` step 2 prose still says "updated `main`". These should not diverge — the orchestrator backstop assumes the worktree base IS `origin/<base>`. The plan aligns the orchestrator's wording; flag if `git-wt`'s actual `--from` differs from `origin/<base>` in any path.
