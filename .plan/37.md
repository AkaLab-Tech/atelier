# Plan — #37 backend-resident plans: store the plan in the github-project item body, drop the committed `.plan/<id>.md` requirement for non-`files` backends

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** chore · P1 · ~4h (atelier side; the crt capability is a separate cross-repo task)
- **blocked_by:** `claude-roadmap-tools#6` — the crt `RoadmapBackend.setPlan(id, markdown)` / `getPlan(id)` capability (filed + planned, crt PR #30; this task must NOT be claimed until crt#6 is merged). No intra-repo blockers.
- **Ready gate:** the board `Ready` field is intentionally left UNSET until crt#6 merges (the blocker). Set `Ready` then to make #37 claimable.

## Approach

The remaining `files` coupling in the planning flow is the **plan artifact**. Today §16.5 asserts `.plan/<id>.md` "is always a tracked repo file", `/atelier:plan-task` commits it, and `/atelier:next-task`'s plan-on-base guard probes `git cat-file -e origin/<base>:.plan/<id>.md`. The operator's decided design moves the plan into the **github-project item body** (the draft-issue body) inside a delimited section (`<!-- atelier:plan:start -->` … `<!-- atelier:plan:end -->`) for any non-`files` backend. `.roadmap.json` stays (it is the backend pointer/config, not roadmap content). The `files` backend path is untouched.

The change is a coordinated edit across atelier's planning surface, made **backend-aware** at every point that currently assumes a committed `.plan/<id>.md`. The single most important design decision — not spelled out in the task sketch but load-bearing — is **how the implementer still gets the plan into the worktree** when nothing is committed: `/next-task` **materializes** the plan into the worktree (an untracked `<worktree>/.plan/<id>.md` written from `getPlan(id)`) right after the claim, so the orchestrator → implementer contract ("read `.plan/<id>.md` from the worktree", `agents/task-orchestrator.md` step 4 line 121) keeps working unchanged. This keeps the blast radius small: downstream of the worktree-cut, nothing changes.

Order of edits (all prose/markdown — no code in this repo):

1. **`commands/plan-task.md` — Phase 4 `github-project` branch (write path).** Replace "commit `.plan/<id>.md`" with: call the backend's new `setPlan(id, <approved-plan-markdown>)` via the `roadmap-tracking-flow` skill (same way Phase 4 already drives `setReady` at line 102), which writes the delimited plan section into the item body; then `setReady(id, true)` as today. **Do not** `git add .plan` / commit for this backend — there is no committed artifact. Update Phase 5 (discard) to drop the `git clean -f .plan` / commit-revert mechanics for the backend branch (nothing was committed); the rollback for a mid-Phase-4 failure becomes "if `setPlan`/`setReady` ran, the item body/Ready field is the only state — clear `Ready` via `setReady(id,false)` and leave/overwrite the body section on the next attempt". Update the `## Hard refusals` + the Phase-1/Phase-3 `git status`/`git diff` guards that currently expect `.plan` working-tree changes for this backend.

2. **`commands/next-task.md` — step 3 planning gate + plan-on-base guard (read path).** For a non-`files` backend: (a) the planning gate validates the plan **exists in the backend** — the task record from `getTask(id)` carries `Ready`; add a `getPlan(id)` call (or use the body already returned by `getTask`) and treat a missing/empty delimited plan section as the `ready-without-plan` inconsistency, refusing exactly as today. (b) **Skip the `git cat-file -e origin/<base>:.plan/<id>.md` plan-on-base guard** for backend repos — there is no committed plan to be "on base"; the plan lives in the backend, which is already the merged source of truth. The decomposed-epic plan-on-base sub-clause (lines 100) stays `files`-only. Then add a new sub-step after step 6 (claim) / before step 8 (hand-off): **materialize** the plan into the worktree by writing `getPlan(id)` output to `<worktree>/.plan/<id>.md` (untracked) so the orchestrator/implementer read it unchanged.

3. **`agents/planner.md` — return plan content for backend storage.** For a non-`files` backend, the planner still drafts the plan, but instead of (only) writing `.plan/<id>.md` for the command to commit, it **returns the plan markdown body** in its structured output for the command to hand to `setPlan`. Two viable shapes; the plan recommends: the planner keeps writing the draft `.plan/<id>.md` (so the product lead reviews a real file in Phase 3 and the working-tree diff is reviewable), and `/plan-task` reads that file's content to feed `setPlan` on approval, then discards the file (it is never committed for a backend repo). This keeps the planner's authoring/verification logic backend-agnostic and confines the "where does it land" decision to the command. Add a short "Backend-aware storage" note to the planner's boundary/Output sections.

4. **`agents/task-orchestrator.md` — step 4 plan load (defensive).** The orchestrator already `Read`s `<worktree>/.plan/<id>.md` (line 121). Because `/next-task` materializes that file for backend repos, the happy path is unchanged. Update the inconsistency message at line 121 to be backend-aware: for a backend repo the file is materialized at claim time, so its absence means the materialize step failed (not "plan commit not on origin/<base>") — surface that instead, and keep the `files`-backend message as-is.

5. **`skills/task-discovery/SKILL.md` — planning gate wording (step 4 + §"Planning gate for non-`files` backends", lines 81/137).** The committed-`.plan` check that this skill currently calls "identical across all backends … always a tracked repo file (§16.5)" becomes backend-dependent: for a non-`files` backend the gate is "the backend's `Ready` field is set **and** the item body carries a non-empty delimited plan section" (read from the task record / `getPlan`); the `ready-without-plan` inconsistency is unchanged in spirit. The `files` path keeps the committed-file check.

6. **`PLAN.md` §16.5 — the contract change.** Rewrite the "`.plan/<id>.md` stays a tracked repo file" bullet to be backend-dependent: `files` backend → committed `.plan/<id>.md` on `origin/<base>` (+ plan-on-base guard); remote backend → plan in the item body's delimited section, materialized untracked into the worktree at claim time. Keep "the implementer needs the approved plan in the worktree" as the invariant that both backends satisfy by different means. Note the new crt `setPlan`/`getPlan` operations as the backend seam.

7. **`docs/RoadmapBackend.md` contract (in `claude-roadmap-tools`, the cross-repo half).** Add `setPlan(id, markdown)` / `getPlan(id)` to the contract with per-backend semantics: `GitHubProjectBackend` writes/reads the `<!-- atelier:plan:start -->…<!-- atelier:plan:end -->` delimited section in the item body via the item-write / project-detail operations; `LinearBackend` analogue (delimited section in the issue description); `FilesBackend` reads/writes the tracked `.plan/<id>.md` so the contract is uniform. This is the **cross-repo dependency** and is tracked as the `blocked_by` above — it is NOT part of this repo's PR.

## Acceptance criteria

- [ ] On a `github-project` repo, `/atelier:plan-task` stores the approved plan in the Project item body (delimited section) and sets `Ready`; **nothing** is committed to `.plan/`.
- [ ] `/atelier:next-task` claims the task by reading the plan from the backend (`getPlan`/item body); it does **not** require any committed `.plan/<id>.md` in the repo and does **not** refuse on a missing committed plan, and the plan-on-base `git cat-file` guard does not run for backend repos.
- [ ] The implementer still receives the plan: `/atelier:next-task` materializes `getPlan(id)` into `<worktree>/.plan/<id>.md` (untracked) so `task-orchestrator` step 4 and `implementer` read it unchanged.
- [ ] The `files` backend behaviour is unchanged: committed `.plan/<id>.md` + `[ready]` token + plan-on-base guard all still enforced.
- [ ] `PLAN.md` §16.5 no longer asserts ".plan is always a tracked repo file"; it states the backend-dependent contract (files → committed; remote → item body).
- [ ] `skills/task-discovery/SKILL.md` and `agents/task-orchestrator.md` planning-gate / plan-load wording is backend-aware and consistent with §16.5.
- [ ] A cross-repo `blocked_by` on the `claude-roadmap-tools` `setPlan`/`getPlan` capability is declared and resolvable (or the crt task is merged before this one is claimed).
- [ ] Existing already-committed `.plan/*.md` on backend repos: a decision is recorded (migrate vs leave-as-legacy) — out of scope for this PR unless trivial (see Risks).

## Risks / open questions

- **Cross-repo dependency (the hard one).** This task cannot deliver its acceptance against a `github-project` repo until `claude-roadmap-tools` ships `setPlan(id, markdown)` / `getPlan(id)`. crt already has `setReady` (RoadmapBackend.md line 140), so this mirrors an existing pattern, but `setPlan`/`getPlan` do **not** exist yet. **The operator must file that crt task** (crt uses the `files` backend → it will be a `TASK_NNN`) and #37 should carry `blocked_by: claude-roadmap-tools#<that-id>`. Until then, an atelier-only PR can land the prose/contract changes but cannot be end-to-end validated. Decision needed: (a) split into "crt capability" (file in crt now) + "atelier wiring" (#37, blocked on it) — **recommended**; or (b) hold #37 entirely until crt lands.
- **Offline-mirror & reverse-migration interaction.** The item body now contains BOTH the task spec AND the delimited plan section. crt's offline-mirror reconstruction (`docs/RoadmapBackend.md` §"Reverse reconstruction") and `/migrate-roadmap --to files` (#21/M9.4) rebuild `roadmap/TASK_NNN_*.md` from the body — they must **strip or relocate** the `atelier:plan` section so it does not pollute the task-detail file, and ideally materialize it as `.plan/<id>.md` on `github-project → files` migration (and inversely push a committed `.plan` into the body on `files → github-project`). This is crt-side work that belongs with the `setPlan`/`getPlan` task; flag it so the migration matrix stays lossless.
- **Should `.plan/` be gitignored on backend repos?** Since the materialized worktree plan is untracked, an accidental `git add .plan` would re-introduce a committed plan. Adding `.plan/` to `.gitignore` for backend repos prevents that — **but** this repo (atelier-dev, itself a `github-project` backend) already has 24 committed `.plan/*.md`; gitignoring would not untrack them and could confuse the diff. Recommend: leave existing committed plans as **legacy** (untouched), do **not** gitignore in this PR, and let new backend plans live only in the body. Confirm.
- **Delimiter robustness.** If an operator hand-edits the item body, the `<!-- atelier:plan:start/end -->` markers could be removed or duplicated. `getPlan` must define behaviour for missing/duplicate markers (treat missing as "no plan" → `ready-without-plan`; first match wins on duplicates). Specify in the crt contract.
- **`/plan-task` non-interactive path.** Approval stays interactive-only (§16.5, plan-task.md line 28). The draft-write step is unchanged; only the on-approval landing differs. No new headless concern, but the Output/refusal text mentioning "commit SHA" must become backend-aware (no SHA for a backend repo — surface the item id + "plan written to item body" instead).
