> This item had no id in its board title (assigned `#45` while planning). Its scope was **already operator-approved on 2026-07-01** (recorded in the item body) — this plan transcribes that approved scope rather than inventing a new one.

## Approach

M7.1.F31 added branch-protection *detection* (`atelier-doctor`'s `check_branch_protection`) and an *opt-in* apply path (`atelier-setup-project --apply-branch-protection`), but applying it was never made default, and the bot identity can't apply it alone (`admin=false` on AkaLab-Tech repos → the `PUT branches/<b>/protection` 404s). Without protection, GitHub never computes `reviewDecision`, so auto-merge guardrail #2 holds forever even after a genuine approval — every member repo ships unprotected today (see memory `auto-merge-needs-branch-protection`). Fix, per the approved scope:

1. **`setup-project` applies it by default.** Promote `step_branch_protection` from opt-in to default-on. Since the bot lacks admin, the step must invoke the **operator's personal admin identity** (e.g. `Miguelslo27`) for this one call — degrade gracefully to a clear manual instruction when no admin identity is configured/available. Keep a skip/decline flag for the operator who doesn't want it.
2. **`atelier-doctor --fix` applies it.** Today `--fix` only prints a manual instruction when the token lacks admin; make it actually apply the rule via the admin identity when available, same degrade-gracefully fallback otherwise.
3. **`/atelier:align` propagates it.** Add branch-protection apply/verify to align's per-repo convergence pass (bulk, across every registered project).
4. **Twin fix: `classify_branch_protection` in `atelier-setup-project`.** The doctor's 404-ambiguity fix (PR #284: consult the no-admin `.protected` flag rather than assuming 404 == unprotected) must be mirrored here so setup-project's detection agrees with the doctor's — otherwise a protected-but-non-admin branch gets misreported as unprotected by one tool and not the other.

## Affected areas

- `scripts/atelier-setup-project` — `step_branch_protection` promoted to default-on; `classify_branch_protection` gets the same 404-ambiguity fix as the doctor (PR #284).
- `scripts/atelier-doctor` — `--fix` path actually applies protection via the admin identity, not just prints instructions.
- `commands/align.md` — new convergence step: apply/verify branch protection per registered repo.
- `hooks/tests/doctor-branch-protection.test.sh`, `hooks/tests/setup-project-branch-protection.test.sh` — extended for the new apply-by-default / `--fix` / align paths.

## Acceptance criteria

- [ ] A fresh `setup-project` on an admin-owned repo leaves `main` protected (≥1 review) without the operator running raw `gh api`.
- [ ] `atelier-doctor --fix` protects an unprotected repo when an admin identity is available; degrades to a clear manual instruction otherwise.
- [ ] `/atelier:align` reports and applies branch protection per repo during its convergence pass.
- [ ] `setup-project`'s classifier no longer misreports a protected-but-non-admin branch as unprotected (matches the doctor's PR #284 fix).
- [ ] Existing `doctor-branch-protection` / `setup-project-branch-protection` test suites are extended and green.

## Risks / open questions

- **Requires the operator's personal admin identity to be configured and reachable** for the apply step in all three entry points; the degrade-gracefully path (clear manual instruction) is not optional — every entry point must handle "no admin identity available" without crashing or silently skipping.
- **This unblocks other backlog items.** `#37`/`#38` and general auto-merge reliability on member repos are gated on branch protection existing (guardrail #2 needs a real `reviewDecision`) — sequencing this early has outsized leverage on the rest of the auto-merge backlog.
