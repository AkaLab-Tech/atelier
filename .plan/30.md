# Plan — #30 `/atelier:align` Tier 3 ignores `decisionPolicy=auto`

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** bug · P1 · ~3h
- **blocked_by:** none

## Approach

The bug is a **contract/prose** defect in `commands/align.md`, not a bash-helper defect.
`scripts/atelier-align` already surfaces each member's current policy
(`policy_default_of()` at line 90; it emits `decisionPolicyDefault` per project in
`--json` at lines 129–132 and prints `policy <pol>` per project in the human plan at
line 156). So the autonomy signal align needs is **already available from the survey
output** — the command simply never consults it to govern its own interactivity. Today
`align.md` is unconditionally interactive: Step 2 and Step 4 hard-code
`AskUserQuestion` confirmation gates, and the Hard rules say `"the operator merges"` /
`"never push to or merge a base branch"`, which override `decisionPolicy.default=auto`
regardless of the operator's standing full-autonomy decision. The dangling
"operator must merge" end-state is what invites the session to improvise the off-spec
reviewer prompt, the merge-confirmation prompt, and the spurious `gh pr merge`
permission deflection (none of those phrases exist in `align.md` today — they are
improvisation filling the autonomy gap).

Fix it by making `align.md` **policy-aware**, in order:

1. **Read the effective policy.** Add an explicit step (after Step 1 survey) that
   resolves the effective `decisionPolicy.default` per member from the survey output
   (`decisionPolicyDefault` in `--json`, or `policy <pol>` in the plan). The *effective*
   policy is post-Tier-1: if this same run passes `--policy auto`, members are being set
   to `auto`, so Tier 2/3 of the same run are governed by `auto` too. Define `auto` as
   "every member's effective default is `auto`"; if any member is `ask` (or a per-category
   `ask` applies to the relevant decision), that member stays interactive.

2. **Gate Step 2/4 confirmations on policy.** Wrap the `AskUserQuestion` gates so they
   fire only when the effective policy is `ask`. When `auto`: apply Tier 1/2/3 without the
   confirmation gates (the operator has pre-consented via `decisionPolicy.default=auto`,
   exactly as `skills/auto-merge` § "Authorization model — the gate IS the consent"
   establishes for the merge gate).

3. **Route Tier 3 base PRs through `reviewer → auto-merge → base-pull` when `auto`.**
   Replace the `"the operator merges"` dead-end in Step 4 / Hard rules with the normal
   §6 path: after `gh pr create`, drive each base PR through the `reviewer` agent then the
   `auto-merge` skill (which owns the six guardrails + `gh pr merge --squash
   --delete-branch`). After a base PR merges, **fast-forward the operator's local
   checkout** of that base branch (`git -C <repo> pull --ff-only` on the base, since Tier 3
   used a throwaway worktree and left the operator's checkout behind `origin/<base>`)
   rather than emitting a manual "git pull cuando quieras" chore. Keep `ask` behavior
   exactly as today (offer the PR, never merge, print per-repo commands on decline).

4. **Kill the improvisation surface explicitly.** Add Hard rules that (a) forbid the
   spurious *"add a `gh pr merge` permission rule"* suggestion — `Bash(gh:*)` is already in
   align's `allowed-tools` frontmatter, so merge is permitted by design; and (b) forbid
   off-spec reviewer-or-merge questions: under `auto`, reviewing and merging are the
   `reviewer`/`auto-merge` path, not an `AskUserQuestion`. Make the existing
   `"never push to or merge a base branch"` rule conditional: it remains true for `ask`,
   but under `auto` Tier 3 PRs merge through the guardrailed `auto-merge` skill (never a
   direct push to base — that distinction is preserved).

5. **Add a structural regression test** `hooks/tests/align-auto-policy.test.sh` following
   the established prose-assertion pattern (`hooks/tests/plan-task-github-project-ready.test.sh`
   uses `grep -qF` `chk_prose`/`chk_absent` helpers over the committed command file —
   hermetic, no network). Assert: align reads `decisionPolicy.default`; Step 4 has no
   *unconditional* `AskUserQuestion` gate; the `reviewer → auto-merge → base-pull`
   sequence is present for `auto`; the spurious `gh pr merge` permission suggestion is
   absent; and the `ask`-path interactive behavior is still referenced (no-regression).

Open design decision for the product lead (see Risks): how `decisionPolicy.default=auto`
reconciles with the existing **headless** print-and-stop safety (Step "Interaction mode":
`--yes`/`$ATELIER_AUTO` → print plan, apply nothing). The bug report targets the
*interactive* default gating under `auto`; the recommended resolution is that `auto`
removes the in-session confirmation gates and drives Tier 3 autonomously, while the
`--yes`/`$ATELIER_AUTO` flag remains a deliberate **dry-run** (survey-only) path. This
keeps the two axes orthogonal (policy = who decides; headless flag = apply vs preview).

## Risks / open questions

- **Headless flag vs `auto` policy (RESOLVED — product lead, orthogonal).** The two axes
  are kept **orthogonal**: `decisionPolicy.default=auto` governs *who decides* (under `auto`,
  align drops the in-session confirmation gates and drives Tier 3 autonomously), while
  `--yes`/`$ATELIER_AUTO` remains a deliberate **dry-run / preview** path (survey, apply
  nothing). `ATELIER_AUTO + align` therefore still previews and applies nothing — the
  implementer must NOT make the headless flag apply autonomously under `auto`.
- **Unprotected member repos hold on guardrail #2 (known, out of scope).** Per the task's
  "Known interaction" note and the auto-merge skill's guardrail #2: on repos without branch
  protection, GitHub leaves `reviewDecision` empty/`REVIEW_REQUIRED` even after a real
  `reviewer` APPROVE, so auto-merge legitimately **holds**. The workspace members are
  currently unprotected. This fix must route Tier 3 through auto-merge correctly and let it
  hold for the right reason — it must **not** be conflated with (or try to work around) the
  branch-protection gap, which is a separate issue.
- **`auto-merge` post-merge bookkeeping push restriction.** `skills/auto-merge` forbids
  pushing `chore(roadmap): mark X done` directly to `main`. The base-branch fast-forward
  added here is a **local** `git pull --ff-only` on the operator's checkout (no push), so it
  does not collide with that restriction — but the implementer must keep it local-only.
- **Mixed-policy workspaces.** If some members are `auto` and others `ask`, align must
  decide per-member, not globally. The plan resolves this per-member; the test should cover
  that the gate keys off the member's own policy.
- **No bash-behavior test is feasible for the merge path** (it needs live `gh`/PRs), so
  coverage is the structural/prose suite per the task's own scope — acceptable and
  consistent with the existing align/plan-task structural tests.
