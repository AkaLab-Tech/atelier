## Approach

Bring `github-project` to **offline-mirror parity with Linear**, so a `github-project` repo with `offlineMirror: true` keeps local `ROADMAP.md` / `IN_PROGRESS.md` / `HISTORY.md` / `roadmap/` as a one-way read-only mirror refreshed on every skill activation, with task files carrying `backend` / `backendId` frontmatter and the same coherence-by-`backendId` guarantees Linear has.

The Linear mirror machinery already exists and is backend-shaped; this slice extends it rather than rebuilding it:

1. **`## Mirror auto-refresh on activation`** (SKILL.md lines 377–448): this section is written almost entirely against "any remote backend" but its concrete calls name `LinearBackend`. Generalize it so it applies to `github-project` too:
   - The "Applies when" preamble (lines 379–384) already says "or, in the future, any other remote backend" — make `github-project` explicit.
   - The refresh procedure (lines 388–411) calls `LinearBackend.isAvailable()` and `listTasks(...)` — make the backend the one resolved at activation (Linear *or* github-project), so the same three-bucket refresh + safe-failure-mode + coherence rules drive either. The coherence rule (lines 403–406) already matches by `backendId` frontmatter, not title — that carries over unchanged; just note `backendId` = the Projects v2 item node id for `github-project`.
   - **History window** (lines 413–442): `linear.historyWindow` becomes backend-scoped — `githubProject.historyWindow` with the same value grammar (`"90d"`, `"<N>d"`, bare integer, `"all"`). Document it parallel to Linear's.
   - The unreachable-MCP fallback warning (lines 391–396) gains a github-project variant (snapshot age + the GitHub-MCP reconnection command — TBD endpoint).

2. **`backend` / `backendId` frontmatter** — document the `github-project` frontmatter shape, mirroring the Linear block in `docs/RoadmapBackend.md` (lines 28–37):
   ```yaml
   ---
   backend: github-project
   backendId: PVTI_xxx   # the Projects v2 item node id
   ---
   ```
   The `TASK_NNN` stays the human handle; `backendId` is canonical for sync. Add this to the `### GitHubProjectBackend` notes subsection that #19a created (so it lives next to the Identity-table note).

3. **`addTask` / `moveTask` / `appendHistoryEntry` mirror side effects** — #19b *noted* these per operation but deferred the semantics here. This slice fills them in within the `## Operations (GitHubProjectBackend)` section's per-operation "when `offlineMirror: true`" clauses, mirroring the Linear operation blocks (SKILL.md lines 280, 313, 327, 361): `addTask` writes a new `roadmap/TASK_NNN_<slug>.md` with the frontmatter; `moveTask` rewrites the local index link lines (task file never moves); `appendHistoryEntry` removes the `IN_PROGRESS.md` link + appends the standard `HISTORY.md` entry — all bundled with the GraphQL call as one logical transaction.

4. **Command mirror-toggle wiring** (`commands/*`) — confirm `--mirror`/`--no-mirror` from #19c flow into the `github-project` `.roadmap.json` `offlineMirror` field and the indexed-mirror-file creation. If #19c already wired this fully, this is a no-op confirmation; otherwise complete it here.

Because #19d depends only on #19b (not #19c), if #19c and #19d land in parallel they may both touch `commands/*` and SKILL.md → **merge collision risk** at the command-file mirror wiring. Mitigation: sequence #19d after #19c despite the lack of a hard data dependency, OR keep #19d's command edits to a minimum (prefer landing the mirror wiring entirely in #19c so #19d touches only SKILL.md + docs). Flag this to the product lead / orchestrator.
