- **Status:** ready (approved — product lead, batch approval via AskUserQuestion, 2026-07-07)
- **Type / priority / estimate:** docs · P2 · ~2h
- **blocked_by:** none

## Approach

Write `docs/product-owner-guide.md`: a short, example-driven reference for a non-technical product owner writing a PLAN.md §5-shaped `ROADMAP.md` by hand. Cover:

1. **Section structure** — `## 🔥 P0`, `## 🎯 P1`, `## 💭 P2` and what belongs in each (P0 = blocks shipping, kept small).
2. **Item-line grammar** — `- [ ] [ready] \`type\` Title \`#id\` \`~estimate\` \`blocked_by:#NN\``, with a worked example for each `type` tag (`bug`/`feat`/`chore`/`docs`/`refactor`).
3. **Acceptance criteria / sub-bullets** — `Repro:`, `Acceptance:`, free-form notes; what makes a good acceptance bullet (concrete, testable).
4. **`blocked_by`** — intra-repo (`#NN`) vs cross-repo (`token#NN`) with one example of each.
5. **The `[ready]` gate** — why an item needs `/atelier:plan-task` before it's claimable, and that a product owner should NOT hand-add `[ready]` themselves.
6. **Epics** — `Epic:` prefix + indented sub-tasks, when to split a task instead of writing one giant item.
7. **A full worked example** — one complete small `ROADMAP.md` (3-4 items across two priorities) the reader can copy as a starting template.

Cross-link from `docs/operator-guide.md` (if one exists / gets a "roadmap" section) and from `README.md`'s docs index.
