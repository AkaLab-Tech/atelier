# Plan — #44c Rewire /atelier:release + /plan-task tracking PR to delegate coordination; add the structural regression test

- **Status:** ready (approved — product lead, batch approval via AskUserQuestion, 2026-07-07)
- **Type / priority / estimate:** bug · P1 · ~TODO (suggest ~1.5h — release command rewrite + plan-task guidance + one new structural test + one test update)
- **blocked_by:** #44a (orchestrator non-task mode + invariant), #44b (align rewired — the new structural test asserts across all standalone sites and can only pass once align, release, and plan-task are all corrected)

## Approach

Close out the remaining offending sites and add the CI enforcement.

1. **`commands/release.md` — delegate the whole coordination to the orchestrator.** Today the release *command* dispatches `implementer` (bump) → `pr-author` (step 7) → `reviewer` (step 8) → `auto-merge` skill (step 9) **from its own turn**. The command's premise — "two separate `Task` dispatches from the command's own turn is safe because they map to two `gh` identities" — is precisely the disproven authoring-only theory (git-identity axis satisfied, actor/session axis not): the release session coordinates both author and review of the same PR. Rewire so the command hands the whole author→review→merge coordination to `task-orchestrator` (non-task mode):
   - **Steps 5–6 stay in the command:** create the `task/release-<version>` worktree (unchanged — release rides a `task/*`-shaped branch so it is pushable) and dispatch `implementer` to bump `.claude-plugin/plugin.json` `.version`. Preparing the worktree + bump is not PR authoring/coordination, so it stays in the release session.
   - **Replace steps 7–9 with a single orchestrator dispatch** in non-task mode: `{ mode: non-task-pr, repo, worktree, base: main, head: task/release-<version>, title: "chore(release): v<next>", body: <pr_body_lines>, interactive: <mode> }`. The orchestrator dispatches the author → `reviewer` → `auto-merge` as its sub-agents and returns the merge SHA (or `held`/`oversized`). Note: the release branch is `task/release-*`-shaped, so the orchestrator's authoring primitive here is `pr-author` (it is a `task/*` branch, not a `chore/*` non-task branch) — the "non-task" in the mode name refers to the **absence of a roadmap/board item and its bookkeeping**, not the branch prefix. The #44a non-task mode must therefore choose its authoring primitive by branch shape (`task/<id>` → `pr-author`; else → `pr-opener`), which this sub-task depends on and must confirm is stated in #44a's contract. *(If #44a scoped the non-task mode to `pr-opener`-only, surface it — see Risks.)*
   - **Step 10 (push the annotated tag) stays in the release command, after the orchestrator confirms `merged`**, targeting the returned merge SHA — unchanged. The tag push is a release-specific step, not PR coordination.
   - **Frontmatter:** the command already carries no `Edit`/`Write` and no `gh pr merge`/`gh pr review` — keep it that way; it now also no longer needs to dispatch `reviewer` itself (still via `Task`, but as part of the single orchestrator dispatch). Update the step prose + Hard refusals so "authoring and review are two separate Task dispatches from this command's turn" becomes "this command delegates the whole author→review→merge coordination to the orchestrator; it never dispatches `pr-author`/`reviewer`/`auto-merge` from its own turn."

2. **`commands/plan-task.md` — route the plan-landing PR through the orchestrator.** `/plan-task` deliberately does not open the tracking PR itself (it commits the plan and instructs "push/merge this commit to origin/<base> first"; the main session opens the landing PR — memory `plan-landing-pr-cannot-self-review`). The fix here is guidance, not a large code change: add a note (in the `Next:` output block and Hard refusals) that when the plan-landing PR is opened autonomously, it must be coordinated through `task-orchestrator` (non-task mode) rather than authored-and-reviewed by the driving session — the same reason the align base PR is. Do not add PR-authoring tools to this command's frontmatter (planning-only stays planning-only); the note points the operator/driving session at the orchestrator. Confirm with the product lead whether they want a first-class command-owned landing step (see Risks / open questions in `.plan/44.md`).

3. **`hooks/tests/no-self-review-standalone.test.sh` — new structural regression test (the CI enforcement).** A hermetic grep-only test (mirroring `pr-opener-delegation.test.sh`'s idiom) asserting: for each standalone command that prepares a PR (`commands/align.md`, `commands/release.md`, `commands/plan-task.md`), the command does **not** coordinate review of that PR from its own context — i.e. it does not, in its autonomous/coordinating path, invoke `reviewer` (as an agent) followed by `/atelier:auto-merge` for a PR it prepared; instead it names `task-orchestrator` as the coordination target. Concretely assert: (a) the auto/coordinating sections name `task-orchestrator` delegation; (b) they do not pair an inline `reviewer` dispatch with an inline `auto-merge` for their own prepared PR; (c) a positive assertion that the corrected invariant string is referenced. Keep the benign exception legible: a command may still *open* a PR without self-reviewing (the assertion targets the author+review *pairing* in one command context, not authoring alone). Land this test last, when align (#44b), release, and plan-task are all corrected, so it goes green on first run.

4. **`hooks/tests/release-independent-chain.test.sh` — update to the delegated-coordination contract.** The current test asserts the old model: "authoring and review are two separate `Task` dispatches" from the command (Group 2). Retarget those assertions to the corrected contract: the command delegates the whole coordination to `task-orchestrator`; steps 7–9's "the command never authors/reviews/merges from its own turn" becomes "delegates author→review→merge to the orchestrator". Keep the still-true groups (tag pushed only after merge; `task/release-*` branch shape, never `release/*`/`hotfix/*`).

## Affected areas

- `commands/plan-task.md` — `Next:` output + Hard refusals note that the plan-landing PR routes through `task-orchestrator` (non-task mode) when opened autonomously; no PR-authoring tools added (planning-only preserved).

## Risks / open questions

- **Authoring primitive selection by branch shape.** The release branch is `task/release-*`, so the orchestrator's authoring primitive must be `pr-author`, not `pr-opener`. This means #44a's non-task mode must pick the author by branch shape (`task/<id>` → `pr-author`; else → `pr-opener`), not hard-wire `pr-opener`. If #44a scoped the mode to `pr-opener` only, either #44a must be widened or the release dispatch must pass an explicit `author_agent: pr-author` hint. Flag as the key coordination point with #44a; the implementer should verify against merged #44a before writing the release dispatch.
- **`/plan-task` landing PR ownership (unresolved design choice).** Whether the plan-landing PR should become a first-class `/plan-task`-owned step routed through the orchestrator, or stay operator-driven with corrected guidance, is a product-lead decision (also raised in `.plan/44.md`). This sub-task takes the lighter "corrected guidance" path by default; escalate if the product lead wants the heavier first-class step (which would grow this sub-task's scope and might warrant its own unit).
- **Structural test false-positives on prose.** A grep-based assertion that a command "does not pair reviewer + auto-merge in its own context" must be written carefully so it does not trip on the *corrected* prose that *mentions* reviewer/auto-merge while describing what the orchestrator does. Anchor the negative assertions on the command's own coordinating verbs (e.g. an inline `SlashCommand`/`Task` dispatch of `reviewer`), not on any mention of the words — mirror how `pr-opener-delegation.test.sh` scopes its `tier3_auto_section` extraction.
