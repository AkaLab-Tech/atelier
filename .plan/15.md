## Approach

Add baseline-diff visual regression on top of the existing raw-screenshot capture in `e2e-runner` / `visual-validation` (today: screenshots are captured and uploaded as secret gists, with no comparison):

1. **Baseline storage.** Per-project baselines, versioned — reuse the existing "upload as secret gist" mechanism `visual-validation` already has for PR screenshots (consistency over inventing a second storage path), keyed per test name.
2. **Diff on each e2e run.** After capturing a screenshot, diff it against its baseline with a configurable pixel/perceptual threshold (a lightweight diff lib — evaluate via the `/safe-install` §4 process: self-question, ≥2 alternatives compared, no <7-day-old packages, no moderate+ vulns).
3. **Gate behavior.** A diff above threshold flags the PR (attach before/after/diff images to the PR description, mirroring how screenshots are already attached) — flags, does not hard-fail the PR gate outright (false-positive risk on a new feature's first real screenshot), so the reviewer sees it and judges.
4. **First run / missing baseline.** Capture-and-store as the new baseline; never fail on a missing baseline.
5. **Baseline updates.** An explicit operator/reviewer action (not automatic) updates a baseline once a flagged diff is judged intentional — avoids silently absorbing real regressions as new baselines.

- [ ] Missing baseline → capture-and-store, never a failure.
