# Plan — #28 `atelier-pr-size-check` aborts under `set -u` (`repo_flag[@]` unbound) → counts 0/0 in PR mode

- **Status:** ready (approved — product lead)
- **Type / priority / estimate:** bug · P2 · ~2h
- **blocked_by:** none

## Root cause (verified in-repo)

`scripts/atelier-pr-size-check` runs under `set -euo pipefail` (line 34) with `#!/usr/bin/env bash` (line 1) — no bash-version guard. On macOS the default `/bin/bash` is **3.2.57**, where expanding an **empty** array via `"${arr[@]}"` while `set -u` is active raises `arr[@]: unbound variable`. (Bash only stopped treating an empty-array `[@]`/`[*]` expansion as "unset" in 4.4.)

The PR-mode codepath in `emit_diff_jsonl()` is:

```sh
224    local repo_flag=()
225    [ -n "$REPO" ] && repo_flag=(--repo "$REPO")
226    gh pr view "$PR" "${repo_flag[@]}" --json files --jq '.files[]' \
```

When `--repo` is **not** passed (the common case — `gh` auto-detects the repo from the cwd, exactly how `reviewer` / `auto-merge` call it), `repo_flag` stays empty, so `"${repo_flag[@]}"` on line 226 aborts the function under bash 3.2 + `set -u`. The abort happens inside the process-substitution producer `< <(emit_diff_jsonl)` (line 255), so the `while` loop reads zero lines, `total_lines`/`total_files` stay `0`, and the script prints `0 line(s) across 0 file(s)` plus the stray `repo_flag[@]: unbound variable` on stderr. This is the false `0/0` that lets the §6 size guardrail pass an oversize PR in PR mode. Confirmed as the reported symptom on PR #236.

### Audit of every `"${arr[@]}"` site (whether each can be empty)

`grep -n '\[@\]' scripts/atelier-pr-size-check`:

- **L154 `EXEMPT=("${DEFAULT_EXEMPT[@]}")`** — `DEFAULT_EXEMPT` is a literal non-empty list (L52–71). Cannot be empty → **safe today**, but see hardening note below.
- **L188 `for pat in "${EXEMPT[@]}"`** — `EXEMPT` is seeded from the non-empty `DEFAULT_EXEMPT` (L154) and is only *reassigned* when `exempt_len > 0` (L169–172), so it is never empty in practice → **safe today**, but defensively fragile (a future edit that allows an empty `EXEMPT` would reintroduce the same class of bug).
- **L226 `"${repo_flag[@]}"`** — **THE BUG.** Routinely empty (no `--repo`).
- **L274 `[ "${#counted_paths[@]}" -gt 0 ]`** — the *count* form `${#arr[@]}` does **not** trip `set -u` on bash 3.2 even when empty → **safe**, and it guards…
- **L276 `printf '%s\n' "${counted_paths[@]}"`** — only reached after the L274 `… || return` guard proves the array is non-empty → **safe**.

Conclusion: the only expansion that is empty on a normal run is **L226**. To prevent the next empty-array path from reproducing the bug, the fix applies the empty-array-safe idiom to **L226** (required) and to **L154 + L188** (defensive hardening), and leaves the already-guarded L274/L276 as-is.

## Approach

1. **Fix `scripts/atelier-pr-size-check`** using the bash-3.2-portable empty-array-safe expansion idiom `"${arr[@]+"${arr[@]}"}"` (expands to nothing when the array is unset/empty, to the quoted elements otherwise — works on bash 3.2 *and* 4.x):
   - **L226 (required):** `gh pr view "$PR" "${repo_flag[@]+"${repo_flag[@]}"}" --json files --jq '.files[]'`
   - **L154 (defensive):** `EXEMPT=("${DEFAULT_EXEMPT[@]+"${DEFAULT_EXEMPT[@]}"}")`
   - **L188 (defensive):** `for pat in "${EXEMPT[@]+"${EXEMPT[@]}"}"; do`
   - Add a one-line comment at L223–226 noting *why* (bash 3.2 + `set -u` empty-array unbound; macOS system bash), so a future cleanup doesn't "simplify" it back.
   - **Why not the alternative** (`require bash 4.4+` via a shebang/version guard, or `set +u` around the call): the script's portability-to-bash-3.2 contract is already an explicit, load-bearing design choice elsewhere in the same file (the `case`-glob comment at L175–185 and the `sort | uniq` comment at L272 both call out "portable to bash 3.2 / macOS system bash"). Requiring 4.4+ would break that contract on every stock macOS host and force operators to install a newer bash just to gate PR size; `set +u` would locally defeat the safety net the rest of the script relies on. The `${arr[@]+…}` idiom is the minimal, in-keeping fix.

2. **Add a hermetic regression test** `hooks/tests/pr-size-check-empty-array.test.sh` following the established `chk_*` / `pass`/`fail` pattern:
   - `set -uo pipefail`; `REPO_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/../.." && pwd)"`; `SCRIPT="$REPO_ROOT/scripts/atelier-pr-size-check"`.
   - `command -v jq >/dev/null 2>&1 || { echo "  SKIP: jq not on PATH"; exit 0; }` (mirrors `atelier-align-survey.test.sh`; jq is the script's hard dep).
   - **Stub `gh` on PATH** using the `SHIM_BIN` pattern from `safe-commit-worktree.test.sh` (L33–47): `mkdir -p "$TMP/bin"`, write a `gh` shim that, on `gh pr view … --json files …`, emits a **fixture** JSON files array (no network), and `export PATH="$TMP/bin:$PATH"`. The shim asserts hermeticity — a real `gh`/network call never happens.
   - **Force the empty-`repo_flag` codepath**: invoke `bash "$SCRIPT" --pr 1 --project "$TMP"` **without** `--repo`, so `repo_flag` is empty at L226 (the exact bug trigger).
   - **Run under the script's own `set -u`** (the script sets it at L34; the test just needs to *not* pass `--repo`). Capture stdout+stderr separately.
   - **Assertions:**
     - (a) **no `unbound variable` on stderr** — `grep -q 'unbound variable'` ⇒ fail. (Reproduces red on the unfixed script.)
     - (b) **non-zero count for a non-empty fixture** — fixture has a non-exempt file with `additions+deletions > 0`; assert the `Counted:` line reports a non-zero line count and ≥1 file (not the false `0/0`).
     - (c) **regression guard on the `--repo` path** — second invocation *with* `--repo owner/name` still counts correctly (shim ignores/echoes the flag), proving the fix didn't break the populated-array branch.
     - (Optional) a fixture that exercises the exempt globs (e.g. a `**/*.test.*` path) to confirm exemption still subtracts — keeps the test honest about the counting path it drives through.
   - Name ends in `.test.sh` so it is **auto-discovered** by `.github/workflows/structural.yml` (the "hooks/tests regression suite (auto-discovered)" step, L41–54) — no workflow edit needed (which also keeps the change off the `.github/workflows/**` auto-merge guardrail). `bash -n` on the new file is also exercised by the syntax-check step (L29).
   - `chmod +x` the new test file (existing runnable tests carry the bit; the runner uses `bash "$t"` so it's not strictly required, but match convention).

## Risks / open questions

- **Reproduction depends on the bash version.** On a CI runner with bash 4.4+ (`ubuntu-latest` ships ≥5.x), the *unfixed* L226 expansion does **not** abort — so a test that merely runs the script under the CI bash would pass even before the fix and would not guard the regression. The fixture/stub test guards the *observable contract* (no `unbound variable` on stderr; non-zero count), which holds on both bash versions; but to truly exercise the 3.2 failure mode the test cannot rely on the CI bash. **Open question / decision for the product lead:** is the stderr+count contract assertion sufficient (recommended — it is hermetic and version-independent, and directly encodes the acceptance), or should the test additionally force the failure mode (e.g. emulate the empty-`[@]`-unbound behavior, or skip-with-note when `BASH_VERSINFO >= 4.4`)? The implementer should confirm the test is red on the unfixed script *on a bash-3.2 host* (macOS) and document that in the test header.
- **gh stub fidelity.** The shim must emit the same `--json files --jq '.files[]'` shape the script consumes (`{path, additions, deletions}` per file). If the script's `gh` invocation shape changes later, the stub must track it. Mitigate by keeping the fixture minimal and asserting on the `Counted:` output line, not internal plumbing.
- **Defensive L154/L188 changes are behavior-neutral today.** Including them slightly widens the diff beyond the strict bug. Justified by the task's explicit instruction to cover *every* same-class expansion so the next empty-array path can't reproduce the bug; the size is still well within the per-PR budget.
