# Plan — #9 `Stop`-hook auto-reprompt on validation failure (exceptional path)

- **Status:** ready (approved — product lead, batch approval via AskUserQuestion, 2026-07-07)
- **Type / priority / estimate:** feat · P2 · ~3h
- **blocked_by:** none (M4.14, the orchestrator-driven implement↔validate loop, shipped in PR #65 — the intra-repo blocker is resolved)

## Approach

M4.14 already puts the implement↔validate loop inside `task-orchestrator` (it reads `/validate` output and re-dispatches `implementer`). This task adds a **harness-level** complement: a `Stop` hook that fires when an assistant turn ends with a failed `/validate` (or `/validate --full`), so the retry loop can survive session restarts / high per-turn dispatch latency without going through the orchestrator each time.

1. New hook script `hooks/stop-validation-reprompt.sh`, registered in `hooks.json` under `Stop`, **off by default** (opt-in via an `.atelier.json` flag or env var — the task's own acceptance requires this).
2. Detection: the hook inspects the last turn's transcript for a `/validate` invocation and a failure exit. If not found (or no failure), the hook is a no-op (exit 0, no reprompt).
3. Budget: reads/increments `<worktree>/.task-log/attempt-count` — the **same** counter M4.14 writes/reads (never a second counter). If the count exceeds the 3+3 (6-attempt) budget from PLAN.md §8, the hook does nothing and lets the orchestrator's `blocked`-issue path take over.
4. On a countable failure, emit a structured reprompt: `RETRY-attempt-N/6` header + the failing command's full stdout/stderr verbatim + the directive "correct the issues below without restarting the task; do not reset the worktree."
5. Composition with M4.14: since both mechanisms share the same counter file, the hook must not double-increment when the orchestrator is *also* going to re-dispatch on its own turn — the simplest safe rule (and the one this plan recommends) is that the hook only fires when the **session is about to stop** (i.e., genuinely no further orchestrator turn is queued), which is exactly what the `Stop` hook contract already guarantees; no extra coordination needed beyond sharing the counter file.

## Affected areas

- `hooks/stop-validation-reprompt.sh` — new hook script.
- `hooks/hooks.json` — register under `Stop`, gated by the opt-in flag.
- `templates/settings.template.json` / `.atelier.json` — new opt-in key (e.g. `stopHookReprompt.enabled`, default `false`).
- `hooks/tests/stop-validation-reprompt.test.sh` — new hermetic test (transcript-failure detection, budget respect, no-op when disabled, no-op when no failure).
- `operator-rules.md` / `PLAN.md` — document the opt-in mechanism briefly.

## Acceptance criteria

- [ ] Hook detects a failed `/validate`/`/validate --full` turn and emits a `RETRY-attempt-N/6` reprompt with the verbatim failure output.
- [ ] Hook respects the shared `<worktree>/.task-log/attempt-count` budget — never exceeds 6, never bypasses the `blocked` issue path.
- [ ] Hook is off by default; enabling it is a one-line per-project or env-var opt-in.
- [ ] New hermetic test passes and is auto-discovered by `hooks/tests/*.test.sh` (no CI workflow edit needed).
- [ ] No change to M4.14's orchestrator-driven loop when this hook is disabled (the default).

## Risks / open questions

- **Double-reprompt risk if the opt-in coordination assumption above is wrong.** If in practice the orchestrator's own turn *and* the Stop hook can both fire for the same failure, the shared counter prevents exceeding budget but could still produce a confusing duplicate reprompt in one edge case (orchestrator re-dispatches mid-stream while the hook also fires on the same Stop event). Flag for the implementer to verify empirically once both paths are live simultaneously; if it happens, disabling M4.14's own re-dispatch when this hook is enabled (mutually exclusive, not additive) is the fallback design.
- Genuinely "for later" per the task's own trigger note (captured 2026-05-21) — planning it now (per operator's no-exceptions instruction) does not obligate immediate execution priority; P2 reflects that.
