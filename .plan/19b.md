## Approach

Add a new `## Operations (GitHubProjectBackend)` section to `skills/roadmap-tracking-flow/SKILL.md`, structurally mirroring the existing `## Operations (LinearBackend)` section (SKILL.md lines 249–375). Place it as a sibling after the LinearBackend section (after line 375, before `## Mirror auto-refresh on activation` at line 377). The section documents how each of the six contract operations is realized against GitHub Projects v2 GraphQL — same Inputs / Returns / Side effects / Errors / per-backend-notes shape the LinearBackend section uses.

Per-operation work, mirroring the Linear blocks:

1. **Activation prerequisites preamble** (mirrors lines 253–261): `.roadmap.json` declares `backend: "github-project"`, a valid project/owner selector (`githubProject.owner` + `githubProject.projectNumber` or node id — the fields #19a named), and a `githubProject.stateMap`. A GitHub MCP is registered (OAuth authorized once). State the role-not-tool-name convention (GraphQL operation roles: project-item-list, item-fetch, item-create/draft-issue-create, item-field-update, comment/note-create) and re-state the snake_case↔camelCase `stateMap` translation warning (mirror lines 261).

2. **`listTasks(bucket)`** (mirror lines 263–282): resolve Status values via `githubProject.stateMap[bucket]`; query Project items filtered to those Status values; build task elements (`id` = item node id, `title`, `current_bucket` reverse-mapped from Status). Note mirror-refresh side effect when `offlineMirror: true` (full semantics deferred to #19d). Errors: empty → empty list; MCP unavailable → `backend-unavailable`.

3. **`getTask(id)`** (mirror lines 284–298): fetch the Project item by node id; return `id`, `title`, `body`, `current_bucket`, plus the custom-field values (`type`, `estimate`, `Ready`, `blocked_by`-text). Progress-notes equivalent of `## Status` / Linear comment thread = the Project item's **notes/comments** (the draft-issue body or linked-issue comments) — document where progress lives. Errors: `task-not-found`, `backend-unavailable`.

4. **`addTask(task)`** (mirror lines 300–317): create a Project item (draft issue) with title + body; set Status to the first element of `stateMap.roadmap`; set custom fields (`type`, `estimate`, `Ready` default unset, `blocked_by` text). Capture the assigned item node id. Note the `offlineMirror: true` side effect (write `roadmap/TASK_NNN_<slug>.md` with `backend: github-project` + `backendId: <item-id>` frontmatter; local `TASK_NNN` allocated independently). Errors: missing title → throw; permission denied → throw verbatim.

5. **`moveTask(id, fromBucket, toBucket)`** (mirror lines 319–329): pre-check current Status is in `stateMap[fromBucket]` (else `task-not-in-from-bucket`); resolve target = `stateMap[toBucket][0]`; call item-field-update to set Status. Atomicity: a single `updateProjectV2ItemFieldValue` GraphQL mutation is atomic, satisfying the contract. Forbid `toBucket == "history"` → `move-into-history-forbidden`. Mirror side effect when offline.

6. **`appendHistoryEntry(id, prMetadata)`** (mirror lines 331–363): pre-check Status in `stateMap.in_progress` (else `task-not-in-progress`); update Status to `stateMap.history[0]`; append PR metadata as a Project-item **comment/note** (the GitHub equivalent of the Linear comment). Carry the **same atomicity caveat** as Linear (lines 354–359): the Status change is atomic per mutation, but the comment append is a separate call — do not revert the Status on comment failure; retry once, then surface the comment text to the user. Mirror side effect when offline (remove from local `IN_PROGRESS.md`, append to `HISTORY.md` in the standard entry shape). Errors: `task-not-in-progress`, missing `prMetadata` fields, `backend-unavailable`.

7. **`isAvailable()`** (mirror lines 365–375): return `true` iff a GitHub MCP server is registered (host/name match); **issue no GraphQL call** (would trigger OAuth). Cross-reference the `### GitHubProjectBackend` notes in `docs/RoadmapBackend.md` (added in #19a). This duplicates the pattern documented in #19a — keep them consistent.

Keep all error names identical to the contract's stable strings (`task-not-found`, `task-not-in-from-bucket`, `move-into-history-forbidden`, `task-not-in-progress`, `backend-unavailable`).

## Risks / open questions

- **Size budget.** The Linear analogue is ~128 lines (~64% of the 200-line budget). The GitHub section will be comparable. Bundling anything else (routing, mirror) would overflow — which is exactly why the decomposer isolated it. Keep the prose tight; if it threatens the budget, the activation cross-references (not the operation bodies) are the first thing to trim, since routing is #19c's job anyway.
- **Projects v2 field-update ergonomics.** Setting a single-select Status (and custom fields) via GraphQL requires the field id + option id, not the human label. The operation prose should describe resolving label → id at call time (the skill inspects the Project's field metadata), mirroring how Linear resolves state names. Worth being explicit so the implementer doesn't assume label-based updates.
- **Depends on #19a's named fields.** If #19a names the config selector differently (`githubProject.projectId` vs `owner`+`number`), this section must match. The dependency edge enforces ordering, but the implementer should re-read #19a's merged doc before writing.
