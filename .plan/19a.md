## Approach

This slice pins down the *design contract* for the new backend before any operations or commands are written. It extends one file — `docs/RoadmapBackend.md` — mirroring the existing `LinearBackend` documentation seams. Three concrete edits:

1. **Identity table row** (the table at `docs/RoadmapBackend.md` lines 22–26): add a `GitHubProjectBackend` row. Canonical `id` = the GitHub Projects v2 **item id** (`PVTI_...` node id) for sync; the human handle stays `TASK_NNN` when the offline mirror is on (parity with the Linear `ENG-123` ↔ `TASK_NNN` rule at lines 28–37).

2. **Buckets table column** (the table at lines 43–47): add a `GitHubProjectBackend` column mapping each of the three abstract buckets (`roadmap` / `in_progress` / `history`) to a set of values of the Project's custom single-select **Status** field, resolved via a `stateMap` exactly like Linear's `linear.stateMap` (defaults to propose: `roadmap: ["Backlog", "Todo"]`, `inProgress: ["In Progress"]`, `history: ["Done", "Cancelled"]`). Reuse the *fixed* snake_case-bucket ↔ camelCase-config-key convention already spelled out in the warning block at lines 49 — the GitHub backend must translate identically.

3. **`### GitHubProjectBackend` per-backend-notes subsection** under "Per-backend notes" (after the `### LinearBackend` block at lines 155–163, before "Future backends" at line 165), mirroring the LinearBackend notes shape. It documents:
   - **GraphQL surface** — Projects v2 is GraphQL-only; name the operations the backend depends on by *role* (project-items query, item field-value read, `addProjectV2ItemById` / draft-issue create, `updateProjectV2ItemFieldValue` for Status changes), not exact tool names (same role-not-name convention SKILL.md uses for Linear MCP tools at line 259).
   - **Status-field `stateMap`** — Status ↔ buckets, with the camelCase/snake_case translation.
   - **Custom fields** — `#id`, `type`, `estimate` as Project custom fields; `[ready]` as a dedicated single-select/boolean **`Ready`** field; `blocked_by` as a **text** field (no native dependency field in Projects v2 — store the `#id` list as text, per §16.3 referenced in the task).
   - **Auth** — a **GitHub MCP** over OAuth, analogous to Linear's `mcp.linear.app` registration. **Never `gh` CLI.** Record the open question: the exact hosted GitHub-MCP endpoint URL is unconfirmed (Linear analogue is `https://mcp.linear.app/mcp`).
   - **`isAvailable()` pattern** — mirror LinearBackend's check (lines 128–129, 365–375 in SKILL.md): inspect the registered MCP server list for a host/name match, return boolean, **issue no API call** (would trigger OAuth on first use). Returns `false` when the GitHub MCP is not registered.
   - **Project/owner selection** — the GitHub equivalent of `linear.teamId`: a `githubProject.owner` + `githubProject.projectNumber` (or the Project node id) recorded in `.roadmap.json`. Name the field now so #19c's command template has a target.

Also update the **Future backends** prose (lines 165–175) and the intro (lines 5–10) so `GitHubProjectBackend` is no longer listed as merely "future" — it is now a shipped third backend.

## Risks / open questions

- **GitHub-MCP endpoint is unconfirmed.** The whole auth story hinges on a hosted GitHub MCP that exposes Projects v2 GraphQL over OAuth. If no such hosted MCP exists, the design (and #19c's registration step) needs the product lead's decision — but the constraint is firm: **no `gh` CLI fallback**. This is the single biggest open question and it gates #19a's auth prose and #19c's registration.
- **Projects v2 has no native `blocked_by`.** Storing dependencies as a text field is lossy (no graph queries). Acceptable per the task's §16.3 reference, but worth flagging to the product lead.
- **Status field is per-Project, user-defined.** The default `stateMap` values may not match a user's Project; like Linear, users edit `stateMap` later. The doc should say so.
