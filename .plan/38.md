- **Type / priority / estimate:** bug · P1 · ~2h (Option b, the recommended default below; a/c are larger and not scoped here)
- **blocked_by:** none

> ⚠️ **Overlap with `#44` needs product-lead resolution before this is claimed.** `#44`'s family (44/44a/44b/44c) fixes self-approval for **standalone, non-task** PR flows (`/align`, `/release`, `/plan-task`) by generalizing `task-orchestrator` so a driving session never coordinates both authoring and review itself. `#38`, as written, describes the classifier blocking the **ordinary `/next-task` task chain** (`task-orchestrator` → `pr-author` → `reviewer`, all dispatched by the same orchestrator) — a narrower and more central case, since if *that* self-blocks, no task PR would ever auto-merge. This appears to be in tension with the operator's own observed experience (recorded auto-merges completing via the normal task chain — see memories `merge-autonomy-full`, `release-via-chain-not-main-session`). Two readings: (1) `#38` describes a real, possibly intermittent classifier edge case in the *core* chain that is separate from what `#44` fixes, or (2) `#38` was superseded/only reproduced under conditions `#44`'s corrected invariant already addresses. **Recommend confirming which reading is correct (e.g. by checking whether `#38`'s repro PR — around PR #20 per the task body — predates or postdates `#44`'s underlying design decision) before this is scheduled**, since building a fix for a problem `#44` already solves would be wasted work.

## Approach

Assuming reading (1) holds (a genuine, still-live gap in the core chain) — of the task's own three fix directions, this plan recommends **(b): make the reviewer advisory-only under auto-mode**, as the lowest-effort option that matches current de-facto behavior and doesn't require harness-level changes:

1. `agents/reviewer.md`: when running under auto-mode (detect via the same signal `task-orchestrator` already uses to know it's autonomous), post the verdict as `gh pr review --comment` (not `--approve`/`--request-changes`) — a structured comment carrying the same approve/request-changes verdict in its body/label, so downstream tooling (the `auto-merge` skill) can still parse the decision programmatically.
2. `skills/auto-merge/SKILL.md`: guardrail #2 (review decision) reads the structured comment verdict when running under auto-mode instead of requiring a platform-level `APPROVED` `reviewDecision` — but the squash-merge step itself (`gh pr merge`) still requires a human to execute it under auto-mode, since that's the harness's actual enforcement point, not just GitHub's. Document this explicitly: "the squash-merge is always a human step under auto-mode" becomes the accepted, documented behavior, not a bug.
3. `agents/task-orchestrator.md` + `PLAN.md` §6/§7: update the chain's terminal-state documentation so "reviewer posts an advisory verdict, then the human merges" is the specified auto-mode behavior, not a silent degradation.
4. Options (a) (route the reviewer's `gh pr review` through a distinct GitHub identity) and (c) (a harness-level second-actor channel) are **not** pursued here — (a) doesn't reliably clear the classifier per the task's own analysis (same-automation-control keys on session, not GitHub identity), and (c) is out of atelier's control. Record both as rejected-with-reason in this plan so they aren't re-litigated.
